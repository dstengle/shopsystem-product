# Press Release / Working Backwards (PM technique skill — adapted, EXPERIMENTAL)

**Serves:** product-narrative · **Emits into:** README / site / current-state

A *technique skill* (PDR-033): a lens invoked INSIDE the `product-narrative` PM session; its text lands in the README / site / current-state. It is a narrative lens, NOT a standalone PR artifact.

*Adapted from [`deanpeters/Product-Manager-Skills/skills/press-release`](https://github.com/deanpeters/Product-Manager-Skills/tree/main/skills/press-release) — fetched live 2026-07-10. Adaptation rules + experimental status: [`../README.md`](../README.md).*

## Why the session reaches for this

Working-backwards writes the announcement *before* the build to test whether the
direction is compelling from the customer's side. It is a **forcing function for
clarity**, not launch-day marketing: if you can't write a crisp, benefit-led PR,
the idea isn't ready. The drafted text is then distilled into the shop's
current-state / README narrative — no separate PR document survives.

## The structure (draft these, benefit-first)

- **Headline** — the benefit, in the customer's words.
- **Dateline + one-line intro** — what it is, for whom.
- **Problem** — validated with evidence, not "inefficiency in workflows."
- **Solution** — stated as outcomes, not a feature list. Translate every
  capability: "cuts invoice processing 60%" replaces "includes ML and OCR."
- **Leader quote** — visionary and customer-focused, with substance.
- **Supporting detail** — metrics, how it works at a glance.
- **Boilerplate + call to action.**

## Retain the round-trip (PM mode is interactive)

Draft *with* the product authority: pull the problem, the metric, and the
customer's own language from them one question at a time. Where a claim lacks
evidence, ask for it or mark it as an assumption to validate — do NOT invent
metrics to fill the template.

## Validation gate (must pass before the text lands)

- Would the customer recognize this problem as theirs?
- Are the benefits measurable and specific?
- Could an outsider understand the pain without context?
- Is it free of jargon and feature-listing?

## Consumer / framework-as-product fork

- **Consumer:** the PR announces a product to end-customers; benefits are
  end-user outcomes in a real market.
- **Framework-as-product (bootstrap):** the PR announces a shopsystem capability
  to adopters/operators; benefits are developer-experience outcomes. The
  shopsystem README's own narrative is this fork's home instance.

## Lands in current-state / README

The passing draft is distilled into the product narrative on the README / site /
current-state. Pitfalls to reject: feature-listing over benefits, vague problems,
hollow quotes ("excited to bring innovation"), unsubstantiated claims. Use for
new directions and major capabilities; skip for trivial tweaks.
