# sess-2026-07-16-a — Legacy corpus migration, then the precondition chain

## Outcome

The product authority, on learning that the overnight corpus-validation
pass (brief-019/`lead-5msa9`) found ~97 legacy `adr/`/`pdr/`/`briefs/`
files carrying no frontmatter — including files authored the same week —
did not recall endorsing a permanent exclusion of that corpus from the
typed-schema system `cand-001`/PDR-032 introduced, and observed the split
undermines the schema work's own purpose (progressive-disclosure tooling
needs one consistent corpus to index).

Recorded as `intent-006` (verbatim anchor captured), shaped directly into
`cand-004` given the direction was already convergent, then run through
the deciding conversation on the one genuine open tradeoff — migration
sequencing. Framed three options at parity (leave excluded / single-pass
/ phased gate-verified batches); the product authority chose phased
gate-verified batches, weighing lower fidelity risk on ~90 files of
variably-formatted legacy content against slower full closure.

Converged on `pdr/034`, drafted — not yet ratified. It supersedes only
PDR-032's legacy-migration appetite line, not the six-type schema itself,
and explicitly routes the batching-grouping and migration-mechanism
questions to an Architect feasibility probe rather than deciding them at
the PM altitude.

Before dispatching `pdr/034`, the product authority asked a direct
verification question — does `adr/048-...md` (visibly missing
frontmatter) actually pass `shop-knowledge validate`? It correctly
doesn't. But checking it prompted checking the system around it, and the
same session's own newly-produced artifacts (`pdr/034`, `cand-004`,
`intent-006`, plus this session record itself) failed the same validator
immediately on first run — the same shape as `lead-6n4j6`'s already-filed
finding, now confirmed against five independently-authored instances
including brand-new ones. Direct inspection of `shop-knowledge template
<type>` output, the poured `.claude/skills/*` (zero reference
`shop-knowledge` at all), the installed `shop-templates` version
(`0.52.6`, predating `lead-5msa9.2`'s unreleased fix), this repo's pour
history (last full re-render `693d113`, 2026-07-10, `v0.51.0`), and the
absence of any CI/`bin/doctor` enforcement surfaced ten linked
preconditions, of which only one (the `shop-knowledge` CLI itself)
actually holds.

Recorded as `intent-007`, shaped directly into `cand-005` — five phases
in dependency order (typedef correctness → release+repour → minimal
enforcement → build the actual coherence gate → migrate the legacy
corpus). `cand-004`/`pdr/034` are not superseded — their content stands
unchanged — but both are now explicitly parked as `cand-005`'s Phase 5,
since dispatching the legacy migration ahead of a working typedef/gate
would have added an eleventh broken link instead of closing the chain.

`cand-005` committed same session: full-chain appetite ("fund it all"),
not foundation-first. Phase 1's direction (fix the typedef vs.
restructure practice) stays an Architect feasibility call, not resolved
by the ratification — routed onward as the immediate next dispatch.
