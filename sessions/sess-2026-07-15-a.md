# sess-2026-07-15-a — Structured-corpus query tools as retrieval interface

## Outcome

The product authority proposed a concrete solution shape for `intent-004`'s
previously-open "solution shape is genuinely open" question: route
subagent retrieval against structured corpora (the scenarios corpus, and
analogously the knowledge/decision-record corpus) through each corpus's
canonical query tool, since both carry structured YAML/`@`-tag frontmatter
that supports cheap, exact, complete queries — rather than depending on a
subagent's ad-hoc `grep` correctly guessing scope.

This session's own dispatch trail supplied unusually strong, ready-to-hand
evidence rather than requiring a fresh probe: five wrong-path citations
occurred across dispatches earlier the same day (four cosmetic, zero
functional impact, since `@scenario_hash` — not path — is the real
cross-repo identity), and one caused a genuine scope-verification gap
(`lead-ifye3.6`/`lead-r2bsr`) when a manual grep of a single file missed a
sibling scenario in the same directory. Direct inspection of the installed
`scenarios` CLI's own source confirmed its aggregate operations already
scan the full tree correctly — the tool was never the defect; the
subagent's ad-hoc grep habit was.

Shaped into `cand-003`, small appetite, bounded to the two corpora this
session actually demonstrated problems in. One element (scenario corpus)
is reasonably well-understood; the other (knowledge corpus) needs an
Architect feasibility probe before it can be shaped further, since
whether `shopsystem-knowledge` currently exposes a query interface (versus
only the schema-validation capability already being built under
`lead-5msa9`/brief-019) is unconfirmed.

Presented back to the product authority for go/no-go and appetite
confirmation — not yet ratified.
