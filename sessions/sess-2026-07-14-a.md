# sess-2026-07-14-a — fabro LLM provider selection + spend observability discovery

## Outcome

Multi-mode session (discovery -> shaping; recorded under its dominant/
opening mode, per the sess-2026-07-09-a precedent). Opened on a
directional stakeholder ask ("time to look at using multiple model
providers with fabro, starting with openrouter"), which was itself a
proposed solution rather than the problem. Probing surfaced that the
real friction is provider/model choice being locked inside
release-gated artifacts, under a fast-moving and unlikely-to-improve
OAuth-gating constraint — sharpened into intent-002. A second,
causally-linked but distinct problem surfaced when the stakeholder
named that moving real spend onto per-token billing requires granular
cost observability, with two separable motives (spend-cap guardrail vs.
longer-standing effectiveness observability) — captured as intent-003.
Grounded against current-state.md (seed-only, no BC entries yet — no
conflicting current-state claims to reconcile) and the existing
`lead-obfub`/`lead-txhou`/`lead-2ckf7` backlog beads, cited as evidence
rather than treated as authoritative scope.

Per stakeholder direction ("if you see something easy to prioritize as
a blocker, move forward unless told otherwise"), proceeded directly
into shaping intent-002 — the structural blocker for intent-003's
guardrail motive, and unblocking in-flight fleet work (lead-14xb).
Requested one bounded Architect feasibility probe (config-override
precedent, OpenRouter credential-path shape, BC-decomposition sanity
check); all three questions resolved with high-to-medium confidence
and no contradictions found. Closed on cand-002, shaped.

Product authority then flagged a real gap in the shaped sketch: literal
per-provider model IDs in `model_stylesheet` would defeat the
candidate's purpose (provider switches would still require rewriting
every node-class mapping by hand). Requested a second bounded probe on
whether fabro supports tier/effort indirection natively; finding was
inconclusive on fabro internals (ADR-018 wall) but identified that
bc-launcher's existing text-patch precedent (`BC_NAME`/`WORK_ID`
rewrite) sidesteps the question — resolution happens before fabro ever
parses the file. Revised cand-002 in place (logged in its own
changelog, not silently) to add the tier+effort→model mapping table as
a core element, shifting bc-launcher's slice up and templates' slice
down.

Product authority then caught a factual error in the second probe's
citation: the claimed "`BC_NAME`/`WORK_ID` rewrite" precedent for
post-pour file text-patching does not exist — sourced from stale
`findings/fabro-spike/*` material rather than the canonical, current
ADR-058, exactly the failure mode this shop's spike-precedence rule
exists to prevent. Verified directly against ADR-058 (read in full)
and `features/shopsystem-templates/fabro-dispatcher-ref/*`: both
`BC_NAME` and `WORK_ID` are delivered via fabro's input/overlay
mechanisms, never by patching a poured graph file's content. Corrected
cand-002 again (changelog-logged): removed the false precedent,
elevated "does templating reach `model_stylesheet`" from a
light-validation footnote to the candidate's central open unknown.

Product authority asked how the probe would run and directed that it
produce no filesystem output except a typed-artifact record — an
explicit containment constraint given the `findings/fabro-spike/*`
staleness problem just surfaced above. Ran the probe directly (router,
not delegated): a disposable `docker run --rm` container from the
fleet's own bc-base:latest image, `fabro install --skip-llm` bootstrap,
`fabro workflow create` scaffold + a `model_stylesheet` templating test,
torn down immediately after (`docker rm -f`). `git status` confirmed
zero repo residue. Three independent checks (`fabro validate` error
message, `fabro preflight` resolution, `fabro run --dry-run` execution)
all confirmed fabro's templating genuinely resolves `{{ inputs.X }}`
inside the graph-level `model_stylesheet` attribute — a definitive GO.
This closes cand-002's central open unknown; the finding is recorded
only in cand-002's Evidence section, no new file created.

Product authority ratified and committed cand-002. Routed to lead-po
for brief + Gherkin scenario authoring — this is the PM/shaping mode's
exit boundary; commitment and scenario-writing are lead-po's work, not
this session's.
