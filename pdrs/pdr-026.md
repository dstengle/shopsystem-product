## Context

Today a published image's identity is not verifiable from the image itself or
from a container started from it. `bc-base:latest` carries only the upstream
devcontainer base's `version=3.1.2` label — which is misleading, since that is
not the shopsystem/bc-launcher version — and there is no label or env naming
the bc-launcher release or the baked shop-templates version. The job-to-be-done
is: "when I am told about an image, confirm I am running the right one." The
value at stake is verifiable provenance, and there is a synergy: recoverable
provenance makes two already-filed checks cheap — the bootstrap baked-version
verify (lead-b2iz) becomes a `docker inspect` / `printenv` read instead of a
`pip show`, and `bin/doctor` (lead-q3r1) can report the version from the same
keys.

The desired observable behavior change: an operator (or an agent) runs one
inspect/printenv against an image or a running container and reads the correct
bc-launcher and shop-templates versions, instead of having no recoverable
answer.

## Options considered

The core alternative is *how* provenance is carried on the image:

- **Version tags (rejected).** Tags cannot carry provenance: `:latest` and
  `:vX.Y.Z` resolve to the same digest, and the run-tag is not recoverable from
  a pulled image or a running container. Tag-based identity does not survive
  the pull, so it cannot answer "which one am I running?"
- **OCI labels AND ENV, baked at build time (chosen).** Labels and ENV are
  baked at build time — the values already exist there — and inherit onto every
  container regardless of the run-tag. This is the only path where the answer
  survives onto a container started from a floating tag.

## Decision

Provenance is carried as OCI labels AND ENV, not as tags (D1), because only
baked metadata that inherits onto containers can answer "which one am I
running?"

The contract is the inspect-surfaced key set, named explicitly (D2). The key
*names* are part of the contract — downstream consumers (b2iz, doctor) read
them by name, and step 2's dependency map extends the same `shopsystem.*.version`
label namespace rather than inventing a parallel one:

- OCI labels: `org.opencontainers.image.version` = the bc-launcher release
  version; `org.opencontainers.image.revision` = the source git sha;
  `shopsystem.shop-templates.version` = the baked shop-templates version.
- ENV: `SHOPSYSTEM_BC_LAUNCHER_VERSION` = the bc-launcher release version;
  `SHOP_TEMPLATES_VERSION` = the baked shop-templates version.

The keys are pinned; the Dockerfile mechanism that emits them and the
publish-workflow build-arg wiring are the BC's to author.

The misleading upstream `version=3.1.2` is overridden/disambiguated (D3): what
`inspect` surfaces as the version must be the bc-launcher release version, not
the upstream base label value. This is arguably part-bug, and disambiguating it
is a named assertion, not a side effect; it applies to BOTH published images,
bc-base and bc-lead.

The contract is the observable, not the probe (D4). Scenarios pin what
`docker image inspect` and `docker container inspect` surface; they do not pin
LABEL/ENV instruction ordering or CI step shape. Both surfaces are pinned
because both are real read paths: the image artifact (image inspect) and the
running container where the run-tag is lost (container inspect from `:latest`).

## Consequences

Decomposition and dispatch are the architect's, later — not done here.
bc-launcher already publishes both images but emits no shopsystem provenance,
so this adds a new capability to the publish path (lead-5xnd message-type
discriminator): the author leans `assign_scenarios` to shopsystem-bc-launcher
(scenarios 71, 72). The D3 override of the misleading `3.1.2` has a bug flavor;
the architect confirms the vehicle when verifying pre-state, verifies pre-state
empirically against the contract/artifact surface, and adds the `@bc:` tag (and
reconfirms the reproducing `@scenario_hash`) at dispatch.

Downstream, this provenance path is the enabler that lets lead-b2iz and
lead-q3r1 become cheap inspect/printenv reads, and step 2's dependency map
extends the `shopsystem.*.version` label namespace.
