## Source (pre-modernization)

#### The question

How should the transition from `--bc-root`/`--lead-root` path-based
addressing to `--bc`/`--lead` name-based addressing be handled?

The `--bc-root` and `--lead-root` flags appear throughout:
- The lead shop CLAUDE.md (session start instructions, `shop-msg watch`
  invocation, CLI examples)
- The lead-architect.md agent template (BC pre-state verification,
  `shop-msg send`, `shop-msg read outbox` calls)
- The lead-po.md agent template (clarify-response CLI mechanics:
  `shop-msg read outbox --bc-root`, `shop-msg respond clarify
  --bc-root`)
- The BC primer (inbox/outbox read instructions, `shop-msg respond`
  examples)
- Any operational runbooks or scripts that invoke `shop-msg` with
  path flags

The scope of callers is bounded and known. All of them live in
repositories the lead shop controls directly (the lead shop repo itself
and `shopsystem-templates`, which owns the primer and agent templates).

#### Options considered

##### Option A — Clean break

Remove `--bc-root` and `--lead-root` immediately when name-based
addressing ships. All callers update atomically in the same commit or
PR that ships the new flags. The old flags produce a non-zero exit and
a clear error: `"--bc-root is no longer supported; use --bc <name>"`.

**Pros:**
- Simplest final state: one addressing model, no conditional logic in
  `shop-msg`.
- No risk of callers silently continuing to use the deprecated path.
- Eliminates the two-resolution-paths complexity in the implementation.
- Forces all callers to be found and updated at ship time, preventing
  drift.

**Cons:**
- If a caller is missed, it fails with a hard error after the cut-over.
  Missing callers must be found and fixed before or immediately after
  shipping.
- Requires all call sites to be updated in one coordinated move. For
  the known callers (CLAUDE.md, two agent templates, BC primer), this
  is a manageable number of files.

##### Option B — Deprecation period

Keep `--bc-root` and `--lead-root` working alongside `--bc` and
`--lead` for one release cycle. Callers continue working without changes;
they migrate at their own pace. After the deprecation period, remove.

**Pros:**
- Safer migration: no caller breaks at the moment the new flags ship.
- Enables testing the new flags before all callers are updated.

**Cons:**
- `shop-msg` must maintain two resolution paths (path-based and
  name-based) simultaneously. This increases implementation complexity
  and creates test surface for behavior that is intended to be temporary.
- Callers that are never updated continue using path-based addressing
  indefinitely — the deprecation period may stretch because there is no
  hard forcing function.
- The caller list is small and known; the "callers migrate at their own
  pace" benefit is less compelling when the callers can all be updated
  in a single coordinated dispatch.
- Path-based resolution still works, meaning the system is not fully
  container-safe during the deprecation window. Brief 004's containerized
  scenario is not testable until the callers are migrated anyway.

##### Option C — Alias (path-or-name detection)

`--bc-root` becomes an alias for a unified `--bc` flag that accepts
either a filesystem path or a canonical name. `shop-msg` heuristically
determines which was passed (a name if it matches a registry entry, a
path if it looks like a filesystem path).

**Pros:**
- Zero migration burden for existing callers.
- Backward compatible indefinitely.

**Cons:**
- Introduces ambiguity: a canonical name that could also look like a
  relative path (`shopsystem-messaging` is safe, but a name like `foo`
  could be a directory). The heuristic requires special-casing.
- Preserves path-based addressing permanently (or until a future PDR
  removes it), contradicting the invariant that shops are addressed by
  name.
- The "alias" approach signals to future contributors that path-based
  addressing is still acceptable, which undermines the registry as the
  resolution layer.

#### Decision

**Option A — Clean break.**

Rationale:

1. **The caller list is bounded and known.** The files that use
   `--bc-root` and `--lead-root` are: the lead shop CLAUDE.md,
   lead-architect.md, lead-po.md, the BC primer (in
   `shopsystem-templates`), and any runbooks. This is a small,
   enumerable set. A clean break is feasible because there is no unknown
   long tail of callers; every caller can be audited and updated as part
   of the migration dispatch.

2. **Two resolution paths add implementation complexity for zero
   behavioral benefit.** If the goal is name-based addressing, the
   deprecation period delivers path-based addressing that still works —
   which means the containerization story (brief 004) is incomplete until
   callers migrate anyway. The clean break forces the completion of the
   migration, and the complexity of two resolution paths is never
   introduced.

3. **The hard error is informative.** When a caller with an old flag
   hits the cut-over, it receives: `"--bc-root is no longer supported;
   use --bc <name>"`. This is a clear, actionable error that surfaces any
   missed caller immediately, rather than silently routing by path when
   the intent is name-based routing.

4. **The alias approach undermines the invariant.** Option C is
   attractive for its zero-migration-burden, but the invariant in
   brief 006 is "shops are addressed by canonical name, not by filesystem
   path." An alias that preserves path-based routing contradicts the
   invariant rather than enforcing it. The PO cannot recommend an option
   that makes the invariant a preference rather than a contract.

**Migration commitment:** The Architect's dispatch for brief 006 scope B
must include, as part of the same assign_scenarios dispatch:

1. A scenario that verifies `--bc-root` produces a clear non-zero error
   message after the cut-over.
2. A scenario that verifies `--lead-root` produces a clear non-zero error
   message after the cut-over.
3. An explicit task (in the dispatch description or a separate
   `request_maintenance`) to update: the lead shop CLAUDE.md, the two
   agent templates (lead-architect.md, lead-po.md), and the BC primer
   in `shopsystem-templates`. All four files must be updated before or
   at the moment the new CLI ships.

The lead-po.md's own CLI mechanics section (which currently references
`--bc-root`) is updated as part of this migration, as the PO's own
instructions must reflect the canonical interface.

#### What this leaves open

1. **Exact error message wording.** The PO commits that a clear,
   non-zero-exit error is produced. Exact wording is a scenario-level
   detail.
2. **Whether `--bc-root` produces an error or is silently ignored.** The
   PO commits non-zero exit (the Architect must not choose silent
   ignore). The exact error mechanism (unrecognized flag vs. recognized-
   but-removed flag with custom message) is the Architect's call.
3. **Order within the brief 006 dispatch.** The Architect determines
   whether scope B (name-based flags) ships before or alongside scope A
   (registry commands). Scope A must exist for scope B to work; they may
   ship atomically or A before B.
