## Source (pre-modernization)

#### Point of intent

Convert the front door's promise — *"talk to the lead and it stands up your
product"* — from a missing capability (the journey's deepest dip) into a real,
deterministic Footing plus an explicit agent-driven Discovery. The reframe splits
the bootstrap into three phases (brief §2): **Footing** (deterministic, NO agent
— reaches solid footing: green `git push` + `bd dolt push`), **Discovery** (the
agent-driven product-definition step where the plain-language orchestration
actually lives), **BC creation** (the third step, pairing with `create-bc` /
`lead-okre`). This PDR commits the Footing shape and decomposes it; Discovery UX
and BC-creation mechanics stay explicit follow-ons (brief §5).

#### Empirical finding (the gap this closes)

Verified against the contract/artifact surface (ADR-018) — no BC code read, run,
or git-observed; the full pre-state findings are in ADR-040's "Pre-state
empirical findings" section. In brief: the lead is a router with no skill that
drives services-up → broker-provision → first-BC (`lead-5cgv`, `lead-8vxy`,
`lead-y73x` P1); the OAuth gate is NOW in-script-capturable via the proposal
mechanism (`lead-yrex` CLOSED, correcting the old dashboard-only note);
`bc-base`/`shop-templates bootstrap`/`bc-container` already bake the framework
CLIs the footing needs (`bin/shop-shell`, `shop-templates bootstrap --help`); the
image-tag pin is a live staleness pain (`lead-2xi3`); the manifest `product:`
field is the single declared identity the naming derives from (ADR-038).

#### The three resolved open questions (brief §6)

- **Q1 — Image lineage → ADR-040 D1: the EXISTING `bc-base`/bc-launcher lineage
  in an interactive bootstrap entrypoint MODE, NOT a new image.** Rationale: the
  only thing the footing needs that brokered bc-base does not provide is one
  interactive `claude`+`gh` auth beat; everything else is already baked. A new
  image duplicates that surface and forks the ADR-021/022/039 release cadence
  into two lineages — doubling the `:latest`-lag drift surface. An entrypoint
  mode is strictly cheaper.

- **Q2 — Claude-OAuth capture → ADR-040 D2: IN-SCRIPT via the proposal gate; the
  human only APPROVES, no dashboard route.** Rationale: `lead-yrex` (CLOSED,
  load-bearing-proved against a live 0.32.0 broker) established `agent-vault vault
  proposal create -f -` captures a refresh-preserving `type:oauth` `CLAUDE_OAUTH`
  slot programmatically; the human only `proposal approve <num>
  CLAUDE_OAUTH=<value> --yes` (secret supplied at approve, ADR-026 D4 preserved).
  This consolidates with the GitHub PAT + owner password into ONE up-front gate
  (targets the Stage-5 wall). Depends on `lead-8vxy` for the env-driven +
  single-approval handoff.

- **Q3 — Compose image-tag pinning → ADR-040 D3: the starter `compose` does NOT
  pin the tag; the bootstrap SCRIPT resolves a known-good tag at run time** and
  records it in the run's `.env`. Rationale: a committed pin freezes
  `lead-2xi3`'s staleness trap into every fork; script-resolution keeps the fork
  evergreen and the run reproducible.

Also decided (ADR-040 D4/D5/D6): the credential ratchet (interactive capture →
provisioned vault → brokered steady state); the footing↔Discovery boundary is the
green-push line (script before, agent after); naming enforcement validates the one
user-chosen `*-lead` name, derives `<product>`, offers `gh repo rename`, and forces
the shape for all tooling-created repos via the manifest `product:` (ADR-038).

**U1 ownership decision.** `shopsystem-starter` is a **lead-owned standalone
template repo** (GitHub "template repository"), NOT a new BC. Rationale: it
carries no behavior to pin with scenarios and no code — only compose +
script-as-rendered-artifact + env-example + README. Its *contents* are rendered
by shopsystem-templates (U1 body) so they stay canonical; the repo itself is the
"Use this template" front door, which is a lead/product-owned distribution
surface, not a BC with an implementer→reviewer loop. It does NOT need a BC.

**Ordering / dependencies.** U2 (image mode) and U7 (`lead-2xi3` tag) and U5
(bootstrap extensions) are independent and can dispatch first. U3 (the script)
depends on U2 (interactive mode), U5 (bootstrap renders the scaffold), and U6
(`lead-8vxy` driveable provision). U4 (naming) depends on U3. U1 (the starter
repo assembly) depends on U2/U3/U4 being real (it packages compose+script). So
the critical path is **U2/U5/U6/U7 → U3 → U4 → U1**.

#### Out of scope (brief §5)

Discovery UX (how the adopter is handed into the agent-driven product-definition
step), BC-creation mechanics beyond naming it the third step (pairs with
`create-bc`/`lead-okre`), the example throwaway-product chat, and the
two-phase throwaway-container seed are all explicit follow-ons — not committed
here. This PDR commits the Footing shape, the phase boundary, the three resolved
questions, and the decomposition.
