## Source (pre-modernization)

#### Point of intent

`bin/shop-shell` is a hand-written bash reimplementation of the brokered
interactive-Claude launch that `bc-launcher` (`bc-container launch`) already
performs for every BC. It has drifted from that reference and produced a long
string of one-off bugs that `bc-base` + `bc_launcher` already solve:

- empty agent-vault CA / cold-start CA materialization;
- compose vault-mount path skew (`/vault/file` vs the image's `/data`);
- the default shell image lacking `claude`;
- the Dockerfile installing framework CLIs from non-existent PyPI names;
- the proxy URL missing the broker token → HTTP 407;
- a missing `~/.claude` onboarding/credential placeholder → claude stuck
  "initializing".

Each of these is a re-derivation, in bash, of a problem the launcher's Python
controller has already solved and pinned (`controller.py`: the MITM `:14322`
token-in-userinfo proxy URL, `AGENT_VAULT_CA_PEM`-as-env materialized by the
bc-base entrypoint, the baked placeholder `.credentials.json`, the
trust-prompt readiness sequence). Maintaining two launch codebases for one
behavior is the root cause; convergence onto one is the intent.

A key premise that drove the *separate reimplementation* is now falsified.
`Dockerfile.shopsystem-shell` states "bc-launcher is INTENTIONALLY OMITTED
(private repo)". The `bc-launcher` SOURCE repo is private, but the published
IMAGE `ghcr.io/dstengle/shopsystem-bc-base` is PUBLIC (verified this session:
anonymous GHCR manifest fetch returns HTTP 200) and ships `bc_launcher` /
`bc-container`, `claude`, the framework CLIs, the `agent-vault-ca.sh` CA
entrypoint, and a baked `~/.claude` (placeholder `.credentials.json` +
`.claude.json` `hasCompletedOnboarding: true`). A public, self-contained agent
image with the launcher baked in already exists. The "build a redundant public
shell image because bc-base is private" rationale rests on a false premise.

#### Diagnosis — what the launcher already does, and the one thing it does not

Read-only verification of the installed `bc_launcher` package
(`controller.py`, `driver.py`, `cli.py`, `manifest.py`):

1. **`launch` already produces the exact brokered form the shell got wrong.**
   `controller.launch` (`controller.py:527`) builds the runtime `HTTPS_PROXY`
   via `_build_runtime_proxy_url` (`controller.py:302`) as
   `http://<token>:<vault>@<host>:14322` — the MITM port with token-in-userinfo,
   the precise shape the hand-rolled shell kept getting wrong (407). It injects
   the operator-supplied `AGENT_VAULT_ADDR/TOKEN/VAULT/CA_PEM` triple+CA from
   `--env-file` (`cli.py:275`, `controller.py:677-727`), gates launch on BOTH
   `messaging_db_reachable` AND `agent_vault_reachable` (`controller.py:1093`,
   `:1118`), and wraps the agent as `agent-vault run -- claude
   --dangerously-skip-permissions` (`controller.py:1140`) with the trust-prompt
   readiness sequence (`controller.py:1147-1229`). The broker CA travels as
   `AGENT_VAULT_CA_PEM` and is materialized by the bc-base entrypoint — no
   controller CA bind-mount (`controller.py:713-727`). This is the same broker,
   proxy, and CA model ADR-026/ADR-028 already pin; the launcher is its
   reference implementation.

2. **`--image` is already configurable**, defaulting to
   `ghcr.io/dstengle/shopsystem-bc-base:latest` (precedence `--image` flag >
   `BC_IMAGE` env > built-in default; `controller.py:32`, `:779-784`,
   `cli.py:81`). So "launch a bc-base session" is the default behavior, and any
   pinned image is selectable without source edits.

3. **`attach` gives the interactive human-driven session.** `controller.attach`
   (`controller.py:1237`) execs `docker exec -it -u vscode <container> tmux
   attach-session -t agent` (`driver.py:270`), replacing the operator's process
   with an interactive tmux-attached Claude session. `inject` (send text) and
   `monitor` (capture pane) round out the human-operator surface. This is
   exactly a human driving the lead agent.

4. **`manifest` is NOT consulted by `launch`/`attach`.** The manifest
   (`manifest.py`) is read only by `bc-container manifest {validate,list,sync}`.
   `launch` reads only the top-level `product:` field of a `bc-manifest.yaml`
   in CWD for network/slug derivation (`controller.py:605-608`) — never the
   `bcs:` entry list — and an explicit `--network` short-circuits even that
   (`controller.py:610-613`). **The lead shop needs NO manifest entry to be
   launched as a container.** There is no "lead entry" concept and none is
   required.

5. **THE GAP — `launch` clones; it does not mount an existing local repo.**
   `controller.launch` populates `/workspace` ONLY by `git clone <repo_url>
   /workspace` (`controller.py:826-860`); the sole bind-mount it ever builds is
   the SHOPMSG unix-socket dir, conditionally (`controller.py:760-769`). The
   `driver.run` invocation (`driver.py:236-251`) is
   `docker run --name bc-<name> -d [-e …] [--mount socket] [--network …]
   <image> sleep infinity` — there is **no `-v <local-repo>:/workspace` path and
   no docker-socket mount**. With `--repo-url` omitted, `/workspace` is simply
   empty (all clone/beads/skill provisioning is guarded by `if repo_url:`).
   This is the only thing standing between today's launcher and "run the lead
   shop as just another `bc-container` session": the lead repo is a *local
   working tree the operator is actively editing*, not a remote to clone into a
   throwaipe `/workspace`.

So the convergence is **not** zero-change, and **not** a manifest-shape change.
It is a **small, additive launcher capability**: an option to mount an existing
local repo as `/workspace` (and skip the clone step) instead of cloning a
remote.

#### Decision

1. **The lead shell IS a `bc-container`-launched `bc-base` interactive
   session.** The operator's lead agent runs as a Claude session inside a
   `bc-base` container, started and driven through the SAME `bc-container
   launch … && bc-container attach …` path every BC uses — one launch
   codebase, one public agent image, one brokered-credential model.

2. **`bin/shop-shell` is RETAINED only as a thin wrapper.** Because an adopter
   host may have only docker (no `bc_launcher` installed) and the lead "host"
   is itself a container with the docker socket mounted, the wrapper runs
   `bc-container` from an EPHEMERAL `bc-base` container:

   ```
   docker run --rm -it \
     -v /var/run/docker.sock:/var/run/docker.sock \
     -v <lead-repo>:<lead-repo> \
     ghcr.io/dstengle/shopsystem-bc-base \
     bc-container launch <lead> … [mount the lead repo as /workspace] && \
     … attach …
   ```

   No host install of `bc_launcher` is required; the public bc-base image
   supplies it. The wrapper's remaining job is just compose bring-up (postgres
   + agent-vault, per ADR-028 D1) and forwarding the operator agent-vault
   `--env-file` — NOT re-deriving the proxy/CA/onboarding launch logic.

3. **Retire the reimplementation surfaces.** The hand-rolled brokered-launch
   body of `bin/shop-shell`, `Dockerfile.shopsystem-shell`, and the
   `shopsystem-shell:dev` image are removed. The redundant public shell image
   was justified by the now-falsified "bc-base is private" premise.

4. **Implementation lands LATER, via the normal process, in
   `shopsystem-templates`.** `bin/shop-shell` and `Dockerfile.shopsystem-shell`
   are POURED instance furniture owned canonically by `shopsystem-templates`
   (per the standing "lead owns no code — all infra to shop-templates" rule and
   PDR-003 propagation). The genericized thin wrapper is authored in templates
   and re-poured; it is NOT hand-edited in this repo. The launcher
   local-mount capability (the gap below) is dispatched to `bc-launcher`
   through the message-type discriminator. **None of that happens in this PDR**
   — this PDR commits the decision and the gap, not the code.

#### Relationship to prior decisions — what is superseded, falsified, preserved

- **ADR-028 D1 (`bin/shop-shell` brings up BOTH supporting services):
  PRESERVED.** The compose bring-up of postgres + the agent-vault broker stays
  exactly as decided; the thin wrapper still performs it.
- **ADR-028 zero-host-credential-coupling: PRESERVED, and in fact
  strengthened.** Routing the lead through `bc-container`/`bc-base` + broker
  gives the no-host-credential property by construction (the launcher mounts NO
  host `~/.claude`/`~/.gitconfig` — `controller.py:46-66`, `:754-759`), rather
  than re-asserting it in bespoke bash. What is retired is the *separate
  reimplementation* of the brokered launch, not the coupling posture.
- **ADR-026 brokered-credential model: PRESERVED.** Same broker, `:14322` MITM
  proxy, `AGENT_VAULT_CA_PEM`-as-env, placeholder credentials. The lead shell
  now consumes the launcher's reference implementation of it instead of a
  parallel copy.
- **`Dockerfile.shopsystem-shell`'s "bc-launcher is INTENTIONALLY OMITTED
  (private repo)" rationale: FALSIFIED.** The image is public and ships the
  launcher; the redundant shell image and its Dockerfile are retired.
- **ADR-004 / PDR-004 (bc-container owns containerized agent launch):
  REINFORCED.** This PDR extends that ownership to cover the lead shell, which
  had been the one agent launch *outside* the launcher's scope.

#### Consequences

- **Deleted:** the brokered-launch body of `bin/shop-shell` (proxy URL
  construction, CA fetch/materialization, `~/.claude` handling, the
  `docker run … /bin/bash` agent launch at `bin/shop-shell:197`),
  `Dockerfile.shopsystem-shell`, the `shopsystem-shell:dev` image, and the
  `SHOPSYSTEM_SHELL_IMAGE` knob.
- **New `bin/shop-shell` does, in full:** (a) compose `up -d postgres
  agent-vault` (ADR-028 D1, unchanged); (b) assemble the operator agent-vault
  `--env-file`; (c) `docker run --rm` an ephemeral `bc-base` that runs
  `bc-container launch <lead>` mounting the lead repo as `/workspace` +
  forwarding the docker socket + the env-file, then `bc-container attach` to
  drop the operator into the brokered Claude session. Everything credential /
  proxy / CA / onboarding / trust-prompt is delegated to the launcher.
- **Interface gap → minimal follow-up.** The lead "profile" needs TWO additive
  `launch` capabilities, both absent today only because a normal BC needs
  neither: (i) **mount an existing local repo as `/workspace` and skip the
  clone** (e.g. a `--workspace-mount <host-path>` option that adds the
  `/workspace` bind-mount and bypasses the `if repo_url:` clone/beads/skill
  provisioning a live tree does not need); and (ii) **mount the host docker
  socket into the launched container**, so the lead agent can itself run
  `bc-container` to instantiate and bring up BCs — the orchestration the lead
  exists to do, which a BC never does (the launcher today mounts only the
  shop-msg unix socket, `controller.py:760-769`, never the docker socket). The
  rest — `--image`, the brokered-env injection, `attach`, no-manifest-needed —
  is already in place. Open questions (a) and (e) below bound their exact shape.
- **One launch codebase, one public agent image.** Future brokered-launch fixes
  land once, in `bc-launcher`/`bc-base`, and the lead shell inherits them
  instead of re-deriving them in bash.

#### Open questions / risks

- **(a) Local-mount provisioning semantics.** When `/workspace` is a mounted
  pre-existing working tree (not a fresh clone), which of the clone-path side
  steps still apply? The lead repo already has a committed `.beads/` and poured
  `.claude/skills/`, so the `bd bootstrap` and `shop-templates update`
  re-pours (`controller.py:862-1055`) are likely unwanted for a live tree — but
  ownership (the `chown -R vscode /workspace`, `controller.py:920-930`) on a
  host-owned mount needs deciding, and a host-uid/container-uid mismatch on the
  bind-mount is a real risk. The launcher slice must pin this; it is the
  substance of the follow-up dispatch's scenarios.
- **(b) Lead vs. BC startup prompt.** The default `--startup-prompt`
  (`cli.py:35`) directs a BC session-start (arm `shop-msg watch --bc <name>`,
  drain BC inbox). The lead's session-start is the router sequence (Monitor on
  `shop-msg watch --lead <name>`, the idle checklist). The lead launch must
  pass an explicit `--startup-prompt` total-override carrying the LEAD
  session-start; verify the override path (`cli.py:255-261`) is sufficient and
  no `--bc`-shaped assumption leaks elsewhere.
- **(c) `bc_name` / container-name shape for the lead.** `launch` names the
  container `bc-<name>` (`controller.py:186`) and the tmux session `agent`.
  A lead launched this way would be `bc-<lead-name>`, mingling with BC
  containers in `bc-container list`. Acceptable cosmetically, but the PO/
  Architect should decide whether the lead wants a distinct name shape or is
  content to appear as `bc-<lead>`.
- **(d) Could not be verified from the package surface:** whether the bc-base
  image's baked `~/.claude` onboarding placeholder is byte-current with the
  Claude Code version the image ships — the "stuck initializing" class of bug
  is image-content, not launcher-logic, and is only provable by a real launch
  (a BC/ops concern, not lead-side execution per ADR-018).
- **(e) Docker socket as a LEAD-only privilege.** The lead agent must reach the
  docker daemon to run `bc-container` (instantiate/bring-up BCs); a BC must NOT
  (it has no business launching containers). So mounting the docker socket into
  the launched container has to be an explicit, lead-only launch option — never
  the default — with the security note that a socket-mounted container is
  effectively host-root-equivalent. The launcher slice must gate this behind an
  opt-in flag and the templates wrapper must pass it only for the lead.

#### Addendum (2026-06-23) — live-e2e gaps found post-implementation
Live e2e of the rendered shop-shell surfaced three runnability gaps the substring-scenarios missed:
1. **Image ref** (FIXED, lead-zcob): bare `shopsystem-bc-base` → `ghcr.io/dstengle/shopsystem-bc-base:latest`.
2. **Missing `bc_name`** (FIXED, lead-zcob): launch+attach now pass the slug-derived `<slug>-lead`.
3. **bc-base lacks the docker CLI** (lead-nsj3, OPEN, ARCHITECTURAL): the ephemeral bc-base runs `bc-container launch`, which shells out to `docker` — absent in bc-base → `FileNotFoundError: 'docker'`. The ephemeral-bc-base-as-launcher premise of this PDR requires the launcher image to carry the docker CLI. DESIGN DECISION needed: add docker to bc-base (bloat/security for leaf BCs) vs a separate launcher-flavored image vs revise the approach. Plus dependency lead-cek6 (bc-base `:latest` must track ≥v0.3.6). The convergence is CODE-COMPLETE + scenario-passing but NOT yet runnable until #3 (+lead-cek6) resolve.

#### Addendum (2026-06-23, II) — DECISION on gap #3: a thin `bc-lead` launcher image (option b)

**Decision (product authority, 2026-06-23):** Gap #3 is resolved by **option (b)**
— a separate, *very thin* launcher-flavored image **`bc-lead` = `FROM bc-base` +
the docker CLI**, used ONLY for the ephemeral-launcher role in `bin/shop-shell`.
Leaf BCs continue to run on the lean `bc-base` image unchanged.

**Why (b), not (a) "add docker to bc-base":** the ephemeral-launcher is the ONLY
role in the system that shells out to `docker` (it runs `bc-container launch` to
create containers — see this PDR's gap (e), docker socket as a lead-only
privilege). Every leaf BC (`shopsystem-messaging`, `-scenarios`, `-templates`,
`-bc-launcher` agents, and any adopter-product BC) runs the agent only and NEVER
launches a container. Baking the docker CLI into `bc-base` would put a
docker-in-docker footprint and its host-root-equivalent security surface into
EVERY leaf agent image for a capability only the launcher role uses. Option (b)
confines that footprint to the single image the launcher role pulls, keeping the
broad fleet lean and minimizing the privileged attack surface — the same
lead-only-privilege posture this PDR's open question (e) already requires for the
docker socket mount.

**Why (b), not (c) "revise the launcher approach":** the convergence (one launch
codebase, one brokered-credential reference implementation) is the whole point of
PDR-020 and is already CODE-COMPLETE + scenario-passing. Re-deriving a separate
launcher mechanism would re-introduce the dual-codebase root cause this PDR
retires. `bc-lead` keeps the single `bc-container launch` path intact and only
swaps the *image* the ephemeral launcher runs in.

**Shape:** `bc-lead` is `FROM ghcr.io/dstengle/shopsystem-bc-base:<ver>` plus an
install of the docker CLI, published to GHCR alongside `bc-base` by the same
publish pipeline. `bc_launcher`/`bc-container`/`claude`/the CA entrypoint/the
baked `~/.claude` all inherit unchanged from the base layer.

**Follow-on work this addendum authorizes:**
- `shopsystem-bc-launcher` (owns the image build + publish): add the `bc-lead`
  image build + publish (lead-nsj3), and fix `publish-bc-base.yml` so `:latest`
  tracks newest-on-tag-push for both images + re-tag `:latest`=v0.3.6 (lead-cek6).
- `shopsystem-templates` (owns the poured `bin/shop-shell`): point the
  ephemeral-launcher `docker run` at `bc-lead` instead of `bc-base` — a SEPARATE
  templates change + re-pour workstream (tracked separately, not folded into the
  bc-launcher dispatch).
