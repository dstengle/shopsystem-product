## Source (pre-modernization)

#### Point of intent

The framework treats roles as **first-class agents** — that is the whole
point of the role-template architecture (§3, §4). The lead shop's
**operation** must match this model. Currently the lead-shop CLAUDE.md
(per [PDR-001](001-role-templates-role-complete.md) / 58243d1) collapses
PO + Architect into a single main-agent persona that toggles between
them, holds both postures simultaneously, and self-polices against
drifting into Implementer thinking. That works for routine operational
turns but **leaks under direct pressure**: a request framed as "what
should we build?" lands faster than the agent can re-derive the §3 → §4
process from a once-loaded primer.

The primer in CLAUDE.md is **passive enforcement** — text loaded at
session start, then text-among-text. **Active enforcement** requires
that PO and Architect be invoked as actual subagents, each loading its
own role template into its primary context, with the main agent
operating as a **router** that classifies the incoming request and
delegates.

#### Diagnosis

What happened in conversation 2026-05-12 confirms the gap:

- The (newly written) CLAUDE.md correctly framed PO + Architect roles
  and the discriminator process.
- An exploratory user question ("how can we progress with the test
  harness") still drew an initial impulse to propose implementation
  directly.
- The agent caught itself, but only because the CLAUDE.md language was
  vivid in immediate context. That advantage decays across a session.

The mechanism that catches drift under pressure cannot rely on the
agent remembering to re-read the primer. It must be **structural**: the
role's template is in the dispatching agent's primary context, not the
main agent's secondary context.

#### Decision

Lead-shop operation SHALL be formalized as:

1. **A `lead-po` subagent** at `.claude/agents/lead-po.md` whose system
   prompt sources the canonical `lead-po.md` role template from
   `shopsystem-templates`.

2. **A `lead-architect` subagent** at `.claude/agents/lead-architect.md`
   whose system prompt sources the canonical `lead-architect.md` role
   template from `shopsystem-templates`.

3. **The main agent acts as a router**, not a role enactor. Its
   `CLAUDE.md` responsibilities are limited to:
   - **Classifying** each incoming request: judgment turn (PO
     territory? Architect territory? both?) vs. operational turn (git,
     beads, file ops, status reporting).
   - **Dispatching** judgment turns to the appropriate subagent.
   - **Coordinating** turn-limited PO ↔ Architect exchanges per §3.4
     (the round cap becomes enforceable: the router counts the rounds).
   - **Routine operational work** (git, beads, repo state checks)
     without subagent overhead.

4. **A subagent is invoked when** the user's request requires either:
   - Producing a lead-shop artifact (brief revision, PDR draft, ADR
     draft, scenario authoring, structurizr edit, dispatch
     composition); OR
   - Applying a role's discriminator or sufficiency check (message-type
     selection, clarify-routing, BC decomposition judgment).

5. **A subagent is NOT invoked when** the user's request is:
   - Routine git / beads / shell operations.
   - Reporting current repo state.
   - Conversational clarification of what was just done.

#### Implications for the templates BC

The role templates (`lead-po.md`, `lead-architect.md`) are owned by
`shopsystem-templates`. To support subagent dispatch, two paths exist:

- **(a) `.claude/agents/lead-*.md` includes the template content
  inline** by copy or build-step concatenation. Simpler in the lead
  shop; risks drift between the canonical template and the subagent
  definition.
- **(b) `shop-templates` gains a new export mode** that emits a
  subagent-formatted file (with the required frontmatter). The lead
  shop's `.claude/agents/` files become thin wrappers that pull from
  `shop-templates`.

Path (b) is the cleaner answer — it keeps the templates BC as the
single source of truth for role content — but it requires changes
inside `shopsystem-templates` that are out of scope for this PDR. The
lead shop authors scenarios pinning the new export mode and dispatches
via `assign_scenarios` as a follow-up.

Path (a) is acceptable as an **interim**, with a checked-in note that
the inline copy is provisional until (b) lands.

Either path requires that PDR-001's role-complete restructure of the
templates land first (or concurrently), since the subagent format will
then reflect the role-complete content. Sequencing intent:

```
PDR-001 (role-complete restructure)  ──┐
                                       ├──>  subagent definitions usable
PDR-002 (subagent router) export mode  ──┘
```

#### What this leaves open

- **Cost / overhead empirics.** Subagent invocation has real cost
  (separate context, model call). Whether the cost is justified for
  every judgment turn, or only for high-stakes ones, is unvalidated.
  Defer until a few sessions provide signal.
- **BC-shop parallel.** Whether `bc-implementer` and `bc-reviewer`
  should also be dispatched as subagents inside a BC's own repo is a
  parallel question, not committed here. The §4.4 loop's
  Implementer-blocked-by-Reviewer mechanic is the natural test case —
  if §4.4 enforcement turns out to need active dispatch the way §3
  does, a follow-up PDR commits it.
- **Concrete subagent file contents.** This PDR commits the pattern,
  not the prose. The first version of `.claude/agents/lead-po.md` and
  `.claude/agents/lead-architect.md` is implementation work that
  follows.
- **Router classification heuristics.** The PDR names the *categories*
  (judgment vs. operational) but not the exact decision rule. The rule
  develops empirically across the first few sessions; once stable, it
  is recorded in `CLAUDE.md` as the router's contract.
