## Context

The lead needs to pull a BC's scenario register *with* per-pin detail (hash +
title/text + `features/` file location + live/retired status) so it can
**locate, import, or supersede** a pin. We already have
`request_completion_journal`, which returns bare completed hashes. The open
question is whether the register-visibility need should be met by (A)
**extending** `request_completion_journal` to optionally carry detail, or (B)
authoring a **distinct** `request_scenario_register` vehicle.

## Options considered

- **(A) Extend `request_completion_journal`.** Add an optional "detail" mode to
  the existing request/response so one vehicle answers both "which hashes are
  done?" and "give me the full register entries." Rejected: it makes the
  response shape conditional and harder to validate, and it would weaken the
  journal's pinned bare-hash contract (scenarios 26/27 pin that a
  `request_completion_journal` response is *exactly* a bare set of hashes with
  no per-entry record beyond the hash).
- **(B) Distinct `request_scenario_register` vehicle.** A separate message type
  whose response *always* carries per-entry detail, standing beside the journal.
  Chosen.

## Decision

Author a distinct `request_scenario_register` vehicle (Option B). The register
is **additive**: it stands beside `request_completion_journal`, which keeps its
bare-hash contract unchanged.

Rationale:

1. **Different questions, different answers.** The journal answers *"which
   block-only hashes has this BC completed?"* (a historical done/not-done fact,
   per Brief 009 decision #1). The register answers *"what are this BC's pins,
   where do they live, and are they still live?"* These are distinct jobs;
   collapsing them behind a mode flag on one message type makes the response
   shape conditional and harder to validate.

2. **The bare-hash contract is load-bearing for the journal.** Scenarios 26/27
   pin that a `request_completion_journal` response is *exactly* a bare set of
   hashes "with no per-entry record beyond the hash." Extending it to sometimes
   carry detail would weaken that pinned contract. Keeping the journal's
   bare-hash invariant intact is cheaper than relaxing and re-pinning it.

3. **Register entries carry a different identity surface.** A register entry
   includes `features/` file location and live/retired status — concepts the
   journal deliberately omits (the journal is content-addressed by hash alone,
   ADR-019). These belong to "the register," not "the journal."

4. **Distinct vehicles compose cleanly with the discriminator.** A new
   capability cleanly maps to `assign_scenarios`; an extension would muddy
   whether the change is new-capability vs. unpinned-capability for the
   Architect's discriminator.

## Consequences

- A new `shop-msg` message type, pinned by `features/messaging/36` (request)
  and `37` (response).
- The Architect verifies `shopsystem-messaging` pre-state, applies the
  discriminator (new capability ⇒ `assign_scenarios`), and dispatches.
- `request_completion_journal` is untouched; scenarios 26–29 remain valid.
