## Source (pre-modernization)

#### The question

Where does the BC manifest file live, and which BC owns the CLI that
operates on it (the sync and validate commands)?

The answer determines which BC receives the `assign_scenarios` dispatch
for brief 005's scope items B and C, which repo the implementation
lives in, and where the manifest file's well-known path is anchored.

#### Context

Brief 005 commits:
- **Scope item A** — a machine-readable manifest file committed to the
  lead shop repo at a well-known path.
- **Scope item B** — a sync command that clones repos from the manifest
  and validates the manifest against actual state.
- **Scope item C** — drift detection as part of the validate operation.

All three options below agree that the manifest file itself lives in
the lead shop repo. The ownership question is about the CLI (scope items
B and C), not the file (scope item A). Scope item A is a committed file
in the lead shop; the CLI is what the Architect must dispatch to a BC.

The six current product BCs, relevant to this decision:
- `shopsystem-messaging` — inter-shop messaging surface (shop-msg CLI)
- `shopsystem-scenarios` — scenario hashing and registration
- `shopsystem-templates` — bootstrap scaffolding (shop-templates CLI)
- `shopsystem-test-harness` — framework test harness
- `shopsystem-devcontainer` — devcontainer image build and publish
- `shopsystem-bc-launcher` — per-BC container lifecycle (bc-container CLI)

#### Options considered

##### Option A — CLI owned by `shopsystem-templates`

The templates BC already owns the bootstrap scaffolding CLI
(`shop-templates`). It is the BC from which the lead shop's own
structure is derived. A manifest sync command could be a new subcommand:
`shop-templates manifest sync`, `shop-templates manifest validate`.

**Pros:**
- Bootstrap is about standing up shops; the manifest declares which
  shops exist. The two concerns are adjacent.
- `shopsystem-templates` already knows about the lead shop's structure
  (it generates `CLAUDE.md`, `consumer-wiring.md`, settings files).
  The manifest is a sibling artifact in the same genre.
- No new BC. One fewer repo to manage.
- The templates BC already handles git operations (cloning, updating
  managed files); clone-sync is a natural extension.

**Cons:**
- The templates BC is under active load (PDR-001 restructure, CLAUDE.md
  propagation, bootstrap surface, event-driven settings). Adding
  manifest CLI scope before that work closes risks priority dilution.
- `shop-templates manifest sync` is thematically sound but places
  the manifest management concern inside a BC whose primary surface is
  generating and updating files from templates. Manifest sync is about
  cloning repos and validating GitHub remotes — an operational concern,
  not a templating concern.
- The manifest file itself lives in the lead shop repo; the CLI that
  operates on it would live in a different repo (`shopsystem-templates`).
  This is an acceptable split (the CLI operates on a file in another
  repo) but creates a dependency the operator must remember.

##### Option B — CLI owned by `shopsystem-bc-launcher`

The bc-launcher BC already needs a BC list to know which containers
to launch (`bc-container launch --all`). The manifest is the bc-launcher's
input. Giving the launcher CLI ownership of the manifest operations
makes the launcher the authoritative manifest manager: `bc-container
manifest sync`, `bc-container manifest validate`.

**Pros:**
- The bc-launcher already has the strongest stake in the manifest: it
  cannot implement `bc-container launch --all` without an authoritative
  BC list. The manifest is the launcher's prerequisite, not just a nice
  complement.
- Collocating the CLI and its primary consumer in the same BC means
  the BC that needs the manifest also owns the tooling to keep it
  accurate. No cross-BC dependency for the common case.
- `shopsystem-bc-launcher` is new (no legacy scope conflict), not under
  active load from prior briefs, and has a clear concern boundary
  (container lifecycle and BC orchestration). Adding manifest
  management extends that concern boundary naturally: a BC that
  orchestrates containers also orchestrates the list of BCs.
- `bc-container manifest sync` and `bc-container manifest validate` sit
  naturally alongside `bc-container launch --all` as preconditions in
  an operator's workflow.

**Cons:**
- `shopsystem-bc-launcher` is the newest BC (just created, as of
  2026-05-20). It has the least mature codebase of any product BC. Adding
  manifest management before the container lifecycle work (brief 004)
  is complete means the first work item is manifest tooling, not
  containers. This is fine architecturally but inverts the natural
  build order.
- Container lifecycle and manifest management are distinct concerns.
  A BC named `bc-launcher` that also manages the product manifest may
  broaden its scope beyond what the name suggests. A future `bc-launcher`
  BC in another product would be expected to do container lifecycle, not
  manifest management.

##### Option C — CLI owned by a new BC (`shopsystem-manifest` or similar)

A new BC whose sole product is the manifest CLI, keeping manifest
management independent of both bootstrap (templates) and container
concerns (bc-launcher).

**Pros:**
- Perfect separation of concerns. The manifest BC does one thing:
  manages the BC manifest. No scope dilution.
- Extensible: if the manifest grows (e.g., manifest-driven DB registry
  sync, manifest-driven CI pipeline generation), all of that lands in
  the manifest BC without negotiating scope.

**Cons:**
- Adds a seventh BC repo to a product that already has six. The manifest
  CLI is relatively thin (parse a file, clone repos, validate remotes).
  A new BC for this surface is likely under-weight — the launche BC
  was justified by a multi-subcommand surface (brief 004 has seventeen
  scenarios); the manifest CLI's initial scope is two operations.
- A new BC needs bootstrapping, its own role templates, its own beads
  remote, its own CI pipeline. That is significant overhead for a thin
  CLI.
- The product already has a BC that is the natural consumer of the
  manifest (bc-launcher). Separating the manifest CLI from its primary
  consumer means two BCs must coordinate on the manifest file format
  and schema — a cross-BC coupling that a single-ownership model avoids.

#### Decision

**Option B — CLI owned by `shopsystem-bc-launcher`.**

Rationale:

1. **The bc-launcher has the primary stake.** `bc-container launch
   --all` is the operational command that requires the manifest. If the
   launcher owns the manifest CLI, it can validate the manifest before
   operating on it — no cross-BC round trip. The dependency graph is:
   manifest file → bc-launcher manifest CLI → bc-container launch --all.
   All three hops are within the same BC.

2. **Scope extension is coherent.** A BC that "orchestrates BC
   containers" is also naturally the right owner of "orchestrates the
   list of BCs." The bc-launcher already commits to knowing which BCs
   exist (brief 004, scope item D: `bc-container list`). The manifest
   is the authoritative input to that list. Owning the manifest CLI
   closes the loop within the BC's existing concern.

3. **Avoids adding another BC.** Option C is the cleanest separation
   but is under-weight. Six repos is already a significant operational
   surface; a seventh for a two-operation CLI is not justified by the
   current scope. The bc-launcher can absorb manifest management without
   diluting its primary concern.

4. **Templates is over-committed.** Option A is thematically reasonable
   but adds to an already-loaded BC. The bc-launcher, being the newest
   and lightest BC, is the right recipient of new scope.

5. **Build order is acceptable.** The objection that manifest management
   inverts the bc-launcher's natural build order is real but minor.
   Scope item A (the manifest file itself) requires no BC work and can
   land first. The bc-launcher's manifest CLI (scope items B and C) can
   follow after the manifest file exists. Brief 004's container lifecycle
   scenarios can proceed in parallel — they do not depend on the manifest
   CLI.

**Manifest file path commitment:** The manifest file lives adjacent to
`CLAUDE.md` in the lead shop root (not under `.claude/`). Its filename
is `bc-manifest.<ext>` where `<ext>` is the Architect's format choice
after pre-state verification. Example: `bc-manifest.yaml`. This makes
it a first-class, visible artifact at the repo root — not buried in a
hidden directory.

**CLI surface commitment:** The bc-launcher CLI (`bc-container`) gains
a `manifest` subcommand group:
- `bc-container manifest sync` — clone-sync from manifest.
- `bc-container manifest validate` — validate manifest + drift detect.

Exact flag shapes are scenario-level, not PDR-level.

#### What this leaves open

1. **File format.** YAML vs TOML vs JSON. The Architect verifies what
   the bc-launcher's ecosystem already uses (Python? shell?) and picks
   the format that requires no additional parser dependency.
2. **Schema definition.** The PO commits three required fields (canonical
   name, GitHub remote, role label). The Architect defines the exact
   field names in the first scenario for scope item A.
3. **`--all` flag shape on bc-container launch.** Once the manifest
   CLI exists, `bc-container launch --all` reads from it. That flag's
   behavior is a brief 004 follow-on, not this brief's scope.
4. **DB registration integration.** If brief 006's DB registry lands
   before or alongside this CLI, `bc-container manifest sync --register`
   may be scoped in. Architect determines at pre-state.
