# PDR-034 — Legacy brief/PDR/ADR corpus migrates into the typed artifact system

**Note on this file's own form:** this PDR is itself authored in the typed
frontmatter convention it argues the rest of the legacy corpus should
adopt — deliberately, as the first instance of the target shape rather
than one more file this candidate's own execution will need to touch
later. Section structure below otherwise follows the existing PDR
convention (`pdr/032`) for continuity.

## The question

PDR-032 gave six artifact types (intent record, candidate, session
record, prioritization record, current-state, and the typedef/generator
mechanism itself) a YAML-frontmatter typed schema, but its own appetite
line explicitly excluded the legacy `brief/`/`pdr/`/`adr/` corpus from
that migration: "Explicitly OUT of appetite: legacy corpus migration
(prose-status → enum...)." Should that exclusion be lifted, and if so,
with what appetite and sequencing?

## Context

`cand-001`'s own solution sketch always named "migrated brief/PDR/ADR" as
part of the eventual vision for this system — the exclusion was a
deferred-appetite decision ("its own candidate"), not a judgment that the
migration was wrong or unnecessary. It has sat unrevisited since
2026-07-09.

An overnight corpus-validation pass (brief-019/`lead-5msa9`, run via the
newly-shipped `shop-knowledge validate` CLI) made the exclusion's actual
cost concrete: `shop-knowledge validate` run against all 129 files across
`adr/`, `pdr/`, `briefs/`, `candidates/`, `sessions/`, `intent/`, and
`current-state.md` reported 129/129 failures, split into two populations
— ~97 legacy files (this PDR's target) carrying zero frontmatter, and 13
correctly-typed newer files that instead diverge from the generated
typedef itself (a separate, already-filed finding, `lead-6n4j6`, not in
scope here). The 97 include files authored the same week this PDR is
being written (brief-022, ADR-064, ADR-065), confirming the split is
live and ongoing, not a closed historical artifact.

Raised directly by the product authority (`intent-006`): the exclusion
undermines the schema work's own purpose — progressive-disclosure and
citation/retrieval tooling built on typed metadata (`intent-004`'s theme)
can only be as complete as the corpus it indexes, and the material most
worth citing reliably (architecture and product decisions) is exactly
what's excluded today.

## Options considered

### Option A — Leave the exclusion in place

Legacy corpus stays on the prose-status convention indefinitely; only new
artifact types (and any future PDR/ADR/brief the product authority
explicitly asks to migrate one at a time) get typed frontmatter.
**Rejected** — directly contradicts the stakeholder's stated intent
(`intent-006`) and leaves the schema work's own stated purpose
unrealized for the corpus segment that most needs it.

### Option B — Full-corpus migration, single pass

One dispatch migrates all ~90 files at once, verified by the coherence
gate afterward. Buys fastest closure. Costs highest fidelity risk in one
shot, given legacy formatting has drifted somewhat since 2026-05; a
systemic issue found late is expensive to unwind across the whole batch
at once. **Not chosen** — see Decision.

### Option C — Full-corpus migration, phased gate-verified batches (chosen)

Migrate in batches (by artifact type — ADRs, then PDRs, then briefs, or
another Architect-proposed grouping), running `shop-knowledge validate`
after each batch before continuing to the next. Buys early detection of
legacy-formatting edge cases before they compound across the full corpus,
and independently reviewable increments. Costs more wall-clock/dispatch
rounds than a single pass; the corpus stays partially split for the
duration of the migration.

## Decision

**Option C.** Full-corpus appetite is ratified — `brief/`, `pdr/`, and
`adr/` all migrate onto the typed-schema mechanism ADR-059 already built,
preserving existing decision content verbatim and only restructuring the
status/metadata header. This is run in gate-verified batches rather than
a single pass, per the product authority's explicit choice weighing the
tradeoff above.

**Not decided here, routed onward per the PM/Architect boundary:**

- The exact batching grouping and migration mechanism (scripted/
  mechanical, subagent-driven semantic migration per file, or a hybrid)
  is an Architect feasibility probe on the real legacy-format variance,
  not a product-level decision.
- Whether existing typedef schemas already exist for `adr`/`pdr`/`brief`
  types (unconfirmed at this altitude — see `cand-004`'s Rabbit holes)
  or need to be authored first, as a precondition of any migration
  dispatch.
- Changelog backfill for files that never had one: reconstructed from
  git history where needed, flagged in the migrated file as backfilled
  rather than originally authored, since commit messages are not always
  decision-grade prose.

## Consequences

- Progressive-disclosure and citation/retrieval tooling (`intent-004`,
  `lead-x7bp`) gains a fully consistent corpus to index once this
  completes, rather than a six-type subset.
- The corpus remains visibly split during the batched migration; this is
  an accepted, bounded cost of the chosen sequencing, not a defect.
- Future PDRs/ADRs/briefs are authored directly in the typed convention
  going forward (this file is the first instance) rather than added to
  the legacy population and requiring later migration themselves.

### Scope / appetite (set by dstengle)

Full corpus (`brief/`, `pdr/`, `adr/` — all files), phased/gate-verified
batching. Batching grouping and migration mechanism delegated to an
Architect feasibility probe before dispatch. Content-fidelity is a hard
constraint, not subject to appetite trade-off.
