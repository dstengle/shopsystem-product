## Context

The canonical lead-primer instructs the Architect to verify pre-state by
reading and running BC code — `.claude/canonical/lead-primer.md:37-40`:
*"Reading the BC's code is hypothesis; running it is fact. Construct a
concrete input that exhibits ... the behavior; observe."* — and
`.claude/shop/primer.md:49-50` reinforces it for the lead's own work. The
only `repos/*` prohibition (`.claude/shop/primer.md:43`) forbids *edits*,
leaving reads and runs of sibling clones apparently-permitted. The router
acted on exactly that gap on 2026-05-29, instructing `lead-architect` to `cd`
into `repos/shopsystem-messaging` and `repos/shopsystem-scenarios` to read
and run BC code during reconciliation — the one thing BC isolation exists to
prevent.

PDR-011 (PO intent) diagnoses the conflation: "empirical" was read as "runs
BC code", when the property the clause actually wants is **demonstrated, not
assumed** — a property fully satisfiable against the contract/artifact surface
the lead already owns. Per stakeholder direction (dave, 2026-05-29), the lead
host carries **no BC source at all**: there is no `repos/` directory; BCs run
as `bc-launcher` containers (cloned inside the container, per ADR-004) and
report via `shop-msg`. There is nothing on the lead filesystem to read, run,
or git-observe. PDR-011 leaves four questions explicitly to the
Architect/ADR; this ADR pins the contract and resolves all four (D2–D5).

**Pre-state empirical verification (of this ADR's own load-bearing claim,
per D1).** That `scenarios hash` resolves and runs as an installed entry
point, with no clone, was demonstrated at the artifact level from the lead
CWD (`/workspaces/shopsystem-product`) without `cd`-ing into any clone and
without reading any BC source: `which scenarios` →
`/home/vscode/.local/bin/scenarios` (a `PATH` entry point, not a `repos/...`
path); the entry-point script imports `from scenarios.cli import main` (the
installed `scenarios` distribution, not a clone module); `importlib.metadata`
confirms a distribution named `scenarios` (version `0.1.0`) owning the
`scenarios` `console_scripts` entry point, and `pip list` shows
`scenarios 0.1.0`; `scenarios hash --help` exits 0, and piping contract text
(`Scenario: x / Given a / When b / Then c`) to `scenarios hash` from a
neutral cwd (`/tmp`) returns a hash (`adda2fc4596ff157`), exit 0 — the tool
transforms contract text the lead holds, touching no clone. Separately, `pip
show scenarios` / its `direct_url.json` confirm the `scenarios` distribution
is installed from a VCS pin
(`github.com/dstengle/shopsystem-scenarios @ v0.1.0`, commit `f91bbfe1`), NOT
editable from `repos/shopsystem-scenarios` — the concrete evidence that the
end-state install pattern the migration below targets is already reachable.

Three alternatives were considered and rejected. Option A — keep "running BC
code is fact" and merely scope the `repos/*` prohibition to edits-plus-reads
— was rejected because it preserves the premise PDR-011 identifies as wrong
(that the lead's pre-state is only knowable by running implementation), keeps
the lead coupled to BC internals it does not own (the coupling ADR-004 and
PDR-004 sever), and keeps a `repos/` directory that stakeholder direction has
removed. Option B — allow lead-side execution as a fallback only when a BC is
pre-bc-launcher (open question 3) — was rejected because a conditional "you
may run the clone if the BC isn't containerized yet" re-opens the exact gap
the router fell through and makes the doctrine depend on per-BC migration
state; the cleaner rule (D4) keeps lead-out-of-code unconditional and routes
missing evidence through `clarify`/`nudge` (D5) regardless of launcher
status. Option C — treat `scenarios hash` as BC-code execution and forbid it
too — was rejected because it would leave the lead unable to compute the
canonical hash it must cite when dispatching `assign_scenarios`/
`request_bugfix`, defeating the Architect's own hash-discipline obligation;
D2 draws the correct line (a contract tool over contract text is lead
toolchain, not BC implementation under test), and the empirical check
confirms the tool needs no clone.

## Decision

**D1 — "Verify pre-state empirically" = demonstrate against the
contract/artifact surface.** Hypothesis is still not fact; the demonstration
is still constructed, run, and cited in the dispatch description. Only the
admissible evidence changes. The lead may read, run, and cite as pre-state:
`features/` Gherkin in this repo (the canonical, PO-authored scenarios); this
repo's `adr/`, `pdr/`, `briefs/`; scenario hashes via the installed
`scenarios hash` CLI (see D2); message schemas; `shop-msg` mailbox state
(pending/read inbox & outbox, `watch`); and a BC's reported `work_done`
demonstration. The lead must NOT read or execute a BC's implementation to
establish its behavior. There is no BC code on disk to read, run, or
git-observe. Empirical proof that requires executing implementation is the
**BC's** job, surfaced through its gated Implementer→Reviewer loop and
reported via the mailbox.

**D2 — Contract-tool execution is installed-package invocation, not BC-code
execution (resolves PDR-011 open question 1).** A contract tool is a CLI
whose *input* is contract text and whose *output* is a contract fact.
`scenarios hash` is the canonical example: its input is Gherkin scenario
text, its output is the canonical hash that pins the scenario. Contract tools
are invoked as installed (pip) packages resolved on `PATH` — their resolution
and execution do not depend on, and must not reach into, any `repos/<bc>`
clone. Running a contract tool on contract text is categorically distinct
from executing a BC's implementation to observe *its* runtime behavior: a
contract tool is part of the lead's own toolchain (installed alongside
`shop-msg`, `bd`) and transforms contract text the lead already holds,
whereas executing a BC's implementation observes behavior the lead does not
own and must not couple to (BC-side empirical work, D1). The discriminator:
*does the tool's correctness belong to the lead's contract surface
(admissible) or to a BC's implementation under test (inadmissible)?*
`scenarios hash` is the former, empirically verified at the artifact level
(no `repos/`, no `cd` into any clone).

**D3 — Pure test-infra / non-scenario reconciliation (resolves PDR-011 open
question 2).** When a dispatched task is internal test infrastructure
(conftest, fixtures, test-DB plumbing, CI wiring — e.g. `lead-0bw`) there is
no scenario hash to reconcile against. The lead reconciles on (1) the BC's
reported demonstration carried in the `work_done` payload — the CI/pytest
result the BC's gated loop produced inside its container — and (2) mailbox
state, that the `work_done` landed against the dispatched `work_id`. The lead
must NOT re-run the suite, and (under D1) has no clone in which to do so. If
the reported demonstration is usable, reconciliation proceeds on it; if it is
thin or absent, D5 governs.

**D4 — Pre-bc-launcher transition (resolves PDR-011 open question 3).** For a
BC not yet migrated onto `bc-launcher`, there is no containerized execution
path and no source on the lead disk to fall back to (D1 is unconditional —
the no-`repos/` rule does not wait on bc-launcher). The lead relies entirely
on the BC's reported `work_done`; it never performs lead-side execution as a
stopgap. Reconciliation proceeds on reported evidence when that evidence is
present and usable; it does not *block* merely because the BC is pre-launcher.
Absent/thin evidence falls through to D5. This keeps the doctrine identical
across the transition: the only thing bc-launcher changes is *how* the BC
produces its proof, never whether the lead may reach for the proof itself.

**D5 — Thin/absent reported evidence: clarify/nudge, never self-produce
(resolves PDR-011 open question 4).** If a `work_done` carries no usable
demonstration, the lead sends a `clarify` (or a `nudge` with reason
`stuck-on-you`/`general` per ADR-015, when the gap is operational rather than
architectural) back to the BC for the missing proof. The lead must NOT reach
for the proof itself — not by cloning, not by re-running, not by inspecting
BC internals by any channel. This is the standing fallback for every
reconciliation path above (D3, D4) and for any scenario-bearing
reconciliation where the hash/register evidence is incomplete.

**D6 — The BC demonstrates; the lead reconciles.** Restating the posture the
four resolutions share: the BC produces its proof inside its container and
reports it via `work_done`; the lead joins that report against lead-side
artifacts (register, hashes via D2, ADRs/PDRs, mailbox state) and never
re-derives the proof. This is the lead-pre-state specialization of PDR-010's
transport posture and ADR-017's sovereignty boundary.

**Migration: `repos/` removal is gated on lead-bootstrap tooling, not
bc-launcher (not performed here).** This ADR does not delete `repos/`. The
blocker is not bc-launcher: the launcher is already contract-isolated from
host clones — `features/bc-launcher/02` pins that launch clones the BC repo
*inside* the container's workspace directory, and `features/bc-launcher/15` is
a pinned negative contract (the only bind mount inside a BC container is the
BC's own repo, with no sibling-directory mount), so the launcher neither needs
nor may use `repos/` and is not a removal blocker. The actual coupling is lead
tooling, pinned by `features/templates/67`: the lead-shop bootstrap installs
each `repos/<bc>` clone into the product venv in editable mode, so the lead's
own CLIs resolve to local clone source — that editable-from-clone install is
what keeps `repos/` physically present. The migration is therefore: switch the
lead's BC-CLI installs from editable-from-clone to VCS/published packages, then
supersede/re-ground scenario 67. This end state is concretely reachable — it
is already the pattern for the `scenarios` distribution (VCS-pinned, not
editable, verified via `pip show scenarios` / `direct_url.json`). The ADR pins
the doctrine (lead-out-of-code, no admissible BC-code reads/runs) immediately;
the filesystem migration that physically removes the clones, and the bootstrap
re-grounding of scenario 67, are downstream work named as a consequence, not an
action. Until that migration completes, the doctrine binds even though clones
remain physically present: the lead does not read or run them.

**Reconciliation discipline unchanged in shape, re-grounded in evidence.**
`lead-architect` reconciliation continues to confirm scenario register
landing and hash match — but the hash is computed via the installed contract
tool (D2), the register/coverage facts come from lead-held `features/` and
`shop-msg` mailbox state, and any implementation-behavior question becomes a
`clarify`/`nudge` to the BC (D5), never a clone read.

  - **`features/templates/93`, `92`, `94`, `95`** pin a `grep @scenario_hash`
    enumeration run *across the BC's `features/*.feature` tree* as the
    pre-state step. With no `repos/<bc>` on the lead host (D1), that grep
    target no longer exists on the lead side; these scenarios must be
    re-grounded so the BC `@scenario_hash` pre-state is established from the
    lead-held / mailbox-reported surface, not a clone grep. (Architect's call
    at authoring time whether to amend in place or supersede.)
  - **`features/templates/67`** pins that the lead *bootstrap* installs each
    `repos/<bc>` clone into the product venv **editable** (a `Then` step asserts
    the install is editable and resolves into `repos/<bc>/`). The `repos/`
    migration above contradicts that step — the end state installs the lead's BC
    CLIs from VCS/published packages, not editable-from-clone. Because 67 lives
    in the **same** templates/`shop-templates` bootstrap surface this
    `request_bugfix` already targets, and its revision is a behavioral tightening
    of pinned bootstrap behavior, **67's revision rides this same `request_bugfix`
    to `shopsystem-templates`** — not a separate dispatch. (Its scheduling is
    gated on the migration above, but the supersede/re-ground instruction is
    carried by the same vehicle.)
  - **`features/templates/22`** (lead CLAUDE.md routes mailbox access through
    `shop-msg`, never `repos/<bc>` paths) is **reinforced**, not contradicted,
    by this ADR — it is the corroborating pin for the `.claude/shop/primer.md:35`
    re-point.

  Because the change **tightens existing, scenario-pinned template behavior**
  (and supersedes at least scenario 08), it is a behavioral tightening:
  `request_bugfix`. It is not `request_maintenance` (that vehicle is for flat
  changes introducing no scenario tightenings — here scenario 08 et al. are
  directly contradicted). It is not `assign_scenarios` (the capability — primer
  prose describing empirical pre-state — already exists and is pinned).

- **Carried change.** The `request_bugfix` to `shopsystem-templates` carries:
  (a) the rewrite of the canonical lead-primer empirical-pre-state section to
  D1/D2; (b) superseding/amending scenarios 08 and 92–95 (and any sibling
  hash-enumeration scenarios) to the contract-surface grounding, plus
  re-grounding scenario 67 from editable-from-clone to VCS/published install
  (scheduled behind the `repos/` migration, carried by this same vehicle);
  (c) keeping the
  `.claude/agents/lead-architect.md` inline copy in lockstep (PDR-002). Each
  embedded scenario passes the `assign_scenarios` sufficiency check; the bugfix
  gets its **own fresh lead bead** as `work_id`.

- **Propagation back (PDR-003 §F).** Once the templates BC lands the canonical
  primer change as package data, it reaches this lead shop via
  `shop-templates update`, which overwrites `.claude/canonical/lead-primer.md`
  (canonical-managed) and the managed `.claude/agents/` files. The shop-local
  `.claude/shop/primer.md` edits are the lead's own direct change and do **not**
  propagate through the templates BC.

##### Migration: `repos/` removal is gated on lead-bootstrap tooling, not bc-launcher (not performed here)

This ADR does **not** delete `repos/`. The blocker is **not** bc-launcher: the
launcher is already contract-isolated from host clones. `features/bc-launcher/02`
pins that bc-container launch **clones the BC repo *inside* the container's
workspace directory**, and `features/bc-launcher/15` is a pinned **negative**
contract — the only bind mount inside a BC container is the BC's own repo, with
**no sibling-directory mount**; the launcher is contractually forbidden from
mounting host clones. So the launcher neither needs nor may use `repos/`, and is
not a `repos/`-removal blocker.

The actual coupling is **lead tooling**, pinned by
`features/templates/67`: the lead-shop *bootstrap* installs each `repos/<bc>`
clone into the **product venv** in **editable** mode, so the lead's own CLIs
(`shop-msg`, etc.) resolve to local clone source. That editable-from-clone
install is what keeps `repos/` physically present on the lead host — a
lead-tooling dependency, not a launcher one.

The `repos/` migration is therefore: **switch the lead's BC-CLI installs from
editable-from-clone to VCS/published packages, then supersede/re-ground scenario
67.** This end state is concretely reachable — it is already the pattern for the
`scenarios` distribution, which is installed from a VCS pin
(`github.com/dstengle/shopsystem-scenarios @ v0.1.0`, commit `f91bbfe1`), **not**
editable from `repos/shopsystem-scenarios` (verified at the artifact level via
`pip show scenarios` / its `direct_url.json` — see "Empirical verification").

This ADR pins the *doctrine* (lead-out-of-code, no admissible BC-code reads/runs)
immediately; the *filesystem migration* that physically removes the clones, and
the bootstrap re-grounding of scenario 67, are downstream work the ADR names as a
consequence, not an action. Until that migration completes, the doctrine binds
even though clones remain physically present: the lead does not read or run them.

##### Reconciliation discipline unchanged in shape, re-grounded in evidence

`lead-architect` reconciliation continues to confirm scenario register landing
and hash match — but the hash is computed via the installed contract tool (D2),
the register/coverage facts come from lead-held `features/` and `shop-msg`
mailbox state, and any implementation-behavior question becomes a `clarify`/
`nudge` to the BC (D5), never a clone read.
