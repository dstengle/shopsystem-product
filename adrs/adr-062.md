## Context

The fleet is converging from tmux onto fabro as the in-container BC
orchestration substrate (epic lead-14xb; ADR-048/050/058). During that
convergence, three separately-scoped `request_bugfix` slices each landed a
"Behavior 3" that pins CROSS-RUNTIME PARITY of an operator-facing behavior —
that an operator cannot tell tmux from fabro on that surface, and routes or
reconciles identically from either. In all three, the BC's
`mechanism_observation` reported the SAME emergent structural fact,
independently each time.

Pre-state empirical findings (artifact + installed-tool surface, ADR-018
D1/D2). No BC source read/run/git-observed on the lead host. Verified against
this repo's `features/` and the three BC-reported `mechanism_observation`
evidence paths (the BCs' own reported demonstrations, admissible per ADR-018
D1):

1. **Diagnostics (lead-01jw.3) — Behavior 3 pin
   `@scenario_hash:8af4e27a05ae9a32`**
   (`features/shopsystem-bc-launcher/fabro_diagnostic_blocked_work_done.feature:75`:
   "the fabro block work_done is as actionable as a tmux claude agent's
   clarify/block, so the operator reconciles or routes from either runtime
   identically"). BC-reported evidence: the failure-cause marker VOCABULARY is
   shared across runtimes — `agent-vault` is BOTH a tmux `CAUSE_MARKER_*`
   (reported `src/bc_launcher/diagnostics.py:94
   CAUSE_MARKER_AGENT_VAULT='agent-vault'`) AND the fabro `classify_reason`
   detail-marker, so an operator routes an agent-vault infra failure identically
   from either runtime. BC-reported: GREEN required NO production/workflow-
   mechanism change — the parity was ALREADY latent; the honest work was BINDING
   it (verbatim scenario + both-mechanism assertion tests).
2. **Liveness (lead-8hpz) — Behavior 3 pin `@scenario_hash:81eee7115a2457f4`**
   (`features/shopsystem-bc-launcher/fabro_liveness_heartbeat_parity.feature:51`:
   "the fabro-engaged BC's liveness signals match the tmux-engaged BC's, so the
   operator cannot distinguish runtime from the liveness surface"). BC-reported
   evidence: the ONLY production added is a shared NAMING ANCHOR
   (`src/bc_launcher/liveness.py`) that BOTH call sites (reported
   `src/bc_launcher/fabro/engage.py:hb_verb` and
   `src/bc_launcher/cli.py:DEFAULT_STARTUP_PROMPT_TEMPLATE`) now source the
   heartbeat verb from, with BYTE-IDENTICAL recorded output. BC-reported: GREEN
   is the BINDING itself — the RED failed only because the shared anchor did not
   yet exist, not because parity was missing.
3. **Transient resilience (lead-6ev8) — Behavior 3 pin
   `@scenario_hash:acd8d90bd9d4e4df`**
   (`features/shopsystem-bc-launcher/fabro_llm_node_transient_error_resilience.feature:74`:
   "the fabro LLM path's transient-error resilience matches the tmux claude
   agent's … the operator sees the same completion outcome regardless of
   runtime"). BC-reported evidence: GREEN is a shared parity-pin anchor
   (`src/bc_launcher/transient_resilience.py`) — the direct analog of
   lead-8hpz's `liveness.py` — that names the ONE retry/backoff contract both
   runtimes are read through so they cannot silently diverge and let a transient
   rate-limit decide completion BY runtime. BC-reported: `workflow.fabro` is
   byte-unchanged by this behavior (`git diff 4e04ffb..HEAD` touches only the
   feature file, the parity test, and the new pure-Python anchor module; 0
   deletions).
4. **The three surfaces are structurally IDENTICAL, confirmed independently.**
   Each Behavior-3 was a PIN-EXISTING-CAPABILITY case where the operator-facing
   parity was ALREADY latent (delivered by that slice's Behaviors 1&2 plus the
   other runtime's own pre-existing mechanism), and GREEN was the BINDING to a
   single shared Python anchor module both runtimes source from — NOT a change
   to either runtime's mechanism. Three independent confirmations of one
   principle is the threshold for first-class codification.

## Decision

**D1 — Cross-runtime operator-facing parity MUST be bound via a SINGLE SHARED
Python anchor module both runtimes source from.** For any OPERATOR-FACING
behavior that must be identical across agent-runtimes (today tmux ⇄ fabro), the
parity is bound by a SINGLE SHARED Python anchor module that BOTH runtimes' call
sites SOURCE FROM — not by two independently maintained implementations that
happen to agree today. The three confirmed anchors are:

- **`bc_launcher/diagnostics.py`** — the failure-cause marker VOCABULARY (e.g.
  `agent-vault` as both a tmux `CAUSE_MARKER_*` and a fabro detail-marker).
- **`bc_launcher/liveness.py`** — the heartbeat VERB.
- **`bc_launcher/transient_resilience.py`** — the retry/backoff CONTRACT.

The anchor is the load-bearing COUPLING POINT: it is the single place the
operator-facing vocabulary/verb/contract is defined, and both runtimes read
their operator-facing behavior THROUGH it.

**D2 — Behavior-3 parity green is the BINDING to the anchor, not a
runtime-mechanism change.** When a parity behavior is pinned and the capability
is already latent on both runtimes, the honest production change is the SHARED
ANCHOR plus the both-mechanism assertion test — NOT a change to either runtime's
mechanism. The RED-before-GREEN exists because the shared anchor did not yet
exist, not because parity was missing. This is the discriminator that keeps a
parity pin from being mistaken for a net-new capability build: parity-over-latent
capability is a BINDING, and its teeth are the shared anchor plus a test that
exercises BOTH real mechanisms.

**D3 — The INVARIANT: a new agent-runtime, OR a change to one runtime's
mechanism WITHOUT updating the shared anchor, REDs the parity pin.** The anchor
is what STOPS the two runtimes silently diverging and letting the runtime decide
operator-facing outcomes. Therefore:

- **Adding a new agent-runtime** (a third orchestration substrate beyond tmux
  and fabro) requires wiring that runtime's operator-facing behavior THROUGH the
  same shared anchor; a new runtime that re-implements the vocabulary / verb /
  contract independently violates this ADR.
- **Changing one runtime's mechanism** (marker vocabulary, heartbeat verb, or
  retry/backoff contract) WITHOUT updating the shared anchor REDs the parity pin
  by construction — the both-mechanism parity test fails, which is the intended
  teeth. The anchor and the parity test together are the coupling that makes
  divergence LOUD, not silent.

This is the load-bearing property: operator-facing outcomes must be decided by
the SHARED CONTRACT, never by which runtime happened to run.

**Decision-coherence check (no contradictory / superseded active decision).**
Checked against active decisions: ADR-050 (parity as posture), ADR-058
(lifecycle parity), ADR-057 (single-sourced loop projections). NONE of them
specifies a COMPETING mechanism for how per-behavior operator-facing parity is
bound — they decide WHAT is at parity, not the shared-anchor binding this ADR
names. This ADR is additive and anchoring; no active decision asserts parity is
achieved by independently-maintained per-runtime implementations, so there is no
contradiction to reconcile.

## Consequences

- Cross-runtime operator-facing parity is now a NAMED, checkable mechanism (the
  shared anchor module), not an emergent coincidence re-discovered per slice.
  The three confirmed anchors (`diagnostics.py`, `liveness.py`,
  `transient_resilience.py`) are the concrete precedent.
- The discriminator in D2 protects future dispatches from mis-vehicling a parity
  BINDING as a net-new capability: a parity-over-latent-capability slice is a
  `request_bugfix` that pins the parity and adds the shared anchor, NOT an
  `assign_scenarios` for a "new" behavior.
- The invariant in D3 gives the reviewer/router a durable expectation: any
  future runtime addition or one-runtime mechanism change is inspected for
  whether it flows through the shared anchor; if it does not, the parity pin is
  expected to RED.
- NOT release-blocking. This ADR records a decision confirmed by already-merged
  work; it changes no pinned coverage and retires no `@scenario_hash`.

**`@scenario_hash` impact — NONE retired.** This ADR retires, supersedes, or
contradicts NO pinned coverage. It ANCHORS ON and describes the mechanism behind
three already-green parity pins (`8af4e27a05ae9a32`, `81eee7115a2457f4`,
`acd8d90bd9d4e4df`), which stand unchanged. Retirement set = ∅.

**Follow-ups (named, not designed here).** Coherence-gate check (tracking bead —
filed, NOT release-blocking): a future decision-coherence-gate check can enforce
D3 mechanically — flag any runtime-mechanism change (marker vocab / heartbeat
verb / retry contract) whose diff does not also touch the corresponding shared
anchor module. Filed as a tracking bead; scoped as advisory, deferred behind the
coherence-gate CLI productization (lead-iohr).
