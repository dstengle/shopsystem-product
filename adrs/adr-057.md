## Context

The BC work-loop CONTENT is dual-homed and exposed to prose drift. The fabro
loop def — owned by `shopsystem-bc-launcher`, baked into bc-base and
launch-wired to `/workspace/.fabro/` — carries its OWN node-body copies of the
six BC role prompts (`bc-implementer`, `bc-reviewer`, `bc-review`,
`bc-router`, `bc-sufficiency-check`, `work-done-gate`) AND the vendored skills
(`test-driven-development`, `subagent-driven-development`,
`using-git-worktrees`, `writing-plans-bdd`, `integrating-to-main`) — the SAME
content `shopsystem-templates` owns as the tmux-path role templates + vendored
skills. Changing a role prompt or a skill today requires editing TWO homes:
the templates (tmux engage) AND the fabro def node bodies (fabro engage).
Worse than a textual dupe: the fabro node bodies are a DIALECT TRANSLATION
(Markdown skill/shim prose → fabro-DOT graph nodes with `prompt=`/`script=`,
skills INLINED as node prose), so a mechanical diff cannot catch drift. This
ADR raises a SOURCE OF RISK (prose drift), so its resolution must itself be
minimal and low-risk.

Why the fabro def exists is SETTLED (ADR-050/051) and not re-litigated: the
fabro *engage tier* REPLACES the tmux/claude engage with a headless `fabro run
workflow.fabro` run-graph. Engage is a launch concern, so bc-launcher owns it
— correct for the graph TOPOLOGY/mechanism. The conflation is that the def
bundle also swept in the role-prompt + skill CONTENT.

The resolution is FORCED BY CONTRACT: pinned scenario
`@scenario_hash:2dfefe2ba81e418d` (bc-launcher,
`bc_container_fabro_def_validates.feature`, @origin adr-051) requires the def
be self-contained and *"runnable from the def alone"*, which structurally
mandates the node bodies sit alongside the graph file. Resolving the drift
therefore REQUIRES reopening that pin's DELIVERY premise (baked → poured)
while PRESERVING its VALIDITY assertions.

Pre-state, verified empirically (ADR-018 D1/D2) against this repo's `features/`
Gherkin + `adr/`/`pdr/` + the `findings/bc-workloop-single-source/` spike
outputs. No BC source read, run, or git-observed (the lead host carries none):

- **The `.claude/` pour mechanism (the model to mirror):** bc-base bakes the
  `shop-templates` BINARY from a VCS pin (`@scenario_hash:ccb145d71c7100a2`);
  `bc-container launch` runs the `shop-templates` pour INSIDE the container
  workspace after clone, emitting `.claude/skills/`
  (`@scenario_hash:75ae95be0ecf1640`); `--workspace-mount` SKIPS the pour and
  uses the committed tree byte-unchanged. So `.claude/` changes WITHOUT an
  image rebuild: the binary is baked; the CONTENT is package data the binary
  emits at pour time. This is exactly the mechanism `.fabro/` will mirror.
- **The fabro def is delivered the OLD way:** a bundle at
  `src/bc_launcher/assets/fabro-def/`, BAKED into bc-base, launch-wired to
  `/workspace/.fabro/` (`@scenario_hash:2dfefe2ba81e418d`). This baked-bundle
  delivery is what this ADR rejects.
- **Self-containment is a DIRECTORY property at RUN time, not a bake
  property.** `2dfefe2ba81e418d` requires every referenced node body sit
  alongside `workflow.fabro` so the loop runs from the def alone. A def
  GENERATED into `/workspace/.fabro/` at pour time — before `fabro run` — is
  equally self-contained and equally `fabro validate`-green. Only the def's
  DELIVERY premise ("on the pinned bc-base image") must change; the validity
  holds.
- **Runtime binary vs workflow content is already a clean seam in the pins.**
  fabro v0.254.0 + anthropic-oauth-shim are BAKED executables
  (`@scenario_hash:a3512aedb8763150`, companion `4fc67c610cba6227`) — runtime,
  correctly baked. The `workflow.fabro` topology + node bodies are workflow
  CONTENT — this is what moves from baked to poured.

Spike evidence (`findings/bc-workloop-single-source/02-oq1-generation-spike.md`,
ADR-018-admissible) prototyped a deterministic generator over the UNCHANGED
role/skill Markdown + a thin manifest:

- **Determinism: PASS (byte-identical).** Generator run twice → identical
  `sha256` on every artifact (e.g. `impl.node.dot` =
  `6fc981bc97f617f3ce81acb2660078b7dbb3fb080ab0bcf370ab3a35c156b43f`; combined
  tree `aa915c25a4045b810b60d96f7a32e1327d34cae0ad0fb83b4e380d74f65d3723`).
  Deterministic by construction (sorted iteration, no timestamps, no
  randomness).
- **`fabro validate`: PASS on the REAL binary.** `fabro 0.254.0 (497aaba)` on
  an ADR-051-shaped skeleton assembled from generated nodes: `Validation: OK`,
  `EXIT_CODE=0`.
- **Agent nodes: existing Markdown is sufficient as-is.** A 10-line manifest
  block drove inlining of 23,639 chars of UNCHANGED source prose into a
  gate-complete, validate-OK node body; only 3 mechanical text transforms
  fired across the whole body. The prose was NOT restructured.
- **Native-gate nodes: executable content is NOT derivable from prose** —
  English gate prose does not compile to a bash pipeline. ADR-051 D3 already
  mandates native `script=` for gate/state-change nodes, and those scripts +
  the topology are the STATIC skeleton (authored once), not generated. This is
  precisely why D4 pours them verbatim rather than generating them.

## Decision

**D1 — One canonical source (UNCHANGED authoring surface).** The single
canonical source of the BC work-loop CONTENT is the EXISTING
`shopsystem-templates` role prompts (`bc-implementer`, `bc-reviewer`,
`bc-router`, `bc-review`, `bc-sufficiency-check`, `work-done-gate`) + vendored
skills, UNCHANGED as the authoring surface. No prose is restructured, no new
"kind" taxonomy is introduced (see OUT OF SCOPE). The `.claude/` projection
continues to pour them exactly as today.

**D2 — Two poured projections, deterministically.** The `shop-templates` pour
emits BOTH surfaces: `.claude/` (tmux engage) — unchanged; and `.fabro/`
(fabro engage) — NEW, generated at POUR time into `/workspace/.fabro/`, NEVER
baked into bc-base. Generation MUST be DETERMINISTIC: identical single source
⇒ byte-identical `.fabro/`. This is the no-drift property that makes a
committed `.fabro/` provably equal to a fresh pour (the direct analogue of the
ADR-019 scenarios single-source doctrine and the progressive-disclosure
byte-identical precedent). `.fabro/` is FIRST-CLASS and committable exactly
like `.claude/`: a repo MAY commit its poured `.fabro/`, and a
`--workspace-mount` launch uses it byte-unchanged (pour skipped, as
`.claude/skills` is treated today).

**D3 — DIRECT translation only (no restructuring).** The agent-node bodies are
generated by inlining the UNCHANGED template + skill Markdown, applying ONLY
the mechanical transforms catalogued and proven in the spike:

1. `Skill`-invocation → the named skill body INLINED as node prose;
2. `model:` (e.g. `inherit`) → node `class=` tier;
3. `Task`/subagent fan-out → `parallel=true`;
4. `tools:` → `permissions=`;

plus the per-node `lane` scoping directive (the one thin, hand-authored line
that encodes a graph-topology fact absent from the role prose — "this node is
the implementation step ONLY; MUST NOT push/emit/consume inbox"). The prose is
NOT restructured. No cleverness; no compression pass (the spike's 5.4×
verbosity is a token-cost optimization, explicitly NOT a validity blocker and
NOT in scope).

**D4 — Gate scripts + topology skeleton poured VERBATIM from a static asset;
gate/claude-prompt unification DEFERRED.** The native-gate `script=` nodes AND
the ADR-051 topology skeleton (23 nodes / 45 edges, `emit_r` sole emitter,
native-`script=` vs agent-node split, fail-closed `outcome=failed` edges, plus
`workflow.toml` / `project.toml` / `vaults/default` scaffold with
`__PLACEHOLDER__`-only native vault) are poured VERBATIM from a STATIC asset —
authored ONCE, exactly as the spike did — NOT generated from prose and NOT
unified with the working claude prompts now. Unifying the fabro gate scripts
with the working claude prompts would touch working prompts = risk; this ADR
raises a risk topic, so that unification is DEFERRED to a later iteration. The
ONLY per-node distinction the generator needs is agent-vs-native, and that is
ALREADY structural in the ADR-051 skeleton (native nodes carry `script=`;
agent nodes carry a generated body).

**D5 — Runtime vs content seam.** The fabro binary v0.254.0 +
anthropic-oauth-shim stay BAKED (runtime executables,
`@scenario_hash:a3512aedb8763150` / `4fc67c610cba6227` — unchanged). The
workflow CONTENT (the ADR-051 topology skeleton + the generated node bodies)
is POURED. Runtime is baked; content is poured.

**D6 — Ownership.**

- **MOVE to `shopsystem-templates`:** `.fabro/` node-body GENERATION (the
  dialect translation from single source) AND the ADR-051 topology skeleton
  static asset. ADR-051 REMAINS the governing contract (its invariants
  constrain the skeleton wherever it lives); a NEW templates-side conformance
  scenario pins that the poured skeleton still conforms to it. One home for
  the whole emitted def is forced by self-containment: if templates generated
  the bodies but bc-launcher still baked the skeleton, the pour would have to
  inject bodies into a bc-launcher-baked skeleton — re-introducing the
  two-home coupling and re-baking half the def.
- **STAYS `shopsystem-bc-launcher`:** the launch/engage WIRING — same `fabro
  server start --foreground --no-web`, same `fabro run workflow.fabro -I
  BC_NAME=… -I WORK_ID=…`, same base_url / oauth-shim wiring — now sourcing
  the def from the POURED `/workspace/.fabro/` rather than a baked bundle. The
  baked `src/bc_launcher/assets/fabro-def/` bundle is RETIRED.

**Explicitly OUT OF SCOPE (named as possible future iterations, NOT decided
here):**

- **NO lead-x7bp coupling.** `lead-xinb`, `lead-4qqn`, and `lead-x7bp` are
  INDEPENDENT; x7bp stays parked/focused and is not pulled in here.
- **NO source-unit "kind" taxonomy.** The manifest references source units by
  plain NAME (`role=bc-implementer`, `skills=[…]`); no taxonomy. The only
  distinction the generator needs is agent-vs-native, already structural in
  the ADR-051 skeleton (at most an implicit `agent|native` flag).
- **NO obligation/activation/composition metadata** (MUST/SHOULD/composable).
- **NO gate-script / claude-prompt unification** (per D4 — deferred).

**Reopens `@scenario_hash:2dfefe2ba81e418d`.** The delivery premise moves baked
→ poured: from "the def bundle under `src/bc_launcher/assets/fabro-def/` …
present in that running container on the pinned bc-base image" to "the def
POURED by `shop-templates` into `/workspace/.fabro/` at launch." The VALIDITY
assertions — `fabro validate` exit 0 / zero diagnostics, the ADR-051 graph
invariants (`emit_r` sole gated emitter, fail-closed failsafe edges), the
`__PLACEHOLDER__`-only native vault — all HOLD, and RE-HOME to a new
`shopsystem-templates` conformance scenario (where the generation now lives).
Editing the scenario body changes its hash → the PO re-authors; the eventual
vehicle is `request_bugfix` (correcting an existing pinned behavior's premise).

## Consequences

- **Drift killed on the big surface with minimal change.** Role/skill PROSE is
  single-sourced in templates; `.fabro/` agent bodies become byte-stable
  generated projections. The gate scripts remain the small static skeleton
  (unification deferred). No working prompt is touched.
- **`.claude/` and the runtime binaries are untouched.** The change is
  additive: a second projection out of the same pour, plus a static-asset
  relocation.
- **Self-containment is preserved.** The poured `.fabro/` is a complete def
  directory at run time; `fabro validate` and `fabro run` see no difference
  from the baked bundle except its provenance.
- **Residual risks carried forward from the spike:**
  - **R1 — source resolution.** A real generator needs a defined source-unit
    resolution path for the skills not poured on the lead host
    (`shop-templates` skill export or a canonical skills package). This
    REINFORCES templates ownership (D6): templates already owns and can
    resolve every source unit.
  - **R2 — validate ≠ preflight.** `fabro validate` is permissive on node
    attrs (confirms graph shape, not handler classification). The N3
    conformance pin SHOULD use a live `fabro run` preflight where feasible,
    not `validate` alone, to authoritatively assert the agent-vs-native
    classification.

#### Consequences

- **Drift killed on the big surface with minimal change.** Role/skill PROSE is
  single-sourced in templates; `.fabro/` agent bodies become byte-stable

- **Residual risks carried forward from the spike:**

    (confirms graph shape, not handler classification). The N3 conformance pin
    SHOULD use a live `fabro run` preflight where feasible, not `validate` alone,
