## Source (pre-modernization)

#### Decision

A new Bounded Context, `shopsystem-bc-launcher`, is created to own the `bc-container`
command surface. It is not folded into any existing BC.

The `bc-container` CLI it exports provides: `launch`, `attach`, `inject`, `monitor`,
`stop`, `status`, and `list` subcommands (brief 004 scopes A, B, and D). It is also
the owner of shared-network connectivity for BC containers (scope C): when launching a
container, `bc-container launch` sets the `SHOPMSG_DSN` environment variable inside
the container so that `shop-msg` running in the container connects to the same
PostgreSQL instance as the host.

#### Alternatives considered

PDR-004 documented four options; the two strongest alternatives were:

**Option B — `shopsystem-devcontainer`**: The devcontainer BC already owns the image
that `bc-container launch` uses. "Image + launcher" is a common co-location pattern.
Rejected because the devcontainer BC's concern is building and publishing an image;
container lifecycle orchestration (tracking running containers, tmux session management,
inject/monitor surface) is a distinct operational subdomain. The names are also
misaligned: a BC named `devcontainer` owning `bc-container launch` misleads future
maintainers. The devcontainer BC is also under active load with in-flight scenarios,
and scope addition during an active implementation cycle risks collision.

**Option C — `shopsystem-templates`**: The templates BC owns the bootstrap CLI
(`shop-templates`). Bootstrap and container-launch are both "getting a BC into a known
good state" operations. Rejected because `shopsystem-templates` is already carrying
PDR-001 restructure, PDR-003 CLAUDE.md propagation, brief 002 bootstrap surface, and
brief 003 event-driven activation — priority dilution is a real risk. Docker and
container orchestration concerns are not obviously a templates domain; the naming
(`shop-templates bc-container launch`) hides the container concern.

**Option D — `shopsystem-messaging`**: The messaging BC owns the inter-shop surface;
mailbox volume management could be its concern. Rejected because the messaging BC's
domain is message schemas and routing, not container lifecycle. Adding Docker
orchestration to `shop-msg` is a large scope increase; `shop-msg container launch`
is a category error in CLI naming.

#### Rationale

1. **Domain cleanliness.** Container lifecycle for BC shops — starting Docker containers,
   running container-init sequences, managing tmux sessions, tracking lifecycle state —
   is a distinct subdomain not owned by any existing BC. None of image-building,
   message routing, template management, or scenario registration is the natural home.
   A new BC draws the right boundary.

2. **Weight justification.** The MVP (docker run orchestration + tmux session management)
   is thin, but the full scope of brief 004 — attach/inject/monitor surface, lifecycle
   status tracking, shared-network connectivity, no-sibling-mount isolation, idempotent
   launch — is substantial enough to justify its own repo and scenario register.

3. **Avoids scope addition to loaded BCs.** Templates, devcontainer, and messaging are
   all under active development. A new BC starts with a clean slate, its own inbox,
   its own CI pipeline.

4. **Self-documenting name.** `shopsystem-bc-launcher` names the subdomain directly.
   The CLI it exports (`bc-container`) is short, tab-completable, and distinct from
   existing CLIs (`shop-msg`, `shop-templates`, `bd`).

#### Env var authority

The correct env var for shop-msg's PostgreSQL connection is `SHOPMSG_DSN`, not
`SHOP_MSG_DB_URL`. Empirically verified: running `SHOPMSG_DSN=<bad-dsn> shop-msg read
inbox ...` produces a connection error; running `SHOP_MSG_DB_URL=<bad-dsn> shop-msg
read inbox ...` has no effect (tool connects normally). The source is
`repos/shopsystem-messaging/src/shop_msg/storage.py` line 62:
`os.environ.get("SHOPMSG_DSN", _DEFAULT_DSN)`. PO scenarios 13 and 14 use
`SHOP_MSG_DB_URL` and must be corrected to `SHOPMSG_DSN` before dispatch
(tracked: lead-gug).

#### Consequences

1. **New repo to bootstrap.** `repos/shopsystem-bc-launcher/` does not yet exist (verified:
   `ls repos/` returns five entries, no `shopsystem-bc-launcher`). Before any
   `assign_scenarios` dispatch can land, the GitHub repo must be created, bootstrapped
   with `shop-templates`, and cloned into `repos/`. This is the gating blocker for all
   17 bc-launcher scenarios (tracked: lead-mxy).

2. **New dispatch surface.** Once the BC exists, all 17 brief 004 scenarios dispatch to
   `shopsystem-bc-launcher` via `assign_scenarios`. Scopes A (scenarios 1-5, 16, 17),
   B (6-8), D (9-12, 15) can dispatch immediately once the BC is bootstrapped. Scope C
   (13-14) requires PO vocabulary correction first.

3. **Scope C is a bc-launcher concern, not a messaging concern.** The shopsystem-messaging
   BC already reads `SHOPMSG_DSN` from the environment (behavior pinned in part by
   `watch.feature` scenario `dd52b41c28f2ab14`). No new dispatch to messaging is
   required for scope C. The launcher is responsible for setting `SHOPMSG_DSN` in the
   container environment at launch time.

4. **No implementation in this repo.** As with all BCs, code lives in the BC repo.
   The lead shop's role is to author scenarios and dispatch; `assign_scenarios` is the
   vehicle once the BC is bootstrapped.
