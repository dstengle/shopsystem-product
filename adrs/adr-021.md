## Source (pre-modernization)

#### Context

`bc-container launch` (the `shopsystem-bc-launcher` BC) runs every BC container
from a single base image. That image — `shopsystem-bc-base` — carries the
framework CLIs (`shop-msg` / `scenarios` / `shop-templates`, plus `bd`) that a
BC needs the moment it boots. But the image has **no owner, no build/publish
contract, and no rebuild trigger** anywhere on the contract surface. When a
framework utility ships a fix — e.g. the `lead-pw41` canonicalization fix landed
in `shopsystem-messaging` source — newly-launched BCs keep booting on a stale
base image that predates the fix. This is the same drift class as `lead-xq0`
(the lead's venv installs go stale after BC work lands), surfacing on the
container-image axis instead of the lead-venv axis.

The user has settled two framing decisions (recorded on `lead-c2kp`, 2026-06-02)
that this ADR records rather than re-opens:

1. **Ownership** = `shopsystem-bc-launcher` owns, builds, and publishes the
   bc-base image (Dockerfile + publish CI co-located in the bc-launcher repo).
2. **Propagation** = **auto-propagate** for now (active development): any
   framework-utility release auto-rebuilds bc-base and republishes `:latest`,
   so newly-launched BCs get current CLIs; rollback is the interim mitigation if
   a bad release lands. Controlled/pinned releases are **explicitly deferred**
   future work — recorded here, not designed.

This is a mechanism / operations decision with no product-UX change, hence an
**ADR**, not a PDR.

##### Pre-state empirical findings (artifact surface, ADR-018 D1/D2)

Verified from the lead CWD against the contract/artifact surface only — no BC
implementation code read, run, or git-observed.

1. **bc-launcher is the sole consumer and hardcodes the `:latest` base image.**
   The installed `bc_launcher` distribution in the product venv records
   `BC_IMAGE = "ghcr.io/dstengle/shopsystem-bc-base:latest"`
   (`bc_launcher/controller.py:32`), consumed at the container-run call
   (`controller.py:388`, `image=BC_IMAGE`). This is the installed-dist metadata
   surface (the same admissible surface ADR-019's pre-state used), not a
   `repos/` clone. **Confirmed.**
2. **bc-base is undocumented on the contract surface.** `bc-base` /
   `shopsystem-bc-base` appears in **no** `features/`, `adr/`, `pdr/`, or
   `briefs/` file, and there is no owner BC for it in `features/bc-manifest/`.
   **Confirmed** (grep returns no matches).
3. **Two existing image-CI patterns to mirror, both same-repo:**
   - `features/devcontainer/12–17` — a workflow that **triggers on a push to
     `main` touching `.devcontainer/Dockerfile`** (scenario 13), **builds** the
     Dockerfile (scenario 15), **pushes to `ghcr.io/dstengle/shopsystem-devcontainer`**
     (scenario 16), with **`:latest` + a `sha-<short-sha>` tag** (scenario 17).
   - `features/templates/142–145` — on a **`vMAJOR.MINOR.PATCH` git-tag push**,
     build and publish to `ghcr.io/dstengle/shopsystem-templates` with the
     **version string + `latest`** tags, **public visibility** (scenario 142).
   Both triggers originate **in the repo that owns the Dockerfile**. The
   rebuild-on-utility-release this ADR decides originates **in a different repo**
   (the utility) — there is no cross-repo trigger pattern anywhere on the
   contract surface. `repository_dispatch` / `workflow_dispatch` appear in no
   `features/`/`adr/`/`pdr/` file. **Confirmed novel.**
4. **What a "framework-utility release" means is already pinned.** Scenario 129
   establishes that framework CLIs install from a **VCS/published-package pin**
   in the shape `github.com/dstengle/shopsystem-scenarios @ v0.1.0`, not
   editable-from-clone. A "release" is therefore a version-tag push on a utility
   repo — the same `vMAJOR.MINOR.PATCH` shape scenario 142 already keys a publish
   on. **Confirmed** as the release convention on the contract surface.
5. **Launch does not pull a fresh image.** The launch contract
   (`features/bc-launcher/01-launch-starts-container.gherkin`) says only that a
   container *runs*; no scenario across `features/bc-launcher/` pins a
   pull-before-run of `:latest`, and the installed-dist run call (`controller.py:388`)
   is a plain container run against `BC_IMAGE` with no pull step. So a launch
   today would use whatever `:latest` digest is in the **local Docker cache**,
   which may predate the republish. **Confirmed** that freshness-at-launch is an
   unpinned gap, not existing behavior.

##### What could NOT be verified (asserted, not confirmed)

- **The bc-base Dockerfile contents** — how it currently installs the CLIs,
  from which pins, on which base image — are **not on the lead host**. Per
  ADR-018 there is no `shopsystem-bc-base` repo (and no bc-launcher repo)
  cloned here; the Dockerfile cannot be read, built, or git-observed from the
  lead. This ADR therefore decides the Dockerfile's **co-location and behavioral
  contract**, not its line contents.
- **The utility repos' actual release-workflow YAML** (whether messaging /
  scenarios / shop-templates today fire on tags, and what their CI emits) is
  off-surface for the same reason. Scenario 129 and scenario 142 ground the
  release *convention* (version-tag push); the per-repo workflow realization is
  a BC-scenario detail, not a lead-verifiable fact.
- **Whether the bc-base build can pull bc-launcher itself** depends on
  `lead-aj5f` (bc-launcher repo is private; the other three utility repos are
  public). This is named as a credentialing constraint below, not resolved here.

#### Decision

##### D1 — Ownership: the bc-base Dockerfile and its publish workflow live in `shopsystem-bc-launcher`

The `shopsystem-bc-base` image is owned by `shopsystem-bc-launcher`. Both the
**Dockerfile** and the **publish CI workflow** are co-located in the bc-launcher
repository. bc-launcher is already the sole consumer (`controller.py:32`), so
ownership sits with the consumer that hardcodes the reference; there is no
separate bc-base repo or owner BC to introduce.

The image is published to **`ghcr.io/dstengle/shopsystem-bc-base`** with a
**version tag + `latest`**, **public visibility** — mirroring the
templates-142 publication contract verbatim, applied to the new image path. The
version tag is the bc-launcher release version (the build is bc-launcher's own
CI artifact), and `latest` is the floating tag `controller.py:32` resolves.

##### D2 — Cross-repo auto-rebuild mechanism: utility releases fire a `repository_dispatch` to bc-launcher's build workflow

On a **framework-utility release** (a version-tag push on a utility repo, per
the scenario-129 / scenario-142 release convention), that utility repo's release
workflow **emits a `repository_dispatch` event to the `shopsystem-bc-launcher`
repository**, and bc-launcher's build workflow — listening for that
`repository_dispatch` event type **and** its existing same-repo triggers —
rebuilds bc-base (installing the current utility versions) and **republishes
`:latest`**.

**Trigger-EMIT side vs BUILD side:**

| repo | role | what it carries |
| ---- | ---- | --------------- |
| `shopsystem-messaging` | trigger-EMIT | a step in its release workflow that fires a `repository_dispatch` to bc-launcher on a version-tag release |
| `shopsystem-scenarios` | trigger-EMIT | same |
| `shopsystem-templates` (`shop-templates`) | trigger-EMIT | same |
| `shopsystem-bc-launcher` | **BUILD** (+ trigger-EMIT for its own release) | the Dockerfile, the build/publish workflow listening for `repository_dispatch` and the same-repo Dockerfile-push trigger, and the launch-pull behavior (D3) |

bc-launcher's own release is also a reason to rebuild bc-base (its own version
tag is the image's version tag, D1), so bc-launcher carries both a same-repo
trigger (its own release) and the cross-repo listener (the three utilities).

**Behavioral outcome pinned:** on any framework-utility release, a new bc-base
image incorporating that utility version is built and `ghcr.io/dstengle/shopsystem-bc-base:latest`
is republished to point at the new digest.

**Why `repository_dispatch` over the alternatives:** it is event-driven, so a
fix propagates promptly (the `lead-pw41` "fix landed but isn't live" symptom is
the cost of *latency*, which a scheduled rebuild reintroduces); the
trigger-emit side is a single small step added to a workflow each utility repo
already runs on release; and the build authority stays in one place
(bc-launcher), so there is exactly one Dockerfile and one publish path. See
Alternatives for the scheduled-rebuild and pin-bump-PR options rejected.

##### D3 — Freshness at launch: `bc-container launch` pulls `:latest` rather than using the local cache

Auto-propagation only reaches BCs if launch actually picks up the republished
image. Because the run call resolves a floating `:latest` tag, a republish does
**not** invalidate a locally-cached `:latest` digest; a plain run would launch
the stale cached layer. Therefore **`bc-container launch` must pull the current
`:latest` (or otherwise resolve the registry digest) before running the
container**, not rely on the local Docker cache. This makes the freshness
guarantee end-to-end: utility release → rebuild → republish `:latest` →
**launch pulls the new digest** → the BC boots on current CLIs.

This is a behavioral change to the launch path (`controller.py:388` today runs
without a pull); it is named here as a required consequence and is part of the
bc-launcher decomposition target below.

##### D4 — Deferred: controlled/pinned releases and rollback ergonomics are future work

Controlled/pinned propagation (a BC or the lead pinning a specific bc-base
digest/version rather than floating `:latest`) and rich rollback ergonomics are
**explicitly deferred**. They are recorded as future work, not designed here.

The **interim mitigation** for a bad utility release that produces a broken
bc-base is **rollback by republishing a prior known-good digest as `:latest`**
(re-tag an earlier published digest). This rides the public, digest-addressable
ghcr publication (D1): every prior build remains pullable by digest, so
`:latest` can be repointed at a good one. This is operational mitigation, not a
designed control surface — the designed pinning model is the deferred work.

#### Cross-BC ownership / decomposition

The behavioral scenarios that realize this ADR are authored by the PO **after**
this ADR; this ADR names the targets and does not author or dispatch.

- **`shopsystem-bc-launcher`** (BUILD side) — the bulk:
  - the bc-base **Dockerfile** co-located in the repo (installing the framework
    utility CLIs from their VCS/published pins, scenario-129 shape);
  - the **build/publish workflow** that listens for both its same-repo trigger
    and the cross-repo `repository_dispatch`, builds bc-base, and publishes to
    `ghcr.io/dstengle/shopsystem-bc-base` with version + `latest`, public
    (mirroring templates-142, devcontainer-15/16/17);
  - the **launch-pull behavior** (D3) so `bc-container launch` resolves the
    current `:latest` digest rather than the local cache.
- **`shopsystem-messaging`, `shopsystem-scenarios`, `shopsystem-templates`**
  (trigger-EMIT side) — a small addition to each utility's release workflow that
  fires the `repository_dispatch` to bc-launcher on a version-tag release. One
  step each; no Dockerfile, no publish logic — the build authority stays in
  bc-launcher.

The message-type vehicle is decided at dispatch time by the Architect against
the discriminator: bc-base build/publish CI and the launch-pull behavior are
**new capability** the contract surface does not pin (finding 2, finding 5) →
`assign_scenarios` to bc-launcher; the trigger-emit additions are likewise new
behavior on each utility → `assign_scenarios`. This ADR does not pre-commit the
vehicle.

##### Dependency / constraint: `lead-aj5f` (bc-launcher repo is private)

`lead-aj5f` records that `shopsystem-bc-launcher` is a **private** repo while the
other three utility repos are public. Two touch-points:

- A `repository_dispatch` **into** a private bc-launcher repo needs a token with
  dispatch permission on that repo — the trigger-emit side in each utility must
  carry a credential that can dispatch to a private target.
- If the bc-base **Dockerfile installs bc-launcher itself** (so launched BCs
  carry `bc-container` — plausible, since BCs may themselves launch sub-BCs), the
  build must pull a private repo, needing build-time credentials (the
  `--secret`/netrc path `lead-aj5f` option (b) describes). If `lead-aj5f`
  resolves to **option (a) make bc-launcher public**, both touch-points
  simplify away. This ADR does not decide `lead-aj5f`; it names it as the
  credentialing dependency the bc-launcher scenarios must account for.

#### Alternatives considered

**Option A — Scheduled (cron) rebuild of bc-base.** A nightly/periodic workflow
in bc-launcher rebuilds bc-base against whatever the current utility versions
are. Rejected as the primary mechanism: it reintroduces propagation *latency* —
the exact "fix landed but isn't live" gap (`lead-pw41`) this ADR closes — and
rebuilds even when nothing changed. It remains available as a cheap *backstop*
(a safety-net rebuild) layered under D2's event-driven trigger, but it is not
the propagation contract.

**Option B — Pin-bump PR automation (Dependabot-style).** Each utility release
opens a PR against bc-launcher bumping a pinned utility version, which on merge
fires the same-repo build. Rejected for the auto-propagate posture: it inserts a
human merge step between release and propagation, which is precisely the
*controlled/pinned* model D4 explicitly defers. It is a natural fit for the
deferred controlled-release design and is named there, not adopted now.

**Option C — Keep bc-base unowned / hand-built.** The status quo. Rejected: it
is the root cause `lead-c2kp` filed — no owner, no contract, manual rebuilds that
drift (`lead-xq0`, `lead-pw41`-not-live). Doing nothing leaves every utility fix
silently absent from newly-launched BCs.

**Option D — Each BC builds its own image instead of a shared bc-base.**
Rejected as orthogonal and heavier: bc-launcher already runs all BCs from one
hardcoded base image (finding 1); fragmenting into per-BC images multiplies the
publish/rebuild surface this ADR is trying to make single and auto-propagating,
without addressing the propagation trigger at all.

#### Consequences

- A PO-authored `assign_scenarios` to **`shopsystem-bc-launcher`** carries: the
  co-located bc-base Dockerfile, the build/publish workflow (mirroring
  templates-142 + devcontainer-15/16/17, plus the cross-repo `repository_dispatch`
  listener), and the launch-pull behavior (D3). A companion `assign_scenarios`
  to each of **`shopsystem-messaging` / `shopsystem-scenarios` /
  `shopsystem-templates`** adds the trigger-emit step. The Architect confirms the
  vehicle against the discriminator at dispatch time.
- `lead-xq0` (drift class) is addressed on the **image** axis by D2+D3; the
  lead-venv axis of `lead-xq0` is separate and not closed here.
- The `lead-pw41` "fix landed in messaging but not live in bc-base" symptom is
  structurally closed once the messaging release fires the rebuild and launch
  pulls the republished image.
- `lead-aj5f` is a **blocking dependency** for the bc-launcher scenarios'
  credentialing (dispatch-into-private and/or build-time-clone-of-private); it
  must be resolved or its option-(b) credential path designed before the
  trigger-emit and (if applicable) Dockerfile-pull scenarios can land green.
- Controlled/pinned propagation and designed rollback ergonomics remain open
  (D4); rollback-by-republish-prior-digest is the interim operational mitigation.
- bc-base gains an entry on the documented contract surface (it had none,
  finding 2); the PO/Architect may also record it as an owned image in the BC
  manifest as part of the completing work.
