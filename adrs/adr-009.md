## Context

When a BC emits `clarify` on a lead dispatch, the spec's §4 catalog lists no
`clarify_response` message type. Empirically verified 2026-05-27:

- `shop-msg send --help` exposes only three outbound vehicles
  (`request_maintenance`, `assign_scenarios`, `request_bugfix`) — the vehicles a
  lead uses to introduce or tighten work.
- `shop-msg respond clarify --help` shows `--bc --work-id --question`. The
  `--bc` flag names the BC writing the response, i.e. the caller is the BC, not
  the lead: `respond clarify` is BC-side direction (the BC writes its question
  into the lead's inbox). The CLI has no lead-side counterpart to "answer a
  clarify in-band."

Compounding the gap: even if a `clarify_response` primitive existed,
`repos/shopsystem-messaging/src/shop_msg/storage.py:210-223` enforces a strict
"one inbox row per (bc, work_id)" gate on the lead-to-BC direction when
`allow_multi_type=False` (the default for sends). The storage layer already
carries an `allow_multi_type` flag (lines 178, 186-191) set by BC-to-lead writes
so multiple message types (`work_done`, `mechanism_observation`, `clarify`) can
coexist for the same work_id in the lead inbox; the lead-to-BC send path does
not currently opt into that relaxation, so a hypothetical `clarify_response` row
would collide with the original `assign_scenarios` row on the same `work_id`.

The de-facto precedent for resolving a BC clarify is already established: the
`lead-bxl` bead (closed 2026-05-22) resolved `shopsystem-templates-918`'s BC
clarify by dispatching a fresh `request_bugfix` on a new lead-side bead carrying
the resolving scenario (scenario 08, missing-`.claude.json` warn-and-proceed).
The original `lead-gua` (work_id `shopsystem-templates-918`) `assign_scenarios`
row remains in flight on the wire; the new bugfix is *additive*, not a
replacement. See the `lead-d6o` bead's NOTES entry (2026-05-22) for the
empirical confirmation that this pattern works end-to-end.

**Alternatives considered.**

- **Option A — Implement `shop-msg send clarify_response` now.** Rejected for
  the immediate window. The change is non-trivial (catalog schema + storage gate
  widening + BC-side semantics + tests) and the existing workaround is already
  in routine use without observable failure. Deferring buys time to batch this
  with the other catalog-protocol-gap beads (`lead-d6o`, `lead-8z1`, `lead-rv9`,
  `lead-4wy`) into a coordinated brief revision.
- **Option C — New message type `clarify_resolved` signaling "process the
  originally-dispatched scenarios now."** Listed as design question #3 in
  `lead-d6o`. Rejected as the standalone vehicle for this ADR because it overlaps
  with option A; either path subsumes it. Folded into the future brief that
  resolves path (a).
- **Option D — Carry the original scenarios forward in every follow-up
  dispatch.** Avoids the storage-collision concern by reusing the existing
  vehicles' carry-forward semantics. Rejected because it produces wire-form hash
  drift between dispatches (the feature-title wrapping changes between canonical
  renders), making BDD attestation harder to track and inflating the on-wire
  payload. `lead-d6o` documents the hash-drift symptom from `lead-1pi`
  (scenario 57 hash was `d4b8aac19b34ea8c` in `lead-xkh`, `1621b59b0ea8b20b` in
  `lead-1pi`).

## Decision

Adopt a two-layer position.

**Layer (b) — operational, in effect immediately.** The lead resolves a BC
`clarify` by dispatching a fresh outbound message on a *new lead-side bead*,
carrying the resolving scenario set or maintenance instruction in the
appropriate vehicle per the standard discriminator:

- BC clarify pertaining to **unpinned existing behavior** → `request_bugfix` on
  a new lead bead, carrying the tightening scenario(s).
- BC clarify pertaining to **new capability** the original dispatch did not yet
  name → `assign_scenarios` on a new lead bead, carrying the additional
  scenarios.
- BC clarify pertaining to a **flat correction** with no scenario change →
  `request_maintenance` on a new lead bead.

The dispatch description must reference the original work_id (the one that
elicited the clarify) so the BC can correlate. The new bead's ID is the work_id
on the wire. The original dispatch remains in flight; the BC's existing clarify
row in the lead inbox is consumed by the architect once the resolving dispatch
is on the wire (per existing `shop-msg consume` discipline).

**Layer (a) — durable, deferred to a future implementation.** A
`shop-msg send clarify_response --bc <name> --work-id <id> --resolution <text>
[--scenario-file ...]` primitive is the longer-term shape. It requires:

1. A new `ClarifyResponse` Pydantic schema in `src/catalog/`.
2. The lead-to-BC send path opting into `allow_multi_type=True` for
   `clarify_response` so the row can coexist with the original
   `assign_scenarios` / `request_bugfix` / `request_maintenance` row on the same
   work_id.
3. A BC-side semantics decision: does receipt of `clarify_response` re-open the
   original dispatch for re-processing, or does it stand alone as a pinned answer
   the BC reads alongside the original? The `lead-d6o` bead itemizes this design
   question; the answer dictates whether the original row's lifecycle changes
   when the response lands.

Path (a) is not blocked on path (b); they are complementary. Path (a) is
deferred because the storage gate relaxation, the catalog schema addition, and
the BC-side re-processing semantics are a coordinated change that wants its own
brief/PDR cycle, and the operational pressure to ship it has not yet exceeded
the friction of path (b).

## Consequences

- The de-facto pattern already in use (per `lead-bxl`) is now canonical. Future
  lead-shop turns resolving a BC clarify file a new lead-side bead and dispatch
  via the appropriate existing vehicle; this is no longer a "workaround," it is
  the documented procedure until path (a) ships.
- Lead-shop role templates (`lead-architect.md`) should learn the
  clarify-resolution dispatch shape: cite the original work_id in the new bead's
  description, scope as "additive — original dispatch remains in flight," and use
  the standard discriminator on the resolving content. A follow-up bead tracks
  this template edit.
- The storage gate at `storage.py:210-223` is the load-bearing dependency for
  path (a). Any future PDR that proposes a `clarify_response` primitive must
  widen this gate or set `allow_multi_type=True` for the new message type's
  lead-to-BC writes.
- The `lead-d6o` bead is **not closed** by this ADR. It tracks the deeper design
  question of multi-turn clarify semantics; this ADR commits the operational
  vehicle but defers the underlying protocol shape.
- `mechanism_observation` from a BC reporting a different kind of protocol gap
  (e.g., the cross-tracker work_id pattern observed with
  `shopsystem-templates-918`) is unaffected by this decision; it routes per the
  standing rules in the lead-primer.
