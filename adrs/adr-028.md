## Context

PDR-017 names a live gap: ADR-026's brokered-credential model is pinned on the
contract surface (`features/bc-launcher/44–50`) but is not real in the running
fleet — BCs still authenticate via the host-mounted `~/.config/gh` token, the
broker server is not stood up, and no credential flows through it. ADR-026
§Consequences deferred exactly one thing to the parent initiative: standing up
and provisioning the broker is an operational precondition of brokered launch,
tracked under the parent initiative (`lead-mdng`), not inside that scenario slice.
That deferral is now a decision gap with two coupled questions (PDR-017 §4):

- Q1 — Where does the broker live and who owns its lifecycle?
- Q2 — Where do the broker's OWN behaviors get pinned and tested?

`lead-vycn` recorded the design intent for Q1 (a supporting-services compose
service alongside postgres) for the purposes of the launcher slice. This ADR
verifies that intent against the empirical pre-state and resolves Q1 and Q2
together, because the answer to Q1 (the broker is infra, not a BC) is precisely
what makes Q2 load-bearing (infra has no BC contract surface to pin the broker's
own behaviors). This is a structural / decomposition decision with no product-UX
surface change to the framework's outward face — hence an ADR, not a PDR.

### Pre-state empirical findings (artifact surface, ADR-018 D1/D2)

Verified from the lead CWD against the contract/artifact surface only — no BC
implementation code read, run, or git-observed. The lead host carries no `repos/`
BC source (ADR-018); the admissible surface here is this repo's `features/`,
`compose.yaml`, `bin/shop-shell`, the `adr/`/`pdr/` records, the installed
`scenarios` CLI, and the bead registry.

1. What is already pinned (has an owner). Scenarios `features/bc-launcher/44–50`
   are all tagged `@bc:shopsystem-bc-launcher` and pin the launcher-side
   end-state: no host credential mounts (44), Claude OAuth brokered (45), GitHub
   brokered (46), the `agent_vault_reachable` launch gate (47), readiness
   composition of both supporting servers (48), the one-time human paste
   precondition (49), the inside-container negative invariant (50). These pin what
   the launcher configures and gates on. Confirmed (hash enumeration over
   `features/bc-launcher/44–50`, sixteen `@scenario_hash` entries, all
   `@bc:shopsystem-bc-launcher`).

2. What has NO owner. The broker server's own lifecycle — running it as a fleet
   supporting-service, persistent volume, restart, master-password auto-unlock on
   restart, the one-time secret-load, coherent two-server bring-up — is pinned by
   nothing. `compose.yaml` today declares only the `postgres` service (one
   supporting server, container `shopsystem-messaging-postgres-1`, alias
   `postgres`, persistent `pgdata` volume, `restart: unless-stopped`, network
   `shopsystem`); there is no broker service. Confirmed (read of `compose.yaml`).
   The `lead-mdng` notes corroborate: the remaining umbrella work is "(1) stand up
   the agent-vault BROKER component itself + re-home the
   GitHub-credential-substitution scenarios."

3. The host-FS coupling is still the live mechanism — even in the lead's own
   shell. `bin/shop-shell` still bind-mounts `$HOME/.claude` and `$HOME/.gitconfig`
   (lines 41–42) into the shell container. This is the live host-mounted
   credential path PDR-017 names, exhibited on the lead's own artifact surface (not
   only the launcher's). Confirmed (read of `bin/shop-shell`).

4. The broker-substitution scenario is mis-homed on the launcher boundary — and is
   tautological there. `features/bc-launcher/46` carries two scenarios:
   `97734ca69a510e37` (an authenticated GitHub op from inside the container
   succeeds via the broker — an end-to-end outcome the launcher participates in)
   and `f23dfbe84c899968` (the broker substitutes the GitHub credential on the
   outbound request rather than the container holding it). The second asserts
   broker/network substitution behavior — what the broker does on the wire — which
   is outside the bc-launcher boundary: the launcher configures the broker arm
   (env, proxy, CA), it does not perform the substitution. Against a launcher that
   merely wires the proxy, this scenario is either tautological (it restates the
   wiring) or unfalsifiable from the launcher's vantage. Confirmed: `scenarios
   hash` over the scenario body (block-only, ADR-019) returns `f23dfbe84c899968`,
   matching its pinned tag — so the body→hash binding is exact and the hash can be
   cited for relocation.

5. The readiness-gate composition precedent exists on the launcher surface.
   `features/bc-launcher/48` (`f73afae009c283fc`, `64aaff804dc4bf98`) already pins
   that the launch readiness barrier passes only when BOTH the messaging database
   AND the agent-vault broker are reachable, mirroring scenario 33's
   `messaging_db_reachable`. So the `agent_vault_reachable` gate's composition with
   `messaging_db_reachable` is already an owned launcher behavior; what is unpinned
   is the broker being up to be reachable (finding 2). Confirmed.

### What could NOT be verified (asserted, not confirmed)

- bc-launcher's internal readiness-check code (`driver.py:312`
  `messaging_db_reachable`, the `agent_vault_reachable` call site) is not on the
  lead host (ADR-018). This ADR decides the decomposition contract — who owns the
  broker lifecycle and where broker behaviors are pinned — not the launcher's
  line-level wiring, which scenarios 47/48 already pin and the BC realizes.
- Whether `agent-vault`'s `github` template substitutes over the same MITM
  `HTTPS_PROXY` path as the Anthropic Bearer services remains the OPEN mechanism
  question ADR-026 D2 recorded. This ADR's Q2 home (a lead-owned integration
  check) is where that mechanism is exercised and pinned for the first time
  honestly — but resolving the mechanism is the integration check's job, not this
  ADR's.

### Alternatives considered and rejected

- Q1 candidate (b) — a new BC shop owning the broker. Rejected (D1): the broker is
  vendored external `agent-vault`, configured and operated, not a deliverable we
  author. A BC shop manufactures a contract surface and an Implementer/Reviewer
  loop where there is no code to build — we do not run a "postgres BC shop" for the
  same reason.
- Q1 candidate (c) — bc-launcher-managed broker. Rejected (D1): wrong cardinality
  (one broker per fleet, not per launch) and wrong lifetime (must outlive any
  launch, like postgres). The launcher's correct relationship is to gate on the
  broker (already pinned, 47/48), never to own it.
- Q2 alternative — leave broker behaviors on the launcher surface (status quo).
  Rejected (D2/D3): `f23dfbe84c899968` is tautological or unfalsifiable on the
  launcher boundary (finding 4) — the launcher cannot observe broker wire
  behavior. Keeping it there entrenches a dishonest pin.
- Q2 alternative — a thin integration check owned by the lead but with broker
  behaviors left implicit (untested). Rejected: SB-2's GitHub-substitution arm is
  precisely the ADR-026 D2 OPEN mechanism question; leaving it untested means the
  "elimination is real" claim (PDR-017's whole point) rests on an unexercised
  assumption. The integration surface must actually exercise the substitution.

## Decision

### D1 (Q1) — The agent-vault broker is a lead-shop supporting-service: a second compose service alongside postgres, NOT a BC and NOT bc-launcher-managed

The `lead-vycn` locked intent survives the empirical pre-state check, and this ADR
confirms it. The broker is added to the lead shop's `compose.yaml` as a second
supporting service on the `shopsystem` network, peer to the messaging postgres,
with:

- Persistent volume for the broker's vault state (the encrypted credential store +
  CA material), outside the repo, under the same `SHOPSYSTEM_DATA` convention as
  `pgdata` (e.g. `${SHOPSYSTEM_DATA}/agent-vault`).
- `restart: unless-stopped`, matching postgres — the broker is long-lived fleet
  infrastructure, not per-launch.
- A `shopsystem`-network alias reached by name (ADR-026 D2's hard requirement,
  proven by the spike: a host-process broker was unreachable from sibling BCs
  through the docker gateway).
- Master-password auto-unlock on restart from a lead `.env` file (the `lead-vycn`
  MVP decision), so a broker restart re-opens the vault without a human-gated
  re-provisioning event (PDR-017 SB-7).
- The one-time human secret-load (ADR-026 D4 / PDR-017 SB-6) as an explicit,
  human-gated, out-of-band paste performed once into the running broker —
  scriptable down to the single token-paste per the spike (owner register,
  service/agent/token mint, `ca fetch` are all CLI-doable).

`bin/shop-shell` brings up BOTH (the `lead-vycn` intent): extend its existing
`docker compose up -d postgres` step to bring up the broker too, so a single
entrypoint stands up the coherent two-server fleet. (`bin/shop-shell`'s own
`$HOME/.claude` / `$HOME/.gitconfig` mounts, finding 3, are the lead-shell's
coupling — flipping them off the host mount is in scope for the SB-1…SB-7 family;
see the Open question, resolved in Consequences by D4.)

How the `agent_vault_reachable` gate composes. bc-launcher's
`agent_vault_reachable` readiness check (already pinned by scenarios 47/48,
composed with `messaging_db_reachable`) targets this compose-managed broker by its
`shopsystem`-network name. No change to the launcher's gate contract is needed —
finding 5 establishes the composition is already an owned launcher behavior. What
this decision supplies is the thing being gated on: a real, running,
restart-surviving broker. The gate and the service meet at the network name; the
launcher owns the gate, the lead-shop compose owns the service.

Why not a new BC shop (PDR-017 Q1 candidate b). A BC shop (ADR-004) owns a
deliverable with its own contract surface, its own Implementer/Reviewer loop, its
own repo, and scenarios it pins and tests. The broker is agent-vault — a vendored
external tool installed from a GitHub release (spike finding: "installed
agent-vault from the GitHub release tarball"). We do not build, own, or modify its
code; we configure and operate it. Standing up a BC shop around a vendored binary
would manufacture a contract surface where there is no deliverable to build, and
would put a `bc-launcher`-style container loop around an operational concern. Infra
we operate but do not author is a supporting-service, exactly as postgres is — we
do not run a "postgres BC shop." Rejected.

Why not bc-launcher-managed (PDR-017 Q1 candidate c). The launcher is a tool to
start BCs, not a service owner (the `lead-vycn` rationale, confirmed). A per-launch
or launcher-owned broker would make every BC launch responsible for a fleet-wide
singleton's lifecycle — wrong cardinality (one broker serves the whole fleet) and
wrong lifetime (the broker must outlive any single launch, exactly as postgres
does). The launcher's correct relationship to the broker is the one it already
has: gate on its reachability (47/48), never own it. Rejected.

### D2 (Q2) — The broker's OWN behaviors are pinned by a lead-owned integration-check surface, not by a BC

Because D1 makes the broker infrastructure (a compose service, not a BC), it has
no BC contract surface to pin its own behaviors. The broker's behaviors — most
pointedly the GitHub-credential-substitution ADR-026 D2 marks OPEN, and the
credential substitution / restart-survival / vault-isolation behaviors SB-2…SB-7
describe — therefore land on a lead-shop supporting-services / integration surface
that the lead owns, parallel to how the lead owns `compose.yaml` and
`bin/shop-shell`.

Concretely, the broker's own behaviors are pinned as lead-owned integration checks
(a `features/`-adjacent supporting-services surface in this repo, authored as
Gherkin by the PO but NOT dispatched to a BC — there is no BC to dispatch them to),
exercised against the real compose-stood broker. These are the honest home for:

- The GitHub substitution behavior (SB-2's broker arm; the relocated
  `f23dfbe84c899968`, see D3) — does the `github` template substitute the stored
  credential on the outbound request, with no credential leaving the container?
  This is the integration check that finally exercises the OPEN ADR-026 D2
  mechanism question rather than asserting it from the launcher's blind side.
- The Claude-OAuth substitution behavior (SB-3's broker arm) — the broker-side
  counterpart, already proven creds-free by the spike's kill-line.
- Vault isolation (SB-4) — only the revocable `AGENT_VAULT_TOKEN` is reachable; the
  real credentials live only in the broker vault.
- Restart survival (SB-7) — the broker re-opens its vault on restart via the `.env`
  master-password auto-unlock, no re-paste.
- The one-time human paste (SB-6) — the only out-of-band provisioning step.

Why a lead-owned integration surface, not "leave it on the launcher" or "a new BC."
Leaving broker behaviors tagged `@bc:shopsystem-bc-launcher` is the current
dishonest state (finding 4): the launcher cannot observe or own the broker's wire
behavior, so those pins are tautological or unfalsifiable there. Inventing a BC to
own them is D1's rejected candidate (b) — there is no deliverable. The lead shop
already owns the broker's existence (D1: it lives in the lead's `compose.yaml`), so
the lead shop is the honest owner of the broker's behavior verification too. The
lead owns the infra; the lead owns the infra's integration checks. This keeps the
owner of the thing and the owner of its tests the same party, which is the property
that made the launcher-side homing dishonest when the broker moved out of the
launcher boundary.

This is not BC code execution (ADR-018). A lead-owned integration check exercises
the lead's own compose-managed supporting-service (postgres-peer infra the lead
operates), the same class of thing as `bin/shop-shell` bringing up postgres. It
does not read, run, or git-observe BC source. It is admissible lead-shop work, not
a violation of the no-`repos/` doctrine.

### D4 — `bin/shop-shell` joins the zero-host-coupling model: brokered creds, env git-identity, no host `~/.claude`/`~/.gitconfig` mounts; the GitHub PAT's silent-expiry risk is surfaced by a 30-day advisory (2026-06-12 amendment)

The Open question below ("does fleet bring-up include flipping `bin/shop-shell`'s
own host mounts?") is resolved in the affirmative by user direction (2026-06-12):
the lead shell now runs zero-host-coupled, exactly like a BC container. Concretely:

- No host credential mounts. The `-v "$HOME/.claude:..."` (rw) and
  `-v "$HOME/.gitconfig:..."` (ro) mounts (finding 3) are removed from
  `bin/shop-shell`. Claude and git auth now flow through the agent-vault broker via
  the same env the launcher injects into BCs: `AGENT_VAULT_ADDR`,
  `AGENT_VAULT_TOKEN`, `AGENT_VAULT_VAULT`, `AGENT_VAULT_CA_PEM`, and the MITM
  `HTTPS_PROXY=http://<token>:<vault>@agent-vault:14322`. The bc-base entrypoint
  materializes the CA from `AGENT_VAULT_CA_PEM` and bakes the placeholder
  `~/.claude` the broker fills — so a shell container with the broker env and NO
  host mount is fully brokered, the same mechanism that makes BCs creds-free.
- Git identity, not a credential, via env. What `~/.gitconfig` supplied
  (author/committer name + email — identity, not a secret) is passed as
  `GIT_AUTHOR_NAME`/`GIT_AUTHOR_EMAIL`/`GIT_COMMITTER_NAME`/`GIT_COMMITTER_EMAIL`,
  read from the host `git config user.name`/`user.email`. No `.gitconfig` and no
  token leave the host.
- Broker-cred sourcing, with a hard no-fallback rule. `bin/shop-shell` sources the
  proxy token + CA from `$ROOT/.env` if it records `AGENT_VAULT_TOKEN` (the durable
  operator-recorded source — see `.env.example`), else recovers them from a running
  BC container's env via `docker inspect` (works today, before the token is
  persisted). If neither source yields a token, the shell errors out with
  provisioning guidance — it does not fall back to host creds, because
  no-host-coupling is the whole point.
- GitHub PAT silent-expiry advisory (30 days). Of the broker's three credential
  kinds, only the GitHub PAT can lapse silently: the agent proxy token
  (`av_agt_...`) is non-expiring (rotate/revoke via `agent-vault agent rotate`),
  and `CLAUDE_OAUTH` self-refreshes (D2 / ADR-026 D2). `bin/agent-vault-check`
  (invoked non-fatally at shell startup) makes a brokered request to
  `api.github.com` and reads the `github-authentication-token-expiration` response
  header GitHub returns for expiring PATs: it warns at <=30 days (or past), reports
  days-remaining otherwise, and prints "no expiration set" when the header is
  absent (the honest answer to "can we tell when it expires"). The check is
  advisory — it never blocks the shell. Empirically confirmed 2026-06-12: brokered
  `git ls-remote` to `dstengle/shopsystem-product` succeeds with no host creds; the
  probe authenticates (X-OAuth-Scopes present) and reports the current fleet PAT as
  non-expiring.

This is consistent with D1–D3: D1 makes the broker the lead-shop's own supporting
service; D4 makes the lead's own shell a first-class consumer of that broker,
closing the last host-FS credential coupling on the lead surface (finding 3) rather
than leaving it as the launcher-only flip. The smoke verification of the brokered
GitHub path also exercises, from the lead's own surface, the GitHub-substitution
behavior D2 homes on the integration surface.

### D3 — The tautological launcher scenario `f23dfbe84c899968` is RETIRED from the launcher surface and its intent RELOCATED to the lead integration surface (SB-2's broker arm)

`features/bc-launcher/46`'s second scenario, `f23dfbe84c899968` ("the broker
substitutes the GitHub credential on the outbound request rather than the container
holding it"), is retired from `@bc:shopsystem-bc-launcher`. Its intent — broker
GitHub substitution — relocates to the lead-owned integration surface (D2), where
it can be honestly exercised against the real broker as the SB-2 broker-arm check.

The companion scenario in feature 46, `97734ca69a510e37` ("an authenticated GitHub
operation from inside the container succeeds via the broker with no mounted
credential"), stays on `@bc:shopsystem-bc-launcher`: it is the end-to-end outcome
the launcher legitimately participates in (it configures the no-mount +
broker-proxy launch, and the success-from-inside-the-container is observable at the
launch boundary). The launcher keeps the outcome pin; the broker-mechanism pin
moves to where the broker lives.

Consequence for f23dfbe8 / bclaunch-7ys. The retirement is a supersession the lead
is authorized to make on its own surface: scenario 46 lives in this repo's
`features/bc-launcher/`, on the lead's ADR-018 artifact surface — not in a BC repo
the lead cannot see (contrast ADR-027 / `lead-uhd2`, where the contradicted
scenarios were BC-committed and invisible). When the PO re-homes this behavior as
SB-2's broker-arm integration check, the dispatch (if any future launcher
re-dispatch touches feature 46) must carry an explicit retirement instruction
naming `@scenario_hash:f23dfbe84c899968`, per the conflicting-hash enumeration
discipline. The launcher's `bclaunch-7ys` realization of this scenario (if it
exists BC-side) is correspondingly retired: the launcher is no longer asked to
prove broker substitution, only to prove the no-mount launch + the end-to-end
success (`97734ca69a510e37`). This is recorded as a follow-up the PO/Architect
carry, not actioned in this ADR (no Gherkin or dispatch here).

## Consequences

- `compose.yaml` gains a second service (the broker) on the `shopsystem` network
  with a persistent vault volume and `restart: unless-stopped`; `bin/shop-shell`
  brings up both postgres and the broker. These are lead-shop infra edits (not BC
  dispatches) the lead owns — actioned under `lead-mdng` once the PO has authored
  SB-1 / SB-7 to pin the expected shape.
- A lead-owned supporting-services integration surface is established for the
  broker's own behaviors (SB-2…SB-7 broker arms). The PO authors these as Gherkin
  against the real compose-stood broker; they are not dispatched to a BC (there is
  none) — they are lead-owned checks.
- Scenario `f23dfbe84c899968` is retired from `@bc:shopsystem-bc-launcher` and
  relocated to the lead integration surface as SB-2's broker arm.
  `97734ca69a510e37` stays on the launcher (end-to-end outcome). Any future
  launcher re-dispatch touching feature 46 must carry the explicit
  `@scenario_hash:f23dfbe84c899968` retirement instruction.
- The OPEN ADR-026 D2 GitHub-brokering mechanism question finally gets an honest
  home: the lead integration check (D2) is where the `github` template's
  HTTPS-proxy substitution is exercised and pinned, not merely recommended.
- The split is clean: bc-launcher owns configuring and gating on the broker (44–50
  minus the retired hash); the lead-shop compose owns the broker's existence and
  lifecycle (D1); the lead-shop integration surface owns the broker's own behavior
  verification (D2). One party owns the broker's existence and its tests — the
  property whose absence made the launcher homing dishonest.
- `bin/shop-shell` is now a brokered consumer (D4, 2026-06-12): its host
  `~/.claude`/`~/.gitconfig` mounts are removed; creds arrive via `AGENT_VAULT_*` +
  the MITM `HTTPS_PROXY`, git identity via `GIT_*` env. A new `bin/agent-vault-check`
  surfaces the GitHub PAT's expiry as a 30-day advisory (the only broker credential
  with a silent-expiry risk). `.env.example` gains a documented optional
  `AGENT_VAULT_TOKEN` the operator records from `bin/agent-vault-provision`
  (step 5). This closes finding 3 — the last host-FS credential coupling on the
  lead surface.

   *succeeds* via the broker — an end-to-end outcome the launcher participates
   in) and **`f23dfbe84c899968`** ("the broker *substitutes* the GitHub
   credential on the outbound request rather than the container holding it").
   The second asserts **broker/network substitution behavior** — what the broker
   does on the wire — which is **outside the bc-launcher boundary**: the launcher
   configures the broker arm (env, proxy, CA), it does not perform the
   substitution. Against a launcher that merely wires the proxy, this scenario is
   either tautological (it restates the wiring) or unfalsifiable from the
   launcher's vantage (the launcher cannot observe the broker's upstream
   request). **Confirmed**: `scenarios hash` over the scenario body
   (block-only, ADR-019) returns `f23dfbe84c899968`, matching its pinned tag —
   so the body→hash binding is exact and the hash can be cited for relocation.

5. **The readiness-gate composition precedent exists on the launcher surface.**
   `features/bc-launcher/48` (`f73afae009c283fc`, `64aaff804dc4bf98`) already
   pins that the launch readiness barrier passes only when BOTH the messaging
   database AND the agent-vault broker are reachable, mirroring scenario 33's
   `messaging_db_reachable`. So the `agent_vault_reachable` gate's *composition
   with* `messaging_db_reachable` is already an owned launcher behavior; what is
   unpinned is the **broker being up to be reachable** (finding 2). **Confirmed.**

##### What could NOT be verified (asserted, not confirmed)

- **bc-launcher's internal readiness-check code** (`driver.py:312`
  `messaging_db_reachable`, the `agent_vault_reachable` call site) is not on the
  lead host (ADR-018). This ADR decides the *decomposition contract* — who owns
  the broker lifecycle and where broker behaviors are pinned — not the launcher's
  line-level wiring, which scenarios 47/48 already pin and the BC realizes.
- **Whether `agent-vault`'s `github` template substitutes over the same MITM
  `HTTPS_PROXY` path** as the Anthropic Bearer services remains the OPEN
  mechanism question ADR-026 D2 recorded. This ADR's Q2 home (a lead-owned
  integration check) is where that mechanism is *exercised and pinned* for the
  first time honestly — but resolving the mechanism is the integration check's
  job, not this ADR's.

#### Decision

##### D1 (Q1) — The agent-vault broker is a lead-shop supporting-service: a second compose service alongside postgres, NOT a BC and NOT bc-launcher-managed

The `lead-vycn` locked intent survives the empirical pre-state check, and this
ADR confirms it. The broker is added to the lead shop's `compose.yaml` as a
**second supporting service** on the `shopsystem` network, peer to the messaging
postgres, with:

- **Persistent volume** for the broker's vault state (the encrypted credential
  store + CA material), outside the repo, under the same `SHOPSYSTEM_DATA`
  convention as `pgdata` (e.g. `${SHOPSYSTEM_DATA}/agent-vault`).
- **`restart: unless-stopped`**, matching postgres — the broker is long-lived
  fleet infrastructure, not per-launch.
- **A `shopsystem`-network alias** reached by name (ADR-026 D2's hard
  requirement, proven by the spike: a host-process broker was unreachable from
  sibling BCs through the docker gateway).
- **Master-password auto-unlock on restart** from a lead `.env` file (the
  `lead-vycn` MVP decision), so a broker restart re-opens the vault without a
  human-gated re-provisioning event (PDR-017 SB-7).
- **The one-time human secret-load** (ADR-026 D4 / PDR-017 SB-6) as an explicit,
  human-gated, out-of-band paste performed once into the running broker —
  scriptable down to the single token-paste per the spike (owner register,
  service/agent/token mint, `ca fetch` are all CLI-doable).

**`bin/shop-shell` brings up BOTH** (the `lead-vycn` intent): extend its
existing `docker compose up -d postgres` step to bring up the broker too, so a
single entrypoint stands up the coherent two-server fleet. (`bin/shop-shell`'s
own `$HOME/.claude` / `$HOME/.gitconfig` mounts, finding 3, are the lead-shell's
coupling — flipping them off the host mount is in scope for the SB-1…SB-7
family, but the lead shell is a distinct surface from a BC container and the PO
should scope whether SB-1's "fleet bring-up" includes the shell or only BC
launch; see Open question below.)

**How the `agent_vault_reachable` gate composes.** bc-launcher's
`agent_vault_reachable` readiness check (already pinned by scenarios 47/48,
composed with `messaging_db_reachable`) targets *this* compose-managed broker by
its `shopsystem`-network name. No change to the launcher's gate contract is
needed — finding 5 establishes the composition is already an owned launcher
behavior. What this decision supplies is the **thing being gated on**: a real,
running, restart-surviving broker. The gate and the service meet at the network
name; the launcher owns the gate, the lead-shop compose owns the service.

**Why not a new BC shop (PDR-017 Q1 candidate b).** A BC shop (ADR-004) owns a
*deliverable with its own contract surface, its own Implementer/Reviewer loop,
its own repo, and scenarios it pins and tests*. The broker is **agent-vault** —
a vendored external tool installed from a GitHub release (spike finding: "installed
agent-vault from the GitHub release tarball"). We do not build, own, or modify its
code; we *configure and operate* it. Standing up a BC shop around a vendored
binary would manufacture a contract surface where there is no deliverable to
build, and would put a `bc-launcher`-style container loop around an operational
concern. Infra we operate but do not author is a supporting-service, exactly as
postgres is — we do not run a "postgres BC shop." **Rejected.**

**Why not bc-launcher-managed (PDR-017 Q1 candidate c).** The launcher is a
*tool to start BCs*, not a service owner (the `lead-vycn` rationale, confirmed).
A per-launch or launcher-owned broker would make every BC launch responsible for
a fleet-wide singleton's lifecycle — wrong cardinality (one broker serves the
whole fleet) and wrong lifetime (the broker must outlive any single launch,
exactly as postgres does). The launcher's correct relationship to the broker is
the one it already has: gate on its reachability (47/48), never own it.
**Rejected.**

##### D2 (Q2) — The broker's OWN behaviors are pinned by a lead-owned integration-check surface, not by a BC

Because D1 makes the broker infrastructure (a compose service, not a BC), it has
**no BC contract surface** to pin its own behaviors. The broker's behaviors —
most pointedly the **GitHub-credential-substitution** ADR-026 D2 marks OPEN, and
the credential substitution / restart-survival / vault-isolation behaviors
SB-2…SB-7 describe — therefore land on a **lead-shop supporting-services /
integration surface that the lead owns**, parallel to how the lead owns
`compose.yaml` and `bin/shop-shell`.

Concretely, the broker's own behaviors are pinned as **lead-owned integration
checks** (a `features/`-adjacent supporting-services surface in this repo,
authored as Gherkin by the PO but **not** dispatched to a BC — there is no BC to
dispatch them to), exercised against the real compose-stood broker. These are
the honest home for:

- **The GitHub substitution behavior** (SB-2's broker arm; the relocated
  `f23dfbe84c899968`, see D3) — does the `github` template substitute the stored
  credential on the outbound request, with no credential leaving the container?
  This is the integration check that finally **exercises** the OPEN ADR-026 D2
  mechanism question rather than asserting it from the launcher's blind side.
- **The Claude-OAuth substitution behavior** (SB-3's broker arm) — the
  broker-side counterpart, already proven creds-free by the spike's kill-line.
- **Vault isolation** (SB-4) — only the revocable `AGENT_VAULT_TOKEN` is
  reachable; the real credentials live only in the broker vault.
- **Restart survival** (SB-7) — the broker re-opens its vault on restart via the
  `.env` master-password auto-unlock, no re-paste.
- **The one-time human paste** (SB-6) — the only out-of-band provisioning step.

**Why a lead-owned integration surface, not "leave it on the launcher" or "a new
BC."** Leaving broker behaviors tagged `@bc:shopsystem-bc-launcher` is the
current dishonest state (finding 4): the launcher cannot observe or own the
broker's wire behavior, so those pins are tautological or unfalsifiable there.
Inventing a BC to own them is D1's rejected candidate (b) — there is no
deliverable. The lead shop already owns the broker's *existence* (D1: it lives in
the lead's `compose.yaml`), so the lead shop is the honest owner of the broker's
*behavior verification* too. The lead owns the infra; the lead owns the infra's
integration checks. This keeps the owner of the thing and the owner of its tests
the same party, which is the property that made the launcher-side homing
dishonest when the broker moved out of the launcher boundary.

**This is not BC code execution (ADR-018).** A lead-owned integration check
exercises the lead's *own* compose-managed supporting-service (postgres-peer
infra the lead operates), the same class of thing as `bin/shop-shell` bringing
up postgres. It does not read, run, or git-observe BC source. It is admissible
lead-shop work, not a violation of the no-`repos/` doctrine.

##### D4 — `bin/shop-shell` joins the zero-host-coupling model: brokered creds, env git-identity, no host `~/.claude`/`~/.gitconfig` mounts; the GitHub PAT's silent-expiry risk is surfaced by a 30-day advisory (2026-06-12 amendment)

The Open question below ("does fleet bring-up include flipping `bin/shop-shell`'s
own host mounts?") is resolved in the affirmative by user direction (2026-06-12):
**the lead shell now runs zero-host-coupled, exactly like a BC container.**
Concretely:

- **No host credential mounts.** The `-v "$HOME/.claude:..."` (rw) and
  `-v "$HOME/.gitconfig:..."` (ro) mounts (finding 3) are **removed** from
  `bin/shop-shell`. Claude and git auth now flow through the agent-vault broker
  via the same env the launcher injects into BCs: `AGENT_VAULT_ADDR`,
  `AGENT_VAULT_TOKEN`, `AGENT_VAULT_VAULT`, `AGENT_VAULT_CA_PEM`, and the MITM
  `HTTPS_PROXY=http://<token>:<vault>@agent-vault:14322`. The bc-base entrypoint
  materializes the CA from `AGENT_VAULT_CA_PEM` and bakes the placeholder
  `~/.claude` the broker fills — so a shell container with the broker env and NO
  host mount is fully brokered, the same mechanism that makes BCs creds-free.

- **Git identity, not a credential, via env.** What `~/.gitconfig` supplied
  (author/committer name + email — identity, not a secret) is passed as
  `GIT_AUTHOR_NAME`/`GIT_AUTHOR_EMAIL`/`GIT_COMMITTER_NAME`/`GIT_COMMITTER_EMAIL`,
  read from the host `git config user.name`/`user.email`. No `.gitconfig` and no
  token leave the host.

- **Broker-cred sourcing, with a hard no-fallback rule.** `bin/shop-shell`
  sources the proxy token + CA from `$ROOT/.env` if it records
  `AGENT_VAULT_TOKEN` (the durable operator-recorded source — see
  `.env.example`), else recovers them from a running BC container's env via
  `docker inspect` (works today, before the token is persisted). If **neither**
  source yields a token, the shell **errors out** with provisioning guidance —
  it does **not** fall back to host creds, because no-host-coupling is the whole
  point.

- **GitHub PAT silent-expiry advisory (30 days).** Of the broker's three
  credential kinds, only the GitHub PAT can lapse silently: the agent proxy
  token (`av_agt_...`) is non-expiring (rotate/revoke via
  `agent-vault agent rotate`), and `CLAUDE_OAUTH` self-refreshes (D2 / ADR-026
  D2). `bin/agent-vault-check` (invoked non-fatally at shell startup) makes a
  brokered request to `api.github.com` and reads the
  `github-authentication-token-expiration` response header GitHub returns for
  expiring PATs: it **warns at <=30 days** (or past), reports days-remaining
  otherwise, and prints "no expiration set" when the header is absent (the
  honest answer to "can we tell when it expires"). The check is advisory — it
  never blocks the shell. *Empirically confirmed 2026-06-12: brokered
  `git ls-remote` to `dstengle/shopsystem-product` succeeds with no host creds;
  the probe authenticates (X-OAuth-Scopes present) and reports the current
  fleet PAT as non-expiring.*

This is consistent with D1–D3: D1 makes the broker the lead-shop's own
supporting service; D4 makes the lead's own **shell** a first-class consumer of
that broker, closing the last host-FS credential coupling on the lead surface
(finding 3) rather than leaving it as the launcher-only flip. The smoke
verification of the brokered GitHub path also exercises, from the lead's own
surface, the GitHub-substitution behavior D2 homes on the integration surface.

##### D3 — The tautological launcher scenario `f23dfbe84c899968` is RETIRED from the launcher surface and its intent RELOCATED to the lead integration surface (SB-2's broker arm)

`features/bc-launcher/46`'s second scenario, `f23dfbe84c899968` ("the broker
substitutes the GitHub credential on the outbound request rather than the
container holding it"), is **retired** from `@bc:shopsystem-bc-launcher`. Its
*intent* — broker GitHub substitution — relocates to the lead-owned integration
surface (D2), where it can be honestly exercised against the real broker as the
SB-2 broker-arm check.

The companion scenario in feature 46, **`97734ca69a510e37`** ("an authenticated
GitHub operation from inside the container *succeeds* via the broker with no
mounted credential"), **stays on `@bc:shopsystem-bc-launcher`**: it is the
end-to-end outcome the launcher legitimately participates in (it configures the
no-mount + broker-proxy launch, and the success-from-inside-the-container is
observable at the launch boundary). The launcher keeps the *outcome* pin; the
*broker-mechanism* pin moves to where the broker lives.

**Consequence for f23dfbe8 / bclaunch-7ys.** The retirement is a
supersession the lead is *authorized to make on its own surface*: scenario 46
lives in this repo's `features/bc-launcher/`, on the lead's ADR-018 artifact
surface — not in a BC repo the lead cannot see (contrast ADR-027 / `lead-uhd2`,
where the contradicted scenarios were BC-committed and invisible). When the PO
re-homes this behavior as SB-2's broker-arm integration check, the dispatch (if
any future launcher re-dispatch touches feature 46) must carry an explicit
retirement instruction naming `@scenario_hash:f23dfbe84c899968`, per the
conflicting-hash enumeration discipline. The launcher's `bclaunch-7ys`
realization of this scenario (if it exists BC-side) is correspondingly retired:
the launcher is no longer asked to prove broker substitution, only to prove the
no-mount launch + the end-to-end success (`97734ca69a510e37`). This is recorded
as a follow-up the PO/Architect carry, not actioned in this ADR (no Gherkin or
dispatch here).

#### Alternatives considered

**Q1 candidate (b) — a new BC shop owning the broker.** Rejected (D1): the
broker is vendored external `agent-vault`, configured and operated, not a
deliverable we author. A BC shop manufactures a contract surface and an
Implementer/Reviewer loop where there is no code to build — we do not run a
"postgres BC shop" for the same reason.

**Q1 candidate (c) — bc-launcher-managed broker.** Rejected (D1): wrong
cardinality (one broker per fleet, not per launch) and wrong lifetime (must
outlive any launch, like postgres). The launcher's correct relationship is to
gate on the broker (already pinned, 47/48), never to own it.

**Q2 alternative — leave broker behaviors on the launcher surface (status
quo).** Rejected (D2/D3): `f23dfbe84c899968` is tautological or unfalsifiable on
the launcher boundary (finding 4) — the launcher cannot observe broker wire
behavior. Keeping it there entrenches a dishonest pin.

**Q2 alternative — a thin integration check owned by the lead but with broker
behaviors left implicit (untested).** Rejected: SB-2's GitHub-substitution arm
is precisely the ADR-026 D2 OPEN mechanism question; leaving it untested means
the "elimination is real" claim (PDR-017's whole point) rests on an unexercised
assumption. The integration surface must actually exercise the substitution.

#### Consequences

- **`compose.yaml` gains a second service** (the broker) on the `shopsystem`
  network with a persistent vault volume and `restart: unless-stopped`;
  **`bin/shop-shell` brings up both** postgres and the broker. These are
  lead-shop infra edits (not BC dispatches) the lead owns — actioned under
  `lead-mdng` once the PO has authored SB-1 / SB-7 to pin the expected shape.
- **A lead-owned supporting-services integration surface** is established for the
  broker's own behaviors (SB-2…SB-7 broker arms). The PO authors these as
  Gherkin against the real compose-stood broker; they are **not dispatched** to a
  BC (there is none) — they are lead-owned checks.
- **Scenario `f23dfbe84c899968` is retired** from `@bc:shopsystem-bc-launcher`
  and relocated to the lead integration surface as SB-2's broker arm.
  `97734ca69a510e37` stays on the launcher (end-to-end outcome). Any future
  launcher re-dispatch touching feature 46 must carry the explicit
  `@scenario_hash:f23dfbe84c899968` retirement instruction.
- **The OPEN ADR-026 D2 GitHub-brokering mechanism question** finally gets an
  honest home: the lead integration check (D2) is where the `github` template's
  HTTPS-proxy substitution is *exercised and pinned*, not merely recommended.
- **The split is clean:** bc-launcher owns *configuring and gating on* the broker
  (44–50 minus the retired hash); the lead-shop compose owns the broker's
  *existence and lifecycle* (D1); the lead-shop integration surface owns the
  broker's *own behavior verification* (D2). One party owns the broker's
  existence and its tests — the property whose absence made the launcher homing
  dishonest.
- **`bin/shop-shell` is now a brokered consumer (D4, 2026-06-12):** its host
  `~/.claude`/`~/.gitconfig` mounts are removed; creds arrive via `AGENT_VAULT_*`
  + the MITM `HTTPS_PROXY`, git identity via `GIT_*` env. A new
  `bin/agent-vault-check` surfaces the GitHub PAT's expiry as a 30-day advisory
  (the only broker credential with a silent-expiry risk). `.env.example` gains a
  documented optional `AGENT_VAULT_TOKEN` the operator records from
  `bin/agent-vault-provision` (step 5). This closes finding 3 — the last host-FS
  credential coupling on the lead surface.
