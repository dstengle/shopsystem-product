## Source (pre-modernization)

#### AMENDMENT (2026-07-08, lead-b3f0 — David authorized) — the engage runs `dispatcher.toml` (local provider), the dispatcher is a NATIVE zero-LLM poll-loop, and work_id is delivered by a per-child `[run.environment.env]` overlay

The original D1/D2/D3 below described a `fabro run dispatcher.fabro -I
BC_NAME=<bc>` engage entering a cyclic `watch(native) → launch(Haiku) →
back-edge` graph. In-container empirical verification for **lead-b3f0**
(2026-07-08, recorded on `bd show lead-b3f0`) proved that design does not
come up and is heavier than needed. **This amendment supersedes D1/D2/D3 on
the three points below; the rest of the ADR (D4 native parallelism, D5
robustness, D6 no-launch-work-id parity, D7 shutdown seam, and the
load-bearing "ADR-051 is UNCHANGED" invariant) stands unchanged.** The
superseding scenarios are `24d94274b9cbc2b0` (A, engage target),
`a5e16a192f755768` (B, native poll-loop topology), and `6088da7e9e4c4e59`
(C, env-overlay work_id delivery).

**(D1 amended) — the engage target is `fabro run dispatcher.toml`, NOT the
bare `dispatcher.fabro`.** ROOT CAUSE (lead-b3f0 in-container proof): running
`fabro run dispatcher.fabro` — the `.fabro` graph def directly — BYPASSES the
`[environments.local]` provider config and DEFAULTS to a Docker sandbox
executor. The bc-base container has no docker daemon, so the run fails in 0s
("Failed to connect to Docker daemon: /var/run/docker.sock") and the
dispatcher process EXITS before it ever polls the inbox — the BC comes up
OFFLINE and never processes `assign_scenarios`. The fix: the engage runs
`fabro run dispatcher.toml` — the `.toml` entrypoint applies `provider =
local`, so the sandbox comes up IN-PROCESS ("Sandbox: local ready in 1ms")
and every native node executes in-container with no docker daemon. This
corrects the D1/D2/engage-tier argv (`fabro run dispatcher.fabro -I
BC_NAME=<bc>`) throughout.

**(D2/D3 amended) — the dispatcher is a NATIVE cyclic poll-loop with NO
long-running `shop-msg watch` node and NO LLM node in the loop.** The
original `start → watch(native, blocks on `shop-msg watch`) → launch(Haiku
agent) → back-edge` design carried the long-running `shop-msg watch` node and
the Haiku `launch` node ONLY to feed Claude's Monitor — both are unnecessary
under the fabro engage. The validated redesign (lead-b3f0) is a NATIVE cyclic
poll-loop: `start → poll → dispatch → wait → poll` where `wait → poll` is the
back-edge. `poll` (native `script=`) lists the current pending set via
`shop-msg pending inbox --bc <bc>` and returns promptly; `dispatch` (native
`script=`) acts on the pending work ids; `wait` (native `script=`) sleeps a
short interval before the back-edge. There is NO long-running `shop-msg watch`
node and NO Haiku/LLM/agent node anywhere in the loop — the steady-state loop
consumes ZERO model tokens; tokens are spent only on the child's actual work.
(The runtime cyclic-execution property proven under the original ADR-058
Open-Q1 proof still holds — a native cyclic graph is what runs.)

**(work_id delivery amended) — per-child `[run.environment.env] WORK_ID=<id>`
overlay, NOT `-I WORK_ID`.** lead-b3f0 PROVED that `-I WORK_ID=<id>` does NOT
reach a child's native `script=` node env, but a per-child `.toml` carrying
`[run.environment.env] WORK_ID=<id>` DOES (a detached child ran with
`child-ran-WORK_ID=lead-realbead-99` delivered via that overlay). So the
native `dispatch` node needs NO Haiku: per pending work id it MATERIALIZES a
per-child `.toml` carrying the concrete `WORK_ID` in `[run.environment.env]`
and spawns the child detached (`fabro run child.toml --detach`). Children run
in PARALLEL isolated per `WORK_ID` (D4 native parallelism preserved), and the
`dispatch` node does not block before the `wait → poll` back-edge. This
supersedes D3's Haiku `launch` node writing the work_id via the agent; the
design caveat recorded in the original "Design caveat carried into
implementation" section is what this amendment resolves — the resolution is a
native env-overlay write, not a Haiku-agent write.

#### AMENDMENT 2 (2026-07-08, lead-3zzu — David authorized) — the DISPATCH node becomes an ACP-backed NON-LLM SCRIPT AGENT (`backend="acp"`) so it receives context and returns decisions, giving idempotency (in-flight-skip) and concrete WORK_ID delivery; zero-token steady state HOLDS because the ACP agent is a non-LLM script, NOT because the loop has no agent node

AMENDMENT 1 above made the dispatcher a NATIVE poll-loop `start → poll →
dispatch → wait → poll` whose `dispatch` node is a native `script=` command
node. In-container operation for **lead-b3f0** (observed 2026-07-08 on
shopsystem-knowledge v1, recorded on `bd show lead-3zzu`) exposed a design
gap in that native `dispatch` node: **it is context-blind and re-fires for
any still-pending work_id every ~6s poll cycle (D4/D5).** A slow child (the
Implementer→Reviewer loop takes minutes) stays pending until it emits
`work_done`, so each cycle spawns a NEW child keyed the SAME `WORK_ID`, and
the duplicate children COLLIDE on the same per-`WORK_ID` git worktree — the
work may never converge and duplicate LLM tokens burn on the child side.

**This amendment supersedes AMENDMENT 1's `dispatch`-node form on the two
points below; the rest of AMENDMENT 1 (D1 `dispatcher.toml` local-provider
engage target — scenario `24d94274b9cbc2b0`, UNCHANGED; the cyclic
`poll → dispatch → wait → poll` topology; `poll` and `wait` as native
`script=` nodes; the ABSENCE of any long-running `shop-msg watch` node) and
the whole rest of the ADR (the load-bearing "ADR-051 is UNCHANGED"
invariant, D4 native parallelism, D5 robustness, D6 no-launch-work-id
parity, D7 shutdown seam) stand unchanged.** The superseding scenarios are
`7709a671bdfaddb7` (A, ACP dispatch node kind), `713d01c4f4dfd107` (B,
idempotent in-flight-skip decision), and `f38ab66672151669` (C, concrete
WORK_ID delivery preserved through the ACP node).

**(dispatch-node form amended) — the `dispatch` node is an ACP-backed
SCRIPT AGENT (`backend="acp"` + `acp.command`/`acp.config`), NOT a native
command node.** DESIGN DIRECTION (David 2026-07-08): the native command
`dispatch` node cannot fix the re-fire because a native `script=` node is
CONTEXT-BLIND — it acts on the raw pending work ids each cycle with NO
memory of what it already spawned. Replace it with an ACP-backed script
agent (fabro `backend="acp"`, `acp.command="python3 <dispatch_acp_agent.py>"`
OR `acp.config` JSON stdio; fabro implements ACP via the
`agent-client-protocol` crate v0.11.1 — JSON-RPC over stdio:
`initialize` / `session/new` / `session/prompt`). Unlike a native command
node, the ACP agent node RECEIVES the incoming CONTEXT (the pending inbox
set from `poll` PLUS the in-flight run state) and RETURNS structured
DISPATCH DECISIONS out. That context-in / decisions-out property is exactly
what unlocks the two behaviors below and is the property the native command
node lacks.

- **(a) IDEMPOTENCY — track in-flight work_ids; do NOT re-dispatch a
  work_id whose child is still running.** The ACP agent reads the in-flight
  run state from its context and decides to SPAWN only for a work_id with
  NO live child; for a work_id whose prior child is still running it decides
  to SKIP. So a slow child is never duplicated, and the per-`WORK_ID`
  worktree collision (the root concern) is eliminated. A genuinely
  unstarted work_id is still dispatched exactly once. (Scenario B,
  `713d01c4f4dfd107`.)
- **(b) Concrete WORK_ID delivery — preserved through the ACP node.** The
  ACP agent carries the concrete work_id from its context into the
  per-child `[run.environment.env] WORK_ID=<id>` overlay it materializes,
  and spawns the child DETACHED — so the work_id reaches the child's native
  `script=` node env exactly as AMENDMENT 1's env-overlay delivery did
  (the lead-b3f0 delivery guarantee), just now sourced from the ACP
  decision rather than a context-blind command node. The `-I WORK_ID`
  negative control still holds (`-I` does not reach the child's native
  `script=` env). (Scenario C, `f38ab66672151669`.)

**ACP viability (lead empirical check, artifact + installed-tool surface —
NO BC source read/run, ADR-018 D1/D2).** fabro supports `backend="acp"` via
the `agent-client-protocol` crate v0.11.1: JSON-RPC over stdio
(`initialize` / `session/new` / `session/prompt`). The ACP process owns its
own auth / tools / model — **Fabro injects NO LLM creds into it** — and runs
in the active fabro sandbox. Node attrs: `backend="acp"` plus
`acp.command=<shell>` OR `acp.config=<JSON stdio config>`

protocol internals are the BC's build and are NOT pinned by the superseding
scenarios; the scenarios pin the NODE KIND, the in-flight-skip decision
contract, and the env-overlay delivery on the artifact surface.

**Zero-token steady state — WHY IT STILL HOLDS, and a correction to
AMENDMENT 1's framing.** AMENDMENT 1 asserted zero-token steady state
BECAUSE the loop carried "NO LLM/agent node anywhere in the loop." That
framing is now too strong: the loop DOES carry an agent node (the ACP
`dispatch` script-agent). The zero-token property nonetheless HOLDS —
because the ACP dispatch agent is a **NON-LLM SCRIPT** that owns its own
behavior and into which Fabro injects no model creds. The correct
invariant is: **the ONLY agent node in the loop is the NON-LLM ACP dispatch
script-agent; there is NO LLM/model-backed node anywhere in the loop; the
steady-state loop consumes ZERO model tokens (tokens are spent only on the
child's actual work) — zero-token BY the ACP agent being non-LLM, NOT by
the absence of agent nodes.**

#### Context

David authorized fabro as launch-interface PARITY with the tmux path: a BC
that comes up and runs a **reactive inbox-watcher loop**. The implemented
engage runs **one-shot** (`fabro run … -I WORK_ID=<W>` → one ADR-051 loop →
one `work_done` → TERMINATE). ADR-050 titled itself "parity" but D3 dropped
the reactive-persistent lifecycle; the fabro loop went one-shot because
fabro had no reactive node (ADR-048 G4).

**First correction (David 2026-07-06):** make the watcher live IN FABRO as a
single persistent CYCLIC run — a `watch`/`drain` node yields a work_id, the
Implementer→Reviewer body processes it, a back-edge returns to `watch`. That
draft (retained below as a REJECTED alternative) needed a per-iteration
work_id **file hand-off** and a **single shared work_id file breaks
parallelism**: two concurrent work items stomp the same file.

**Pivot (David 2026-07-07) — PARENT-ORCHESTRATES-DETACHED-CHILDREN.** Use
fabro's NATIVE parent/child fan-out. The launcher's ONE persistent engage is
a small NEW def, `dispatcher.fabro` (the OUTER parent run). The dispatcher
watches the inbox and, per work item, spawns a **DETACHED CHILD** run of the
EXISTING `workflow.fabro` — unchanged, one `WORK_ID` per child, exactly like
today's one-shot — linked to the dispatcher via `--parent` and backgrounded
via `--detach`. Multiple children run concurrently, each isolated by its own
per-run `WORK_ID`. There is NO shared file, NO semaphore, NO cyclic INNER
def, and NO ADR-051 revision. This fan-out shape is KEPT unchanged by the
refinement below.

**Refinement (David 2026-07-07) — the dispatcher is a REAL CYCLIC fabro
GRAPH with a Haiku-powered agent LAUNCH node.** The prior draft made the
dispatcher a single native `script=` node running a bash `while` loop, which
deliberately SIDESTEPPED the cyclic-graph-runtime question. David's
refinement instead makes the dispatcher a genuine cyclic graph
(`start → watch → launch → watch …`), for three reasons — each a deliberate
design GOAL, not an incidental choice:

- **(a) Prove fabro executes cyclic graphs at runtime.** Running the
  dispatcher as a genuine cyclic graph deliberately builds confidence, on a
  LOW-STAKES loop, that fabro follows a back-edge and re-invokes a node at
  runtime — the exact runtime property the 2026-07-06 cyclic-inner-def draft
  had to gate on as an UNPROVEN bet. Proving it here, where the cycle only
  drives dispatch (not the gated Implementer→Reviewer contract), retires that
  uncertainty cheaply and de-risks any future cyclic use.
- **(b) An AGENT (Haiku) launch node is a seam to insert JUDGMENT into the
  dispatch loop.** A native `script=` node has no agency; a Haiku agent node
  at `launch` gives a natural place to later add dispatch-time judgment
  (dedupe, prioritize, defer, throttle) without restructuring the graph. The
  node is kept SIMPLE now (its job is to launch); the seam is noted, judgment
  is not over-designed.
- **(c) It BRIDGES the current fabro limitation that native command nodes
  can't accept per-iteration env/args/context.** A native `script=` node
  inherits only the static `[run.environment.env]` overlay — it cannot
  receive a per-iteration value (the exact wall the 2026-07-06 file-hand-off
  draft hit). An AGENT node CAN receive the current work_id as CONTEXT (from
  the upstream `watch` node's output / node inputs — agents get
  context/prompt, native scripts do not). The Haiku launch node is therefore
  a forward-looking bridge: when fabro command nodes gain
  env/arg/context support, swap the Haiku node for a native one and keep the
  same graph.

   and `-d, --detach` ("Run the workflow in the background and print the run

   in-container (Open Q3) — the design does not depend on it, since the
   class-less default already yields Haiku.

##### D1 — The engage is ONE persistent `fabro run dispatcher.fabro` (a NEW parent def), with NO required `--work-id`

`bc-container launch <bc> --orchestrator fabro` (after the unchanged

##### D2 — The dispatcher is a REAL CYCLIC fabro GRAPH: `start → watch(native) → launch(Haiku agent) → back-edge to watch → end(shutdown)`

RECOMMENDED form (per David's 2026-07-07 refinement): `dispatcher.fabro` is a
genuine cyclic graph, NOT a bash `while` loop inside one node. The cycle is
the back-edge `launch → watch`, deliberately exercised so the in-container
run PROVES fabro follows a back-edge at runtime (goal (a)).

**Exact dispatcher graph (nodes / edges):**

- `start` — `shape=Mdiamond` (the single start terminal).
- `watch` — **NATIVE `script=` node** (`shape=parallelogram`, no LLM). Its
  job: yield when there is work, block otherwise. On entry it FIRST checks
  `shop-msg pending inbox --bc "$BC_NAME"` non-blockingly — if non-empty
  (startup-drain / catch-up on messages that arrived between sessions or
  while `launch` was busy), it exits 0 immediately so `launch` runs. Else it
  blocks on `shop-msg watch --bc "$BC_NAME"`, SKIPS the leading `READY`
  sentinel, and on the first real event line exits 0 (a WAKE arrived).
  If the watch stream CLOSES (shutdown / DB gone) it exits nonzero.
- `launch` — **HAIKU-powered AGENT node** (`shape=box`, `class="launch"`
  mapped to `claude-haiku-4-5` via the graph `model_stylesheet`;
  `permissions="read-write"`). See D3. It reads the authoritative pending set
  and spawns one detached child per work item.
- `end` — `shape=Msquare` (the single terminal), reached ONLY on shutdown.

Edges:

```
start  -> watch                                  // enter the loop
watch  -> launch                                 // UNCONDITIONAL fallback: work is ready (a wake / pending)
watch  -> end     [condition="outcome=failed"]   // watch stream closed => shutdown => terminal
launch -> watch                                  // THE CYCLE: back-edge — keep watching
```

**Structural-rule conformance (fabro):** ONE `start` (`Mdiamond`), ONE
terminal (`end`, `Msquare`); the `watch` node carries a conditional edge
(`→ end`) so it has an UNCONDITIONAL fallback (`→ launch`) per the
proven-safe single-success pattern (spike finding: `node -> success`
unconditional + `node -> fail [condition="outcome=failed"]`); the `launch`
node has only its unconditional back-edge (`→ watch`), which is the CYCLE.
`end` is reachable only via `watch`'s failed-outcome (shutdown) edge — so the
graph runs forever until shutdown, exactly the reactive-persistent lifecycle.

Graph header:

```
graph [
  goal="Reactive inbox-watcher dispatcher for BC {{ inputs.BC_NAME }}: watch the shop-msg inbox and, per pending work item, spawn ONE detached CHILD run of workflow.fabro under THIS run as parent. Persistent until shutdown."
  model_stylesheet="* { model: claude-haiku-4-5 } .launch { model: claude-haiku-4-5 }"
  rankdir=TB
]
```

`$BC_NAME` is delivered by the proven static `[run.environment.env]` overlay
(constant per container); there is no per-iteration overlay value — the
per-iteration work_id is handled at the `launch` agent (D3), which is exactly
the bridge reason (c).

##### D3 — The `launch` node is a SIMPLE Haiku agent that receives the work_id as CONTEXT and spawns one DETACHED CHILD per pending work item

The `launch` node is an agent pinned to **`claude-haiku-4-5`** (via the
`model_stylesheet` `class="launch"` selector; per finding 2 a class-less
agent also defaults to Haiku, so Haiku is the native default either way — no
`--model` needed). It is kept SIMPLE: launching is its whole job. The
judgment seam (goal (b)) is NOTED, not built out.

**How the work_id reaches the Haiku launch node.** Two channels, both used:

1. **Context channel (the wake + candidate).** The upstream native `watch`
   node echoes the wake line (the work_id it observed / the fact that pending
   is non-empty) to stdout; that output flows into the `launch` AGENT's
   context (agents receive upstream node context/prompt; native scripts do
   NOT — this asymmetry is exactly bridge reason (c)).
2. **Authoritative channel (what it actually dispatches).** The `launch`
   agent does NOT trust the watch line as authoritative. It reads
   `shop-msg pending inbox --bc "$BC_NAME"` ITSELF and dispatches over THAT
   set — the consumption-robust, idempotent no-matching-outbox source
   (finding 7). This closes G3 and makes the agent naturally a DRAIN: it
   spawns a child for EACH currently-pending work item, so a single wake that
   coincides with a backlog dispatches the whole backlog.

**Child-spawn command** (per pending work item `<W>`):

```
fabro run workflow.fabro -I BC_NAME=<bc> -I WORK_ID=<W> --parent <dispatcher-run> --detach
```

This is **today's proven one-shot argv** (`fabro run workflow.fabro -I
BC_NAME=<bc> -I WORK_ID=<W>`) with only `--parent <dispatcher-run>` and
`--detach` appended. Consequences of that fidelity:

- The child is the EXISTING Implementer→Reviewer def, UNCHANGED. One child =
  exactly one work item = one `work_done`, identical to today's one-shot.
- `WORK_ID` is delivered per-child by the same proven mechanism as today —
  no shared file, so no cross-child stomping.
- `--detach` backgrounds the child and prints its run ID; the agent does NOT
  `wait` on it — it returns immediately and the graph takes the `launch →
  watch` back-edge to keep watching.
- `--parent` links the child under the dispatcher for tracking / lifecycle
  via `fabro parent` / `fabro wait` (used off the hot path only).

**Launch-node prompt shape (simple):** *"You are the LAUNCH step ONLY. One or
more inbox work items are pending. Read the authoritative pending set via
`shop-msg pending inbox --bc $BC_NAME`. For EACH pending work_id W, spawn a
DETACHED CHILD: `fabro run workflow.fabro -I BC_NAME=$BC_NAME -I WORK_ID=W
--parent <this-run-id> --detach`. Do NOT wait on children, do NOT
process the work yourself, do NOT read-destructively consume the inbox, do
NOT emit work_done or shop-msg respond. This is the JUDGMENT SEAM — for now
spawn unconditionally; future dispatch judgment (dedupe / prioritize / defer)
plugs in here. Report the child run IDs and finish."*

The `<this-run-id>` for `--parent` is the dispatcher's own run ID, available
to the agent node from the fabro run environment (`FABRO_RUN_ID`-style) —
Open Q3.

**Cost / latency note (acceptable).** The dispatcher now makes a CHEAP Haiku
call once per wake / work item (the `launch` node). This adds a small
per-dispatch LLM cost and a small latency before the child is spawned. It is
acceptable: the call is Haiku (cheapest tier), it does no heavy reasoning
(its job is to read pending inbox and shell out `fabro run … --detach`), and
it buys goals (a) runtime-cyclic proof, (b) the judgment seam, and (c) the
per-iteration context bridge. When fabro command nodes gain per-iteration
env/arg/context, the Haiku node can be swapped for a native one and the cost
disappears (D3 forward-look).

##### D4 — Parallelism is NATIVE; no file, no semaphore

Multiple detached children run concurrently under the one parent, each
isolated by its own per-run `WORK_ID` env. Concurrency is fabro's own
orchestration; the design does not serialize. This is preserved unchanged
from the pivot: the prior single-run+file design could not run work items in
parallel without stomping a shared `/tmp/fabro_work_id` file, and a semaphore
to protect it would have re-serialized the fabro path. The cyclic dispatcher
does NOT re-introduce that problem — the per-iteration work_id lives in the
Haiku agent's context and the child's own `-I WORK_ID`, never a shared file.

##### D5 — Robustness: the dispatcher outlives any single child; the watch node folds in the startup-drain; child failures are isolated

- The dispatcher graph runs until shutdown (reactive parity) — it is the
  persistent process, matching the tmux `--startup-prompt` watcher lifecycle.
  The cycle `watch → launch → watch` never reaches `end` except on the
  `watch` stream-close (shutdown) edge.
- **Startup-drain** is folded into the `watch` node's entry check: before
  blocking on `shop-msg watch`, it inspects `shop-msg pending inbox --bc` and
  yields immediately if non-empty, so between-session messages are dispatched
  on the first cycle rather than waiting for a fresh NOTIFY (mirrors the
  lead-primer session-start drain).
- A failed / crashed CHILD does NOT kill the dispatcher: children are
  detached, so a child failure is isolated to its own run. The dispatcher
  keeps cycling. (A child's own infra-`halt` remains a FAILED terminal for
  THAT run, per ADR-051 / ADR-012 — surfaced via the child's mailbox, not by
  spinning the dispatcher.)
- A failed `launch` node (agent error / spawn error) takes the SAME
  unconditional back-edge `launch → watch` — log-and-continue, the dispatcher
  survives a bad dispatch attempt. The authoritative `pending inbox` source
  means an un-dispatched work_id is simply re-attempted on the next cycle.
- `fabro parent` / `fabro wait` are used minimally — for tracking / lifecycle
  only (e.g. shutdown-time reaping, D7), NOT on the hot path. The dispatcher
  does NOT `wait` on children before dispatching the next (that would
  re-serialize).

##### D6 — `--work-id` is optional/ignored at the launch interface (true parity)

`bc-container launch … --orchestrator fabro` no longer requires or consumes
`--work-id`; the fabro path takes no work_id at launch, exactly like tmux.
Any `--work-id` on the fabro path becomes an ignored no-op. This is the
interface half of the parity ADR-050 claimed.

##### D7 — Termination is the shutdown seam (lead-4qqn — NOTED only, not designed)

The dispatcher terminates when the `watch` stream closes (today
`bc-container stop` hard-kill; future graceful `request_shutdown` per
lead-4qqn drives the `watch → end` edge). At shutdown, in-flight detached
children either complete or are cancelled — `fabro parent` gives the
dispatcher the handle to enumerate/reap them if graceful shutdown wants that.
The cyclic watcher is precisely the receiver a graceful shutdown targets.
This seam is NOTED for lead-4qqn; it is NOT designed here.

#### ALTERNATIVE (documented, NOT recommended) — single-native-node bash-loop dispatcher

The 2026-07-07 pivot's first form made `dispatcher.fabro` a small def whose
real work was ONE native `script=` node running a bash `while` loop:
startup-drain (`shop-msg pending inbox --bc`) → spawn a detached child per
pending work_id → block on `shop-msg watch --bc` (skip `READY`) → wake →
re-drain, all inside the shell. It deliberately SIDESTEPPED the
cyclic-graph-runtime question (the only "cycling" was a `while` in one
node; the def graph was trivially acyclic).

```sh
while :; do
  for W in $(shop-msg pending inbox --bc "$BC_NAME" --work-id-only); do
    fabro run workflow.fabro -I BC_NAME="$BC_NAME" -I WORK_ID="$W" \
      --parent "$FABRO_RUN_ID" --detach
  done
  shop-msg watch --bc "$BC_NAME" | while read -r line; do
    [ "$line" = "READY" ] && continue
    break
  done
done
```

**Why NOT recommended (David 2026-07-07):** it does NOT prove fabro executes
a cyclic graph at runtime (goal (a)), it has NO agent judgment seam (goal
(b)), and it does not exercise the per-iteration-context bridge a native node
cannot provide (goal (c)). It remains a valid, lower-cost fallback if the
cyclic-graph or Haiku-node proof surfaces a blocker — it needs no per-node
model and no runtime-cycling property. Kept documented for that reason.

#### LESSER FALLBACK (documented, NOT adopted) — launcher-script watcher loop wrapping the one-shot `fabro run`

If the in-container proof (Open Q1) shows the cyclic graph AND/OR
`--parent --detach` does NOT give isolated parallel children, fall back to a
**launcher-tier** watcher entirely OUTSIDE fabro: a `drain-then-watch` shell
loop in the bc-launcher engage step — startup-drain `shop-msg pending inbox
--bc`, block on `shop-msg watch --bc`, per wake fire `fabro run … -I
WORK_ID=<W>` (as a detached background process directly, no fabro parent
link), failed run logged-and-skipped, terminate on shutdown. This closes G4
at the launcher tier with NO fabro-def change and was David's
originally-confirmed-feasible shape. It is strictly the safety net; the
cyclic-graph + Haiku-launch design (D1–D7) is preferred per David because it
keeps orchestration IN fabro AND proves cyclic-in-fabro on a low-stakes loop.

#### REJECTED / SUPERSEDED alternative — single persistent CYCLIC INNER run + per-iteration work_id FILE

The 2026-07-06 draft made `workflow.fabro` ITSELF cyclic: a `watch`/`drain`
node yields one work_id, the inner body processes it, a back-edge
(`emit_r/reported → drain`) re-enters. Per-iteration `$WORK_ID` moved from
the static overlay to a runtime FILE (`/tmp/fabro_work_id`) that every native
node re-reads.

**Rejected — the decisive reason is PARALLELISM.** A single shared work_id
file is stomped when multiple work items run concurrently; the design could
only stay correct by SERIALIZING behind a semaphore, which discards the
concurrency the fabro path should have. Secondary strikes: (a) it required
amending ADR-051 D1 (acyclic → cyclic) AND D3 (static overlay → file), a much
larger blast radius. NOTE: this ADR's recommended design ALSO makes a graph
cyclic — but the OUTER dispatcher, NOT the inner contract — so it keeps the
parallelism win (detached children, no shared file) AND ADR-051 untouched,
while STILL proving cyclic-in-fabro at runtime (the very property this
rejected draft could not de-risk). The parent/child + cyclic-dispatcher
design thus captures this draft's cyclic ambition WITHOUT its two fatal
costs.

#### Proof outcome — Open Q1 RESOLVED (2026-07-07, in-container proof wf_10649334-946)
Both make-or-break runtime facts CONFIRMED on a real fabro 0.254.0 server (throwaway isolated storage):
- **Q1a cyclic execution — CONFIRMED.** The engine follows the `launch → watch` back-edge and re-executes the loop at runtime (native `tick` node visited 3×, counter 1→2→3, terminal reached only via the `outcome=failed` edge; run SUCCEEDED).
- **Q1b parallel detached children — CONFIRMED.** `fabro run child --parent <p> --detach` spawns isolated children that run CONCURRENTLY (3 overlapping runs, confirmed twice), each a distinct parent-linked run carrying its own `WORK_ID` input.
- **Server-setup insight:** the fabro server needs `~/.fabro/settings.toml` with `[server.auth] methods` + a 64-hex `SESSION_SECRET` + a `FABRO_DEV_TOKEN` of form `fabro_dev_`+64hex (NOT bare hex — the prior "invalid format" cause). `fabro install --non-interactive --skip-llm --github-strategy token` generates a valid one.

##### Design caveat carried into implementation
`-I WORK_ID` reaches the child run INPUTS, goal, and AGENT prompts (`{{ inputs.WORK_ID }}`) — but NOT native `script=` node env (the `[run.environment.env]` overlay is applied LITERALLY; minijinja is not evaluated on it). So the child's native per-work nodes cannot be parameterized by `-I` alone. Resolution (consistent with the existing bc-shop loop's Slice-5 pattern): the dispatcher's **Haiku launch node writes the CONCRETE `WORK_ID` into each child's env overlay at spawn** (an agent can do this substitution; a native node cannot — a further reason the launch node is a Haiku agent). No regression; the current launcher already writes the concrete value at launch.
