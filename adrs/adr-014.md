## Source (pre-modernization)

#### Context

PDR-010 diagnoses two failure modes in the lead shop's reactive-only
stance. Failure mode B: a BC is not running (container down, agent
crashed, launcher never started it). The lead has no liveness signal —
Monitor produces no event because there is nothing on the other end to
produce one. The dispatch sits in the BC's inbox unobserved; the lead
treats Monitor silence as "BC is processing" rather than "BC is absent."

**Empirical record (2026-05-29):** `lead-ji28` dispatched scenarios to
`shopsystem-scenarios`. The BC's container was down. The dispatch landed
in the inbox; no Monitor event arrived (none could). The lead's reactive
stance produced no proactive check; the stall was discovered only when
the operator asked "why hasn't scenarios responded?" Under the rule this
ADR proposes, the absence would have classified definitively as `offline`
within 90 seconds of the BC's last tick, and the session-start primer
would have surfaced the stall before the dispatch was even sent.

The cost-free path is to piggy-back the heartbeat onto a process that is
already running anyway. `shop-msg watch --bc <name>` is the BC-side
long-running process that holds the postgres LISTEN connection for inbox
wakeup. It is already connected to postgres; it is already supervised
by the launcher; it is already the canonical "BC is reachable for
wakeup" signal — implicitly. This ADR makes that signal explicit and
queryable.

#### Decision

1. **`shop-msg watch --bc <name>` SHALL upsert a row** in a `bc_presence`
   table on a tick interval. The row schema is
   `(bc_name TEXT PRIMARY KEY, last_seen_at TIMESTAMPTZ NOT NULL,
   watch_session_id UUID NOT NULL)`. The UPSERT is keyed on `bc_name`;
   subsequent ticks from the same or a different watch process update
   the same row.

2. **Default tick cadence: 30 seconds.** The cadence is configurable via
   `--heartbeat-interval` on `shop-msg watch` but defaults to 30s. The
   choice trades responsiveness against postgres write volume; at 30s
   per BC, a 10-BC topology produces ~20 writes/minute against
   `bc_presence`, which is negligible.

3. **Stale-detection thresholds** (lead-side classification, computed
   from `now() - last_seen_at`):
   - `online`: under 90 seconds. Tolerates one missed beat without
     flapping.
   - `stale`: between 90 seconds and 5 minutes. Likely transient — a
     postgres LISTEN drop with reconnect in progress (per `lead-tsj`),
     a brief network hiccup, or a tick skew under load.
   - `offline`: beyond 5 minutes. The BC is not running, or its watch
     process is wedged in a way that prevents ticking.

4. **`shop-msg bc-status [--bc <name>]` SHALL read `bc_presence`** and
   report the classification. With `--bc`, the command reports one BC's
   state; without, it reports all known BCs. Output is one row per BC:
   `<bc_name> <classification> <seconds_since_last_seen>`.

5. **Reconnect semantics.** When `shop-msg watch` reconnects after a
   LISTEN drop (the path canonized by `lead-tsj`), it resumes ticking
   on the same cadence; the next tick UPSERTs `last_seen_at = now()`
   and the BC re-enters `online`. There is no backfill of missed ticks
   — the gap is informational, recoverable by inspection of the prior
   `last_seen_at` value if needed, and is not load-bearing for any
   downstream consumer.

6. **Multi-watcher resolution.** If two `shop-msg watch` processes are
   running for the same BC (rare, primarily during launcher transitions
   when an old watch has not yet exited and a new one has started), both
   UPSERT the same `bc_presence` row, racing on `last_seen_at`. The
   `watch_session_id` field is informational only — it records which
   session most recently ticked, useful for forensics but not for
   classification. The lead cares only whether *anyone* is watching,
   not how many.

7. **Lead-side standing rule (session-start).** After arming Monitor and
   draining `shop-msg pending outbox --lead <name>` /
   `shop-msg pending inbox --lead <name>`, and **before accepting user
   work**, the lead SHOULD run `shop-msg bc-status` and surface any BCs
   in `offline` (or, at the lead's discretion, `stale`) as a stall
   signal. This is the proactive-nudge primitive that closes failure
   mode B from PDR-010: the lead learns of a down BC at session start,
   not on dispatch.

#### Alternatives considered

**Option A — Separate `shop-msg presence-tick` daemon.** Rejected. A
standalone tick process is one more thing for the launcher to start,
supervise, and tear down. It duplicates `shop-msg watch`'s
postgres connection and its supervision surface. Every property the
heartbeat needs (long-running, postgres-connected, lifecycle-tied to
the BC agent) already holds for `watch`. The collapse is not just an
optimization — it makes the two signals (reachable-for-wakeup, and
recently-alive) provably congruent because they are emitted from the
same process.

**Option B — Heartbeat via the BC's bd registry.** Rejected. The BC's
bd database is a private detail per PDR-010's loose-cross-shop
principle; reading or writing it from the lead crosses an abstraction
the spec specifically protects. The lead has no business knowing the
BC's bd schema. The presence channel must be a shop-msg-mediated
artifact, which means a table the messaging BC owns.

**Option C — Heartbeat in the container-orchestration layer (Docker
health checks, k8s liveness probes).** Rejected. Container-level health
checks tell the orchestrator that a process is running; they do not
tell the lead's wakeup channel that the agent on the other side is
ready to receive a dispatch. The signal the lead needs is
app-protocol-level (is the watch process actively LISTENing on
postgres?), not OS-level (is the container up?). Conflating them
produces false-positive `online` classification when the container is
up but the watch process has wedged.

#### Consequences

- **BC launcher invocation.** The launcher's `shop-msg watch` command
  invocation needs to either (a) include the heartbeat behavior
  unconditionally inside the `watch` command itself (the chosen path —
  no flag gate, heartbeat is part of what `watch` *is*), or (b) be
  wrapped in a script that ticks alongside it. (a) is the simpler path
  and is what this ADR commits to; the implementation change is a one-
  liner inside the watch loop. A follow-up bead tracks the
  `bc-launcher` repo edit to confirm no additional flags are needed.

- **Lead session-start primer.** The lead-shop canonical primer
  (`.claude/canonical/lead-primer.md`) gains a `shop-msg bc-status`
  step in the session-start drain block, between "drain pending" and
  "accept user work." A follow-up bead tracks the primer edit in
  `shopsystem-templates` so it propagates to all lead shops.

- **`shop-msg` schema and CLI additions.** The messaging BC adds:
  (i) a `bc_presence` table with the schema in decision (1);
  (ii) the heartbeat-tick behavior inside `shop-msg watch --bc <name>`;
  (iii) the `shop-msg bc-status` subcommand with the read/classify
  behavior in decisions (3) and (4). The work is dispatched as an
  `assign_scenarios` to `shopsystem-messaging` against a fresh lead bead
  carrying Gherkin for each behavior: tick-on-cadence, UPSERT-on-bc-name,
  classify-by-threshold, multi-watcher-shared-row, reconnect-resumes-
  ticking.

- **No backwards-compat concern.** `bc_presence` is a fresh table; the
  `shop-msg watch` command's pre-ADR behavior (LISTEN, emit events on
  notify) is preserved unchanged. BCs running an older `shop-msg`
  release will simply not tick, and will classify as `offline` at the
  lead — which is the correct, fail-safe behavior during the rollout
  window.
