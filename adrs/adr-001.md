## Source (pre-modernization)

#### Decision

Promote the shopsystem framework out of `prototypes/message-catalog-v1/`
into four public repositories under `github.com/dstengle/`, organized by
**bounded context of the shopsystem itself** rather than by technical
concern or shop type.

| Repo | Bounded context | What it owns |
|---|---|---|
| `shopsystem-product` | Lead shop of the shopsystem product | Principles, BC definitions, inter-shop protocol description, work-tracking conventions, ADRs (including this one), findings docs from each prototype, the **scenario register** listing what scenarios the framework's behavior is pinned by |
| `shopsystem-messaging` | Messaging BC | Pydantic schemas for the 8 message types + the `shop-msg` CLI + `features/` directory pinning the messaging scenarios authored by the product-shop |
| `shopsystem-scenarios` | Scenarios BC | Gherkin canonicalization rule + hash + the `scenarios` CLI + `features/` directory pinning the canonicalization scenarios. Will grow more tools (scenario-register management, possibly scenario authoring helpers) |
| `shopsystem-templates` | Templates BC | All four role templates (lead-po, lead-architect, bc-implementer, bc-reviewer) + the `shop-templates` CLI + `features/` directory pinning the template structural tests |

#### Context and reasoning

The earlier framing of "split packages by technical concern" produced four
packages — `catalog`, `scenarios`, `shop-templates`, `shop-msg-bc` — but
two of them (`shop-msg-bc` and `shop-templates`) mixed concerns: the
shop-msg CLI's UL (message, work_id, inbox, outbox) is distinct from the
templates' UL (role, prompt, sufficiency check, anti-rationalization).
Bundling them as a "BC-shop" package or "lead-shop" package mixed two
ULs in one repo — the same anti-pattern bounded contexts exist to prevent
in product code.

Reframing by BC of the shopsystem itself produces packages where each
has one UL:

- **Messaging UL:** message_type, work_id, schema, validation, inbox,
  outbox, send, respond, read, wire format.
- **Scenarios UL:** scenario, Gherkin, canonical, hash, tag (`@bc:`,
  `@scenario_hash:`), feature, step definition, register, assignment.
- **Templates UL:** role, template, prompt, dispatch, sufficiency check,
  anti-rationalization, discriminator, implementer, reviewer, PO,
  architect.
- **Product (lead-shop) UL:** principle, BC definition, brief, PDR, ADR,
  Domain & Context Map, scenario register, work tracking, findings.

The lead-shop content (formerly `docs/shop-system/`) is product-level
content for the shopsystem-as-a-product. "Spec" was framework-internal
language for what is, structurally, the shopsystem's lead-shop output —
the same kind of artifact every consumer product produces in its own
lead shop. Naming the repo `shopsystem-product` matches the DDD framing:
this is the product side of the shopsystem.

The shopsystem now follows its own discipline. It is a product with a
lead shop and three BCs; each BC speaks one UL; the inter-shop protocol
applies recursively (the shopsystem's lead shop can dispatch scenarios
to its own BCs via `assign_scenarios`, just like any consumer product
would).

#### Naming conventions

- **No "shop" suffix in repo names.** The org / product prefix carries
  the structural context; the BC name carries the UL. Examples:
  `shopsystem-product`, not `shopsystem-product-shop`;
  `shopsystem-messaging`, not `shopsystem-messaging-bc`. Applies to
  consumer products too — an ecommerce product's BCs would be named
  `ecommerce-billing`, not `ecommerce-billing-shop`.
- **No deferred-rename of the CLIs in phase 1.** The `shop-msg`,
  `scenarios`, and `shop-templates` CLI binaries keep their current
  names. Renaming them touches every template, BDD step regex, and
  doc reference; out of scope for the packaging move. Eligible for a
  future rename phase if the naming feels wrong in practice.

#### Dependency direction

| Direction | Reason |
|---|---|
| `shopsystem-messaging` → `shopsystem-scenarios` | The messaging BC's `ScenarioPayload.hash` invariant needs the canonicalization rule. The current prototype duplicates this rule in `catalog/schemas.py` to avoid importing from scenarios; with the new structure, importing is fine because no scenarios → messaging dependency exists (scenarios is purely Gherkin canonicalization; no message types). |
| `shopsystem-templates` standalone | Templates are markdown + a thin CLI; no dependency on messaging or scenarios. |
| `shopsystem-product` standalone | Documentation + scenario register. No runtime code dependencies. |

Consumer products will pull in `shopsystem-messaging`,
`shopsystem-scenarios`, and `shopsystem-templates` as dependencies via
git URLs in phase 1; both lead shops and BC shops will pull in subsets
appropriate to their role (lead shops need all three; BC shops need
messaging + templates).

#### Phase-1 wiring

- **Public GitHub repos** under `github.com/dstengle/`.
- **Git URLs in `pyproject.toml`** for dependencies (e.g.,
  `shopsystem-scenarios @ git+https://github.com/dstengle/shopsystem-scenarios@v0.1.0`).
- **Tags for versions** starting at `v0.1.0`. SemVer once shape stabilizes.
- **PyPI publishing deferred** until the shape is stable enough that
  package names can be claimed without rework.
- **No CI yet** beyond pytest on push. Cross-package integration tests
  live in `shopsystem-messaging` (the only repo that imports both
  messaging types AND uses scenarios canonicalization).

#### Migration sequencing

1. **Pre-extraction prep in this repo.** Move the cross-package
   `test_canonical_hash_matches_scenarios_package` test out of
   `catalog/tests/` (where it currently sits with hard-coded hashes)
   into a real integration test in `shop-msg-bc/tests/integration/`
   that imports `catalog._canonical_scenario_hash` AND calls
   `scenarios.hash.compute_scenario_hash`, asserting identical output.
2. **Create the four repos** on `github.com/dstengle/`. Add minimal
   README + LICENSE + pyproject.toml with repository URL metadata.
3. **Extract each package's contents.** Three options for history:
   `git subtree split` (preserves per-file history), `git filter-repo`
   (more surgical), or fresh commits with a summary referencing this
   repo. Choice deferred to extraction time based on what proves
   convenient.
4. **Tag `v0.1.0`** in each repo.
5. **Switch the prototype's editable installs to git URLs** in the
   current repo's `pyproject.toml` (or fresh consumer pyprojects when
   real product BCs exist). Run all tests; confirm green.
6. **Remove the in-tree copies** from `prototypes/message-catalog-v1/`.
   The prototype directory becomes a thin frozen-evidence record
   referenced by the findings docs that now live in `shopsystem-product`.
7. **Migrate `docs/shop-system/` content** into `shopsystem-product`.
   The current ddd-product-system repo can be archived once all
   content has migrated, or repurposed for the first consumer product
   (e.g., the ecommerce-app this work was originally framed around).

#### What this defers

- **CLI renames** (`shop-msg` → `shopsystem-msg`, etc.). Touches
  templates, BDD steps, docs. Phase 2 if it feels needed.
- **PyPI publishing.** Phase 2 once shapes stabilize.
- **CI / release automation** beyond per-repo pytest. Phase 2.
- **Org-level promotion** from `github.com/dstengle/` to a dedicated
  org. Naming convention (no "shop" suffix; consistent prefixes)
  makes this a cosmetic move when ready.
- **`bc-shop` / Temperature toy BC.** Stays in the prototype directory
  as historical evidence; not promoted. Future consumer products
  bootstrap from scratch using the four framework repos.

#### What this depends on

This ADR depends on prototype-1 finding 6 (package boundaries hold
under real-consumer pressure when dogfooding is preserved) and
mechanism-observation-v1 finding 1 (the catalog mechanism scales to a
new message type without structural changes). Both validated that
package-as-boundary is the right primitive; this ADR applies the same
discipline at the repo-as-boundary level, with BC-of-the-shopsystem-
itself as the boundary unit.
