## Context

**The question.** Under PDR-032 the shopsystem-knowledge BC OWNS the artifact
type system — the conventions document, the per-type templates, the frontmatter
JSON Schema, and the coherence gate. PDR-032 left those templates and that
schema as **hand-authored artifacts kept in sync by review**. How should the
knowledge BC MANAGE those formats so that the template a human copies and the
schema the gate enforces cannot drift apart?

**The forcing incident is not hypothetical: template↔schema drift occurred
inside the handoff package within a single authoring pass.** The pkg1 PDR
template frontmatter showed `derives-from` (and `description`) as fields, but
the JSON Schema for that same type did NOT require them — the template promised
a shape the schema did not enforce. This is the same class of defect the
Round-1 architect caught reconciling the payload against PDR-031's pinned
surface: two hand-maintained representations of ONE format, edited
independently, silently diverging. Review is the only thing standing between
that drift and a shipped format, and review missed it once already in a single
pass.

The handoff package carried a PoC (`payload/typedef-poc/`) testing the
assumption that this drift can be eliminated **by construction rather than by
review**. A per-type `typedef/*.yaml` (e.g. `typedef/pdr.yaml`) declares, in
one place: the field set with per-field type / required / min-items / link
semantics, the id pattern, the status enum, and the body sections with
guidance. A ~120-line generator (`gen.py`) emits BOTH the template (`pdr.md`)
and the schema fragment (`pdr.schema.json`) from that one source. A `--check`
mode regenerates to a temp dir and diffs against the committed outputs, exiting
nonzero on any mismatch — so a hand-edit of a generated file fails CI. The PoC
round-tripped the PDR type with full fidelity (frontmatter comments, section
guidance, conditional link rules) and drift-detection was verified by a
deliberate hand-edit. It also surfaced a bonus: because the typedef marks which
body sections are `required`, the generator emits an `x-required-sections` list
into the schema fragment, giving the coherence gate a **body-structure check
that was previously unenforced** (nothing checked that a PDR actually carried
its `## Options considered` heading).

This is an implementation decision UNDER PDR-032's charter — it does not re-open
who owns formats (the knowledge BC does), only HOW that BC mechanizes the
ownership. It is a natural generalization of ADR-019's single-source doctrine
(one canonical scenario text, byte-identical everywhere) from the
scenario-block domain to the artifact-format domain.

**Options considered.**

*Option A — Hand-authored templates + schema, kept in sync by review (the
PDR-032 status quo ante).* Templates and the frontmatter JSON Schema are
authored and edited directly, as two independent files per type; the coherence
gate and code review are relied on to catch divergence. Pros: zero new tooling;
directly readable/editable; what PDR-032 already assumed. Cons: it is exactly
the model that already FAILED — drift entered within a single authoring pass and
only a careful architect caught it. Two representations of one format, edited by
hand, will diverge; review is a probabilistic gate, not a structural one. Every
new artifact type doubles the hand-maintenance surface. Rejected: it re-installs
the defect this decision exists to remove.

*Option B — Schema-first: the JSON Schema is the single source, the template is
generated from it.* Author only the JSON Schema per type; derive the human
template from the schema. Pros: one source; JSON Schema is a standard the gate
already consumes directly, so the "authored = enforced" artifact needs no
round-trip. Cons: JSON Schema is a poor home for the human-facing concerns a
template carries — field-ordering intent, inline guidance comments, section
prose, `<placeholder>` scaffolding, and "empty only for root decisions" style
nuance have no natural schema expression, so they would be smuggled into
`description`/`x-` extension keys and reconstituted. The authoring surface (a
schema) is also the WORST surface for a human to reason about the format's
INTENT. It inverts the ergonomics: the thing humans read most (the template)
becomes a second-class derivation of the thing machines read. Rejected:
single-source is achieved, but on the wrong canonical artifact.

*Option C — Typedef single-source: a per-type `typedef/*.yaml` generates BOTH
template and schema; generated files read-only; `--check` drift gate (CHOSEN).*
The canonical source is a per-type `typedef/*.yaml` expressing the format at the
level of INTENT (fields + semantics + sections + guidance). A generator
deterministically emits the template `.md` AND the schema fragment `.json`; both
are marked GENERATED / read-only; a `--check` mode fails CI on any hand-edit.
Pros: drift is eliminated by CONSTRUCTION — the template and schema are two
projections of one source and cannot disagree, so review is freed from a job it
was failing. The canonical artifact is authored at the right level (intent, not
schema mechanics), carrying human concerns (comments, guidance, ordering) AND
machine concerns (types, enums, patterns, link rules) in one legible file.
Generalizes the proven ADR-019 single-source doctrine. Surfaces
`x-required-sections` as a net-new body-structure gate check at no extra
authoring cost. The PoC demonstrated full-fidelity round-trip and verified drift
detection. Cons: introduces a generator the knowledge BC must own, test, and
keep deterministic (a `--check` false-negative would silently re-admit drift);
adds a build step to format-authoring; one more concept for a newcomer to learn.
Accepted as the cost of structural (vs. probabilistic) drift-freedom.

## Decision

**Option C.** The shopsystem-knowledge BC manages each artifact format via a
per-type `typedef/*.yaml` **single source**. A generator emits BOTH the
template (`.md`) and the JSON Schema fragment (`.json`) from each typedef; **the
generated files are READ-ONLY**; a `--check` **drift gate fails CI on any
hand-edit** of a generated file. This ELIMINATES template↔schema drift by
construction rather than by review, and it **supersedes the hand-authored
templates + schema approach recorded in PDR-032's Consequences** — under this
decision the templates and the schema are GENERATED artifacts, not hand-authored
ones. The generator also emits **`x-required-sections`** into each schema
fragment, giving the coherence gate a body-structure check (per ADR-047's gate
host) that was previously unenforced.

**Scope.** This is scoped, per PDR-032, to the knowledge BC's OWNED formats (the
8 artifact types: intent-record, candidate, session-record,
prioritization-record, brief, pdr, adr, current-state). It does not govern
role-template CONTENT (owned by the templates BC), and it does not touch
artifact INSTANCES (each product's lead-repo data). It refines PDR-032's
ownership charter at the implementation layer; it does not re-open the ownership
question.

**REQUIRED RECONCILIATION — `description` must be added to `SHARED_REQUIRED`.**
The PoC generator hard-codes:

```
SHARED_REQUIRED = ["type", "id", "title", "status", "created", "updated", "authors"]
```

This omits **`description`**. The adopted model **MUST add `description` as a
required shared field** — non-negotiable per **PDR-032 item 3**, which KEEPS
`description` as a required frontmatter field: it is pinned by scenario
`d121b489919c177e` (the L0 card projection draws
`id`/`title`/`status`/`description` from frontmatter), and the handoff payload
schema had already dropped it once. **The PoC's omission of `description` from
`SHARED_REQUIRED` is a defect to be corrected when the model is dispatched, NOT
a property to be carried forward.** Recorded explicitly so it is not lost
between this decision landing and the BC dispatch (`lead-oji4`): the shared
required set the generator emits is `[type, id, title, description, status,
created, updated, authors]`.

## Consequences

**Easier:** template↔schema drift becomes structurally impossible, not merely
review-caught — the class of defect that hit the handoff package within one
authoring pass cannot recur. Formats are authored once, at the level of intent,
in one legible file per type. The gate gains a free `x-required-sections`
body-structure check. The single-source doctrine (ADR-019) now spans both
scenario blocks and artifact formats, one fewer special case.

**Harder:** the knowledge BC now owns and must test a generator plus a `--check`
CI gate (a `--check` false-negative silently re-admits drift, so the gate's own
correctness is load-bearing); format authoring gains a regenerate step; the
generated files must be marked read-only in a way humans notice before editing
them (the `# GENERATED … do not edit; edit the typedef.` header the PoC emits).

**Supersession:** PDR-032's Consequences treated the artifact templates and the
frontmatter JSON Schema as hand-authored artifacts kept in sync by review. Under
this ADR they are GENERATED, read-only, and drift-gated. PDR-032's OWNERSHIP
decision (knowledge BC owns the type system + gate) and every other item (the
`type` unification, `description`/`created`/`updated`/`authors` reconciliation,
gate rules 4–8) are UNCHANGED — only the templates-and-schema-are-hand-maintained
implementation posture is superseded. A back-reference is recorded on PDR-032
(Amended-by / Refined-by note + Changelog), mirroring the PDR-031↔PDR-032 and
PDR-012↔PDR-033 bidirectional-linkage pattern.

**Downstream (not this landing):** the BC dispatch on `lead-oji4` carries this
model to the shopsystem-knowledge BC INCLUDING the `description` reconciliation
above; the PO re-authors the scenario register on `lead-zo9e`. The two spike
assumptions `cand-P01` still flagged — (1) a gate prototype over the existing
corpus with legacy exemptions produces signal not noise, (2) the pour can
distribute non-role assets without structural change (the PDR-032 item-7 fenced
follow-on) — remain open and are NOT resolved here.
