## Context

A `mechanism_observation` from `shopsystem-bc-launcher`
(`shopsystem_bc_launcher-5xbg`, observed during `lead-6tu6o.1`) found that a
lead dispatch instructed retiring `@scenario_hash:76badc67216f0d91` with "the
body kept BYTE-IDENTICAL on-disk for hash-provenance, following the same
retirement convention already used twice in this file." That instruction is
mechanically **unsatisfiable**: `bc-emit work-done --retire-hash <h>`
hard-refuses the emit while `<h>` remains reachable — recomputable — from any
scenario block under the as-committed `features/` tree, whether on the old block
left in place or duplicated onto a new one
(`features/shopsystem-templates/bc_emit_work_done_retirement_removal.feature`,
`@scenario_hash:777dcc6bfe4bdd03` / `800e7f9317a1b884`; ADR-042 D1(iii)). A BC
complying literally with such an instruction does the full implementation and
only discovers at emit time that the work is unshippable — the late-detection
failure shape this shop's own gates exist to prevent elsewhere.

The BC's own diagnosis, verified by a throwaway-worktree counterfactual
(reconstructing the literal "keep byte-identical" instruction and observing a
hard refusal, exit 1): **"no single authoritative statement of the retirement
convention that both lead dispatch-authoring and the `bc-emit --retire-hash`
precondition read from, so the lead can author prose the wrapper refuses."** The
dispatch was also wrong about its own evidence: the file's actual convention —
delete the retired body, keep only a provenance comment outside any canonical
hash region — had been used, not twice, but repeatedly, with zero retired bodies
surviving on disk.

Pre-state empirical findings (artifact surface only, ADR-018 D1/D2). Verified
2026-07-15 against this repo's `adr/`, `features/`, and the installed `scenarios
hash`/`grep` tools. No BC source read, run, or git-observed — there is no
`repos/` tree on the lead host.

1. **The wrapper's mechanical precondition is already pinned and unchanged.**
   `features/shopsystem-templates/bc_emit_work_done_retirement_removal.feature`
   pins: the emit is refused while any scenario hash a consumed dispatch named
   for retirement is still reachable under the as-committed `features/` tree (on
   the old block or duplicated onto a new one), and is satisfied only once every
   such hash is absent — independent of any newly added blocks carrying fresh
   hashes. This is ADR-042 D1(iii)'s `scenario_hashes-match` check, block-only
   recompute, refusing on stale/missing/orphan.
2. **No existing ADR states the comment-placement convention that satisfies (1)
   in practice.** `grep -rl "provenance" adr/ pdr/` returns files about
   skills/image-label provenance, none about scenario-retirement comment
   placement. The nearest neighbor, ADR-056 D4.6/D5, establishes that
   canonicalization is block-only and tag/comment/`Feature:`-insensitive (and
   that `E_HASH_MISMATCH` is the validator's code for a hash that doesn't match
   its block) — but does not itself instruct where to place a retirement's
   provenance text, and no ADR connects that rule to a named retirement
   convention.
3. **The convention the BC verified is confirmed independently, on-disk, in
   this lead's own copy of the file.**
   `features/shopsystem-bc-launcher/fabro_llm_provider_openrouter_override.feature`
   carries **7** `RETIRED (...)` provenance headers (`grep -c "RETIRED ("` = 7)
   and **7** live `@scenario_hash:` tags (`grep -n "^  @scenario_hash:"` = 7);
   **none** of the 7 retired hashes (`b3054f5439369fa8`, `14290420156c5ee0`,
   `4c9f5b265c5098b7`, `98b956adece2b7e0`, `22f2a5bda5c29044`,
   `c99e79ac24f56f5c`, `76badc67216f0d91`) appears as a live tag. All 7
   provenance blocks sit in a comment region above the
   `@bc:.../@origin:...`-tagged `Feature:` line — i.e., outside any
   `Scenario:`-headed canonical region — consistent with the block-only,
   comment-insensitive canonicalization (finding 2) never seeing them.

**Alternatives considered.**

*Option A — Leave the convention as unwritten, BC-local precedent.* Rejected:
this is precisely the gap the mechanism_observation surfaced. An implicit,
per-file precedent does not stop a dispatch author from asserting a contradictory
instruction from memory, as happened in `lead-6tu6o.1`; a BC without an
adversarial reviewer willing to deviate from an explicit instruction would have
complied literally and stalled at emit.

*Option B — Fold this into ADR-042 as an amendment.* Rejected: ADR-042 is itself
scoped and disciplined to the 105-116 retirement leg, and its own D4 explicitly
rejects re-litigating settled ground with a competing decision ("do not draft a
competing layering ADR; this ADR is reconciliation, not re-decision"). Amending
it to add a distinct citation-target/comment-placement decision would blur that
scope discipline. A separate, narrowly-scoped ADR keeps each decision
independently citable, matching this repo's existing granularity (e.g., ADR-060
split out wire-hash canonicalization alignment from ADR-019 rather than folding
in).

*Option C — Encode the convention only in the `bc-emit` wrapper's refusal
message, with no ADR.* Rejected: the wrapper's refusal fires at emit time, after
full implementation — exactly the late-detection failure shape this fix exists to
prevent for dispatch-authoring, which acts *before* emit. Dispatch-authoring
needs a citable pre-emit source, not a post-hoc error string.

## Decision

**D1 — The mechanical precondition is cited, not re-decided.** A retired
`@scenario_hash:<h>` satisfies `bc-emit work-done --retire-hash <h>` **only**
when `<h>` is unreachable by block-only recompute from every scenario block
under the as-committed `features/` tree — not merely untagged, not merely absent
a live `@scenario_hash:<h>` tag, but absent as a **recomputable hash of any live
block's text**. This is ADR-042 D1(iii), pinned by
`features/shopsystem-templates/bc_emit_work_done_retirement_removal.feature`.
This ADR adds no new mechanics here; it names the existing pin as the canonical
citation target.

**D2 — The satisfying convention: delete the body, provenance outside any
canonical region.** To satisfy D1, a retirement:

1. **Deletes** the retired scenario's `Given`/`When`/`Then` body from the live
   scenario-block region — never keeps it byte-identical, tag-stripped, or
   otherwise reachable in a form block-only recompute would reproduce.
2. **Records provenance** (which hash, why retired, what supersedes it, and —
   for audit — the original body text) in a **comment block placed outside any
   canonical scenario region**: above the `Feature:` line, or otherwise never
   inside a `Scenario:`-to-next-blank-line span. Per ADR-056 D4.6/D5,
   canonicalization is block-only and comment/tag/`Feature:`-insensitive, so a
   provenance comment placed outside a canonical region never affects any live
   hash. Placing it **inside** a canonical region is a known failure mode:
   `E_HASH_MISMATCH`.

This is the convention verified in finding 3 (D1 evidence), used 7-for-7 with
zero survivors in the lead's own held copy of
`fabro_llm_provider_openrouter_override.feature`.

**D3 — This ADR is the canonical citation target for lead dispatch-authoring.**
Lead dispatch-authoring — concretely, the `lead-architect` template's
retirement-instruction prose, composed whenever a dispatch retires, supersedes,
or contradicts prior BC-side scenario coverage — cites D1/D2 of this ADR by
reference rather than restating the retirement convention from memory. This
closes the gap the mechanism_observation named: there was no single
authoritative statement that both dispatch-authoring and the `bc-emit
--retire-hash` precondition read from, so dispatch-authoring could (and did)
author prose the wrapper mechanically refuses. A companion `assign_scenarios`
dispatch to `shopsystem-templates` (see Consequences) pins a new scenario
requiring the `lead-architect` template to cite this ADR, extending the existing
`lead_architect_empirical_pre_state_contract_surface.feature` pattern rather
than introducing a new one.

  `lead-po`) dispatch-authoring that composes a retirement instruction cites
  this ADR's D1/D2 rather than restating from memory.
- **Companion template dispatch.** A fresh lead bead / `assign_scenarios`
  dispatch to `shopsystem-templates` pins a new scenario requiring the
  `lead-architect` template to cite this ADR when composing
  retirement-instruction prose, extending
  `features/shopsystem-templates/lead_architect_empirical_pre_state_contract_surface.feature`.
  Tracked separately (see dispatch report); this ADR does not by itself change
  template content.
- **No change to enforced mechanics.** `bc-emit --retire-hash`'s precondition
  (ADR-042 D1(iii)) and the block-only canonicalization rule (ADR-056 D4.6/D5)
  are unchanged; this ADR only names them as a joint citation target and records
  the comment-placement convention (D2), which was previously tacit and
  observed-only.
- **Connection to intent-004.** `intent-004` (recorded 2026-07-14) captures the
  broader theme that subagent citations/conventions drift because there is no
  provenance-aware, authoritative source subagents read from, only restatement
  from memory. This ADR is a narrow, concrete instance of that same class of
  gap, resolved here for the retirement-convention case specifically; it does
  not resolve `intent-004`'s broader retrieval-scoping / mechanical-citation-
  validation ask, which remains open for future shaping.
