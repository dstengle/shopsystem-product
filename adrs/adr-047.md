## Source (pre-modernization)

#### Context

PDR-030 settled the product-level model: the shopsystem system version is an
independent standard semver (D1) mapped to a component tuple by a lead-owned
declarative YAML manifest (D2), the component release cadence is unchanged and
composes into it (D3), coherence validation is advisory at dev-time and
blocking at adopter stand-up (D4), and BC self-tagging is deferred (D5). PDR-030
explicitly framed four mechanism questions for this ADR: the exact schema, the
release-tool wiring, the coherence-check internals, and how each reconciles with
ADR-039/ADR-021/lead-h2p0/lead-5xnd. This ADR settles those.

#### Decision

##### D1 — Schema: `system-manifest.yaml` is a `releases:`-history list at repo root; each entry pins `system_version` + a `components:` map keyed by canonical `shopsystem-<id>`; dependency metadata is REFERENCED, not embedded; in-file history is the source of truth, git tags are an additive pointer (implements PDR-030 D2)

The manifest is `system-manifest.yaml`, committed at the lead repo root, sibling
to `bc-manifest.yaml`. Its top-level shape is a **`releases:` history list**, not
a single-version file:

```yaml
### system-manifest.yaml — the lead-owned Bill of Materials (PDR-030, ADR-047).
### Maps each shopsystem system semver to the coherent component tuple it composes.
current: 1.3.0                 # the active system release (resolves to a releases[] entry)
releases:
  - system_version: 1.3.0      # standard semver, on its own line (PDR-030 D1, product-chosen)
    components:                # the BOM proper — canonical shopsystem-<id> keys
      shopsystem-templates: 0.48.0
      shopsystem-bc-launcher: 0.3.41
      shopsystem-messaging: 0.4.4
      shopsystem-scenarios: 0.2.0
  - system_version: 1.2.0
    components:
      ...
```

- **A `releases:` list, not a single-version file.** The BOM's job is to resolve
  "shopsystem vN" to its tuple for ANY N an adopter might be running (the
  lead-6b53 motivation). A single-version file loses the historical mapping; an
  adopter on an older tuple could no longer resolve its own version. Prior system
  versions are therefore retained **in-file** as `releases[]` entries.
- **In-file history is the source of truth; git tags on the lead repo are an
  ADDITIVE pointer, not the source.** The manifest must be resolvable from the
  file alone — adopters bootstrap from a starter fork (ADR-040, deterministic and
  agent-less) and may not carry full lead-repo git history. Optionally tagging the
  lead repo `system-vN` at each release is a convenience cross-reference; it is
  not where the tuple is read from. (BC-repo tag-reflection remains the deferred
  PDR-030 D5 leg — out of scope.)
- **Content floor (PDR-030 D2):** each `releases[]` entry carries
  `system_version:` (the D1 semver, product-chosen — the tool records it, never
  computes it) and a `components:` map. `current:` names the active release.
- **Component keys are the canonical `shopsystem-<id>` form** that `bc-manifest.yaml`
  uses (pre-state finding 1), so the two sibling manifests align on identity and
  the BOM's component set can be cross-checked against fleet membership. The tuple
  carries at minimum the four released component lines (`shopsystem-templates`,
  `shopsystem-bc-launcher`, `shopsystem-messaging`, `shopsystem-scenarios`);
  additional components are additive map entries.
- **lead-h2p0 dependency metadata is REFERENCED, not embedded.** The manifest
  pins the tuple only; it carries NO dependency-edge fields. Whether a tuple is
  internally coherent is lead-h2p0's view, CONSUMED by the validator at check time
  (D3), not duplicated into the manifest (PDR-030 D2 / Brief 015 §4: consume
  lead-h2p0, do not re-derive it).

##### D2 — Release-tool wiring: a shop-templates-rendered `bin/system-manifest` with `assemble`/`validate` verbs; CAPTURE-first (drive deferred); reads already-released ADR-039 versions through the ADR-021-rebuilt image provenance (implements PDR-030 D3)

- **The assemble/validate tooling is shop-templates-rendered `bin/` furniture**,
  so every product's lead gets it — consistent with `bin/doctor` (PDR-024 D1),
  `bin/ops-coordinates` (ADR-043), and `bin/bootstrap`. It is rendered at
  bootstrap and refreshed by the update path (ADR-043 render/update pattern). Its
  shape mirrors the existing sibling registry's CLI: as `bc-container manifest
  list|validate|sync` operates `bc-manifest.yaml`, **`bin/system-manifest
  assemble|validate`** operates `system-manifest.yaml`.
- **The manifest is CAPTURING-first** (PDR-030 D3's "at minimum the authoritative
  captured record"). `assemble` composes a new `releases[]` entry FROM
  already-released component versions and the product-chosen system semver: the
  component lines release independently under ADR-039 FIRST; `assemble` reads
  their published versions and pins the tuple. The tool does NOT drive component
  bumps in increment 1 — manifest-DRIVEN coordinated release (the tool
  orchestrating component pins/tags) is a deferred follow-up, because driving
  couples into the BC-dispatch orchestration that PDR-030 D5 explicitly defers
  (the lead carries no BC source, ADR-018). Increment 1: capture. Follow-up: drive.
- **Reconciliation with ADR-039 (unchanged):** `assemble` consumes the output of
  the ADR-039 cadence; it does not alter it. The component bump/release/reinstall
  loop is untouched. The system release is a lead-side composition layered ON TOP
  of independent component releases.
- **Reconciliation with ADR-021 (read through the rebuilt image):** because
  `bc-base`/`bc-lead` auto-rebuild on a utility release, the authoritative
  published version of the baked components is what the REBUILT image carries.
  `assemble` therefore reads the component versions from the lead-5xnd baked
  provenance of the pulled image (`docker image inspect` labels / container ENV,
  pre-state finding 3) as the published truth, reconciled with component-repo
  tags — not from source tags alone, which can lead the rebuilt-and-published
  image.

##### D3 — Coherence check: BOTH a doctor check AND a bootstrap step, split by gate; advisory at dev-time, blocking at stand-up; published-coherence (lead-5xnd) buildable now, dependency-coherence (lead-h2p0) gated on that view landing (implements PDR-030 D4)

The coherence check is realized in **both** surfaces, with the gate posture split
exactly as PDR-030 D4 locked:

- **Authoring/dev-time — ADVISORY.** `bin/system-manifest validate` (and an
  additive `bin/doctor` check) compares the manifest tuple to reality and WARNS on
  drift but exits 0. The warning rides the PDR-024 D3 diagnosis surface —
  `name (system-manifest coherence) + pass/fail status + remediation hint` — and
  is folded into doctor's aggregate as a NON-FATAL line at dev-time, so it never
  blocks a deliberately-ahead-of-the-view pin (PDR-030 D4 rationale: a hard veto
  at authoring would invert the lead author's authorship of the tuple).
- **Adopter bootstrap/stand-up — BLOCKING.** The deterministic agent-less
  bootstrap (ADR-040) runs the coherence check as a stand-up step and, on an
  incoherent tuple, REFUSES TO PROCEED: it does not stand up the shop and exits
  non-zero with a diagnostic naming the incoherent component(s), the
  manifest-pinned version, the observed value, and a remediation. This reuses the
  feature-226 / lead-b2iz read-and-refuse mechanism verbatim, generalized from a
  single shop-templates floor to a full-tuple match — and is consistent with the
  doctor-gates-operations direction (lead-arp1).

The check compares the manifest tuple against two surfaces:

1. **Published coherence vs lead-5xnd baked provenance — BUILDABLE NOW.** Does the
   tuple the manifest pins MATCH what the pulled images actually bake? Compare the
   manifest's `shopsystem-bc-launcher` / `shopsystem-templates` versions against
   the pulled image's `org.opencontainers.image.version` /
   `shopsystem.shop-templates.version` labels (or container ENV
   `SHOPSYSTEM_BC_LAUNCHER_VERSION` / `SHOP_TEMPLATES_VERSION`), via the same
   `docker image inspect` / `printenv` read feature 226 uses (pre-state findings
   3, 4). **Scope note:** only `bc-launcher` and `shop-templates` are baked into
   the image provenance today; `messaging` and `scenarios` versions are not in the
   lead-5xnd label/ENV set, so their published-coherence compare falls back to
   component-repo tags until provenance for them exists. This leg ships in
   increment 1.

2. **Dependency coherence vs lead-h2p0 — GATED ON lead-h2p0 LANDING.** Does the
   pinned tuple HOLD TOGETHER per the dependency view (which versions
   bake/require which)? The data source is lead-h2p0, which is **OPEN / not yet
   built** (pre-state finding 5). This leg of the check is DESIGNED here but has
   no data source until lead-h2p0 lands; the validator consumes lead-h2p0's view
   when present and otherwise degrades to "published-coherence only" with an
   advisory note that dependency coherence is unevaluated. **This is the named
   data-source dependency** PDR-030 D4 flagged: increment 1 ships the
   published-coherence gate (lead-5xnd, available); the dependency-coherence gate
   is a follow-up activated when lead-h2p0 is built.

#### Consequences

- **The lead-6b53 ambiguity is closed by a resolvable artifact** (D1): "shopsystem
  vN" reads from one `releases[]` entry to a concrete tuple, for the current and
  every prior system version, without reconstruction across four lines.
- **Tooling symmetry with the existing sibling registry** (D2): `bin/system-manifest
  assemble|validate` over `system-manifest.yaml` mirrors `bc-container manifest
  list|validate|sync` over `bc-manifest.yaml`, so operator/tooling expectations
  carry over. It is rendered furniture every product inherits.
- **No cadence change** (D2): ADR-039 component releases and ADR-021 auto-rebuild
  are untouched; the manifest is a composition layer that READS their output
  (through the rebuilt image's baked provenance).
- **The coherence gate has a working leg now and a gated leg later** (D3): the
  published-coherence gate against lead-5xnd ships in increment 1 (the provenance
  is live on this host); the dependency-coherence gate against lead-h2p0 is a
  designed follow-up. The blocking stand-up behavior reuses the proven feature-226
  refuse-loudly mechanism, so it is not new machinery.
- **No Gherkin authored, no `@scenario_hash` retired** (pre-state finding 7).
  Feature 226's single-floor gate is RELATED-not-retired: D3 generalizes it
  additively. lead-po authors the new pins next.

#### Follow-ups / dependencies (named, not designed here)

- **lead-h2p0 (dependency-coherence data source)** — the dependency-coherence leg
  of D3 is inert until this view is built. Increment 1 ships without it.
- **Manifest-DRIVEN coordinated release** — the "drive" half of D2 (the tool
  orchestrating component bumps/tags rather than only capturing) is a deferred
  follow-up, entangled with the deferred PDR-030 D5 BC-self-tag leg (ADR-018: the
  lead orchestrates via dispatch, BCs act).
- **Published-coherence for messaging/scenarios** — lead-5xnd bakes provenance
  only for `bc-launcher`/`shop-templates`; extending the baked-provenance set (or
  the validator's tag-fallback) to cover `messaging`/`scenarios` is a follow-up.
- **Update-path render of `bin/system-manifest`** — like ops-coordinates
  (ADR-043, scenarios 213/228), whether `shop-templates update` refreshes a stale
  rendered `bin/system-manifest` is a render-policy follow-up; increment 1 pins
  bootstrap-rendered presence.

#### Alternatives considered

- **Single-version file instead of a `releases:` list.** Rejected (D1): it loses
  the historical N→tuple mapping an adopter on an older release needs, and pushes
  history onto git tags the agent-less bootstrap (ADR-040) cannot rely on.
- **Embed the lead-h2p0 dependency map in the manifest.** Rejected (D1): the
  manifest would re-derive the dependency view rather than consume it (Brief 015
  §4 / PDR-030 D2). Reference at validation time keeps one source of dependency
  truth.
- **Make the manifest manifest-DRIVEN (orchestrate component releases) in
  increment 1.** Rejected as the first increment (D2): driving couples into the
  BC-dispatch tag-reflection leg PDR-030 D5 explicitly defers; capture-first
  delivers the unambiguous-version outcome with no BC behavior change.
- **A pure bootstrap-only gate (no doctor check) or a pure doctor-only check (no
  bootstrap gate).** Rejected (D3): PDR-030 D4 splits posture BY GATE — advisory
  everywhere a deliberate pin must not be blocked, blocking where an adopter must
  not stand up an incoherent tuple. Both surfaces are required to honor that split.
- **Hard-block coherence at authoring/dev-time too.** Rejected (D3): a hard veto
  there inverts the lead author's authorship of the tuple (PDR-030 D4 rationale);
  advisory warns loudly without seizing authorship.
